# 战斗动效升级方案

日期：2026-07-06

## 目标

让探索遭遇战从“按钮触发数值变化”升级为“看得懂、够短、够爽”的小剧场。P1 先补攻击和技能动效层，不重构战斗逻辑，不强依赖外部 Lottie CDN，不抢占已有首页和音效改动。

## 当前诊断

现有战斗已经具备基础反馈：`js/exploration.js` 会发出 `battle-animate` 事件，`js/app.js` 会播放语义音效、显示伤害数字、触发受击抖动和胜负光效，`css/style.css` 已有 reduced motion 兜底。问题不在“没反馈”，而在反馈层级过于相似：普攻、强力击、必杀都主要表现为抖动和伤害数字，玩家不容易感到技能强弱差异。

## 设计原则

1. 动作三段式：预备、命中、反馈。每个出招控制在 0.8-1.0 秒内，避免拖慢探索节奏。
2. 信息优先：HP、伤害、按钮状态和战斗气泡不能被特效遮住。
3. 技能差异化：普攻是斩击，强力击是爆裂，防御是护盾，必杀是雷击。
4. 可降级：没有 Lottie 渲染器时仍用 CSS/SVG 可见；开启 `prefers-reduced-motion: reduce` 时关闭大幅动画。
5. 小步落地：先做探索遭遇战 P1；卡牌对战和 Agnes/GPT 背景作为 P2/P3。

## 方案取舍

推荐方案是新增 `BattleFx` 轻量动效层，挂在现有 `battle-animate` 事件后面。它只负责“根据事件和 skillId 生成视觉特效”，不参与伤害计算、回合推进和掉落结算。这样可以复用当前战斗事件总线，也能避免把 CSS 动画散落到 `handleBattleAnimate` 里。

备选方案 A 是直接在 `handleBattleAnimate` 里继续加 class。优点是快，缺点是后续技能增加后会越来越乱。备选方案 B 是一次性引入 Lottie Web/CDN。优点是动画表达更强，缺点是静态站离线能力和加载性能会被新依赖绑住。P1 不采用。

## P1 范围

- 新增 `js/battle-fx.js`：暴露 `window.BattleFx.show(type, detail)`。
- 新增 CSS 动效：普攻斩击、敌方爪击、强力击爆裂、防御护盾、必杀雷击、开战/胜利/失败氛围。
- 新增 Lottie JSON 资产：放在 `assets/battle-fx/lottie/`，并同步到 `prj/GPT生图--Lottie动画` 的预览项目中，作为后续接入 Skottie/Lottie Web 的资产基础。
- 修改 `index.html`：在 `app.js` 前加载 `js/battle-fx.js`。
- 修改 `js/app.js`：在 `handleBattleAnimate` 中调用 `BattleFx.show(type, e.detail)`。
- 新增契约测试：确认事件映射、CSS class、Lottie 资产和 reduced motion 兜底存在。

## P2/P3 后续

P2：优化探索战背景氛围。优先复用 `assets/arena`，加暗角、前景光束和章节色调；只有 Boss 战或章节终点再用 Agnes/GPT 生图补高质量背景。

P3：卡牌对战强化战术感。保留全屏竞技台，增加行动顺序提示、技能冷却强调、克制/换人提示和 Finish 动画，不和探索遭遇战做成同一种节奏。

## 验收标准

- 普攻、敌方攻击、强力击、防御、必杀至少 5 类视觉反馈可区分。
- `battle-animate` 原有音效、伤害数字、HP 刷新不回退。
- 无 Lottie 渲染器时 CSS/SVG fallback 正常显示。
- Lottie JSON 均可被 JSON parser 解析，并具备 `v/fr/ip/op/w/h/nm/assets/layers` 顶层字段。
- reduced motion 下大幅动画关闭。
