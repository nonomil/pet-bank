# 目标架构与工程治理

## 目标分层

```text
App Shell / Router
        |
Feature Controllers
(task, pet, learn, game, household, social)
        |
Domain Services
PointLedger | RewardService | ProfileService | MasteryService | PKService
        |
State Gateway
LocalRepository | MigrationRegistry | SyncOutbox
        |
Cloud Gateway
SupabaseClient | Transactional RPC | Edge Functions
        |
Postgres
RLS | revisions | audit log | idempotency
```

这不是框架迁移，而是逐步把 `window.*` 背后的业务规则抽成可测试服务。旧 API 暂时保留为兼容 facade。

## 状态治理

建立机器可读 `storage-registry.json`，每个键至少声明：

- `scope`: `learner | household | parent | device | auth | cloud-config`
- `schemaVersion`
- `profileScoped`
- `cloudSync`
- `retention`
- `migration`

Profile 导出与云快照最终应使用显式白名单。Registry 首阶段只报告未登记键，不参与运行时读写；覆盖现有键并通过 Profile round-trip 测试后，再启用白名单。快照只保存孩子领域状态，并带 `learnerId`、`schemaVersion`、`revision` 和 `createdAt`。

## 积分账本

统一入口：

```text
PointLedger.grant({
  eventId,
  learnerId,
  sourceType,
  sourceId,
  amount,
  occurredAt,
  evidence,
  policyVersion
})
```

账本至少区分 `grant`、`spend` 和 `adjust`，避免把现有消费路径错误塞进只允许正数的 grant。所有任务、学习、游戏、宝箱和管理员调整必须提交不可重复事件。余额由事件归并或可靠快照计算；刷新、重放和多标签页不能重复计分。儿童账户不使用负余额，惩罚改为行为规则而非债务。

## 云同步

当前模型更接近“追加备份 + 首次恢复”。目标模型应增加：

1. 持久 outbox，离线修改在刷新后仍可重试。
2. 服务端 revision，上传时携带 base revision。
3. 幂等键，重复请求只生效一次。
4. 冲突策略：自动合并安全字段，其余提示家长选择。
5. 快照保留策略，限制数量或时间。
6. 邀请、PK、同步等多步骤写入放入事务 RPC。

## 安全与隐私

- 浏览器只使用 anon key，service role 仅在受控后端。
- 使用 secret scanner 检查工作树、Git 历史和发布产物；不在日志、文档或聊天中输出 key。
- 家庭角色至少区分 owner、guardian、viewer，不能只用 active member 表示所有权限。
- 儿童数据提供家长同意、导出、删除、保留期限和第三方外链说明。
- PK 结果服务端重算或验证答案事件，不接受可信分数作为客户端字段。

## 模块拆分顺序

1. 从 `app.js` 抽出 `point-ledger`、`daily-task-service`、`game-result-bridge`。
2. 从 `learn-center.js` 抽出 `content-repository`、`learning-session`、`review-scheduler`。
3. 为 `window.addGrowthPoints` 等旧入口保留适配层。
4. 新模块先写纯函数和契约测试，再迁移调用点。
5. 只有边界稳定后才评估 TypeScript 或打包工具，不把换技术栈当目标。

## CI/CD 门禁

建议发布流水线分三层：

1. 快速门禁：`node --check`、JSON schema、storage registry、secret scan。
2. 业务门禁：每日任务、积分幂等、Profile 隔离、学习调度、PK 合约。
3. 发布门禁：静态路由、资源存在性、体积预算、Playwright 核心流程。

发布产物应由 route/asset manifest 生成，并输出文件数、总大小、最大文件和未引用资产报告。
