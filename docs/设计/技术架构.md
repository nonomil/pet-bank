# 技术架构

> 本文为「宠物积分银行」(pet-bank) 的技术架构总览，描述技术栈、整体架构、数据流、目录结构、关键文件职责与部署方式。
>
> **素材来源映射**：技术栈与目录结构来自 `index.html`、`README.md`、`js/`、`css/`、`data/`、`.github/workflows/` 的源码核实；功能边界与需求对齐见 [../规格/需求规格书.md](../规格/需求规格书.md)；模块清单与接口契约见 [./模块清单与接口.md](./模块清单与接口.md)；演进差距与开发路线见 [../路线/差距清单与开发路线图.md](../路线/差距清单与开发路线图.md)。

## 1. 技术栈

宠物积分银行采用**纯前端零框架**路线，核心目标是「无需账号、浏览器打开即用、家长可离线部署」。

| 层次 | 选型 | 说明 |
|------|------|------|
| 标记 | HTML5 | `index.html` 单入口，承载 13 个页面容器与顶部导航骨架 |
| 样式 | Tailwind CSS（CDN） | 工具类原子样式，配合自写 CSS 组件层 |
| 自写 CSS | 5 个组件样式表 | 主样式、动画、宝箱、遛弯、图鉴各司其职 |
| 交互脚本 | Vanilla JS（ES5+） | 零框架、零构建步骤，12 个模块化 JS 文件（含 `home.js`） |
| 图标 | Lucide Icons（CDN） | `lucide@latest` 通过 `<script>` 引入 |
| 字体 | Noto Sans SC | Google Fonts CDN |
| 数据存储 | localStorage | `petbank_*` 系列键持久化运行态 |
| 数据来源 | `data/*.json` | 首次加载通过 `fetch` 拉取静态 JSON |
| 部署 | GitHub Pages | `.github/workflows/deploy.yml` 静态托管 |

**关键约束**：

- **无后端**：所有逻辑在浏览器端运行，无服务端 API、无数据库。
- **无构建步骤**：不依赖 npm/webpack/vite，源码即产物，`git push` 即发布。
- **CDN 依赖**：Tailwind / Lucide / 字体均走公网 CDN，离线场景需自行替换为本地资源。

## 2. 整体架构

### 2.1 单页应用模型

项目为**单页应用（SPA）**，`index.html` 是唯一入口。页面骨架固定为三栏布局：

```
┌─────────────────────────────────────────────────────────┐
│  TOP NAV  （13 个页面入口 + 顶部积分胶囊）                │
├──────────────┬──────────────────────────────────────────┤
│              │                                          │
│  LEFT        │   MAIN CONTENT                           │
│  SIDEBAR     │   （13 个 .page 容器，按页面切换显隐）     │
│  今日任务     │                                          │
│  成长账户     │                                          │
│  孩子姓名     │                                          │
│              │                                          │
└──────────────┴──────────────────────────────────────────┘
```

### 2.2 页面切换机制

当前共 13 个页面通过 `switchPage(pageKey)` 函数切换，绑定在导航 Tab 的 `onclick` 上：

| 页面 Key | 显示名 | 主导模块 |
|----------|--------|----------|
| `map` | 成长地图 | `app.js` + `exploration.js` |
| `today` | 今日打卡 | `app.js` |
| `review` | 每周复盘 | `app.js` |
| `reward` | 奖励兑换 | `app.js` |
| `works` | 成长作品 | `app.js` |
| `pet` | 🐾 宠物养成 | `pet.js` |
| `home` | 🏠 宠物小屋 | `home.js` + `pet.js` |
| `explore` | 🗺️ 探索冒险 | `exploration.js` + `exploration-detail.js` |
| `mathpk` | 🔢 数学PK | `math-pk.js` |
| `inventory` | 🎒 背包 | `inventory.js` |
| `card` | 🃏 卡片图鉴 | `card-collection.js` |
| `shop` | 🏪 兑换商店 | `shop.js` |
| `tools` | 🧰 工具箱 | `tools.js` |

切换流程：`switchPage(key)` → 隐藏所有 `.page` → 显示 `#page-${key}` → 触发对应模块的渲染钩子（如 `renderMap()`、`renderPet()`）。

### 2.3 模块化分工

12 个 JS 文件各司其职，通过全局函数挂载点协作（无模块打包器）：

- **核心编排**：`app.js`（50KB，主入口）负责任务、积分、页面切换、UI 渲染、账户体系。
- **养成线**：`pet.js` 宠物养成。
- **家园线**：`home.js` 宠物小屋、家具摆放、5 维状态、倒下/救援与衰减展示。
- **探索线**：`exploration.js`（地图） + `exploration-detail.js`（场景中间页：故事+互动+战斗）。
- **玩法线**：`math-pk.js`（数学PK）、`shop.js`（盲盒商店）、`treasure.js`（宝箱）、`walk.js`（遛弯）、`card-collection.js`（图鉴）。
- **辅助**：`tools.js`（点名/番茄钟）、`inventory.js`（背包）。

脚本按依赖顺序在 `index.html` 末尾以 `<script>` 串行引入。

## 3. 数据流

### 3.1 三层数据载体

```
┌─────────────────┐   fetch (首次加载)   ┌─────────────────┐   读写   ┌──────────────────┐
│  data/*.json    │ ───────────────────► │  运行态内存       │ ◄─────► │   localStorage   │
│  静态只读资源    │                      │  JS 模块变量      │          │  petbank_* 系列键 │
│  pets/scenes/   │ ◄─────────────────── │  (state 对象)    │          │  单设备持久化      │
│  items/combat   │   仅作模板读取        └─────────────────┘          └──────────────────┘
└─────────────────┘                                                       │
                                                                          │ 下次打开
                                                                          ▼
                                                                   恢复运行态
```

### 3.2 数据流向说明

| 方向 | 载体 | 用途 |
|------|------|------|
| `data/*.json` → 内存 | `fetch` 异步加载 | 拉取宠物物种、场景、物品、战斗配置（只读模板） |
| 内存 → localStorage | `localStorage.setItem` | 持久化积分余额、已领养宠物、解锁进度、背包、图鉴等 |
| localStorage → 内存 | `localStorage.getItem` | 页面打开/切换时恢复运行态 |
| UI ↔ 内存 | DOM 事件 + 渲染函数 | 用户操作触发状态变更，再回写 UI |

### 3.3 localStorage 键命名约定

所有持久化键统一以 `petbank_` 前缀命名（如 `petbank_points`、`petbank_pet`、`petbank_inventory`），避免与其他站点冲突。具体键清单与读写时机见 [./模块清单与接口.md](./模块清单与接口.md)。

### 3.4 数据资源规模

| 文件 | 大小 | 规模 |
|------|------|------|
| `data/pets.json` | ~272KB | 147 种宠物物种 |
| `data/scenes.json` | ~22KB | 12 个探索场景 |
| `data/items.json` | ~16KB | 26 件物品 |
| `data/combat.json` | ~1.3KB | 战斗配置 |

## 6. 部署

### 6.1 部署目标

- **主部署**：GitHub Pages（仓库自动构建）
  - 在线访问：`https://summer-bank.nonom.top`
  - GitHub Pages：`https://nonomil.github.io/pet-bank/`
- **备选**：任意静态托管（Cloudflare Tunnel、Netlify、本地 `python -m http.server`）。

### 6.2 部署流水线

`.github/workflows/deploy.yml` 定义了 GitHub Pages 自动部署：

| 步骤 | Action | 作用 |
|------|--------|------|
| 触发 | `push` 到 `main` 或手动 `workflow_dispatch` | 推送即发布 |
| 检出 | `actions/checkout@v4` | 拉取源码 |
| 配置 | `actions/configure-pages@v4` | 准备 Pages 环境 |
| 打包 | `actions/upload-pages-artifact@v3` (`path: .`) | 整个仓库根目录作为静态产物 |
| 发布 | `actions/deploy-pages@v4` | 上传到 GitHub Pages |

**关键特性**：

- **无构建步骤**：`path: .` 直接把仓库根目录当产物，源码即部署产物。
- **并发控制**：`concurrency.group: "pages"`，避免并发部署冲突。
- **权限**：`pages: write` + `id-token: write`，使用 OIDC 短令牌发布。

### 6.3 本地预览

因 `data/*.json` 通过 `fetch` 加载，**不可直接双击打开 `index.html`**（`file://` 协议下 fetch 受 CORS 限制），需起本地静态服务：

```bash
# 任选其一
python -m http.server 8000
npx serve .
```

浏览器访问 `http://localhost:8000` 即可。

## 7. 约束与边界

### 7.1 技术约束

| 约束 | 说明 | 影响 |
|------|------|------|
| 纯前端 | 无服务端、无 API、无数据库 | 所有状态保存在浏览器本地 |
| 无构建步骤 | 源码即产物 | 无法使用 ES Module `import`、TS、JSX、Tree-shaking |
| CDN 依赖 | Tailwind / Lucide / 字体走公网 | 弱网/离线首屏体验下降 |
| `localStorage` 容量 | 浏览器单源约 5–10MB | `pets.json`（272KB）只读模板不入库，仅存养成进度 |
| `fetch` 协议限制 | `file://` 下不可用 | 本地预览必须起静态服务 |

### 7.2 功能边界

| 边界 | 现状 | 说明 |
|------|------|------|
| 多用户同步 | 不支持 | 单设备单浏览器持久化，换设备/换浏览器数据不互通 |
| 多孩子账户 | 不支持 | 仅 `childName` 输入框，无账户切换 |
| 数据备份 | 手动 | 无导出/导入 UI（待补，见 [../路线/差距清单与开发路线图.md](../路线/差距清单与开发路线图.md)） |
| 数据安全 | 无 | 清浏览器缓存即丢失全部进度，无云端副本 |
| 离线可用 | 部分 | 首次加载后 JS/CSS 有浏览器缓存，但 CDN 资源仍需联网 |

### 7.3 与参考项目对比

| 维度 | 本项目 (pet-bank) | 参考项目 v2 |
|------|------------------|-------------|
| 后端 | 无 | 有（云端账户/数据同步） |
| 多用户 | 单机 | 多用户 |
| 部署 | GitHub Pages 静态 | 服务端部署 |
| 运维成本 | 零 | 需维护服务端 |
| 定位 | 单孩子单设备的轻量玩具 | 多孩子/跨设备的家庭系统 |

> 本项目刻意选择「零后端」路线换取零运维与即开即用，多用户同步为后续演进方向，差距与路线见 [../路线/差距清单与开发路线图.md](../路线/差距清单与开发路线图.md)。
