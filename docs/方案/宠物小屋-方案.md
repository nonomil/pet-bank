## 1. 方案定位与目标

### 1.1 一句话定位

**宠物小屋是宠物的「家」**——一个独立的家园空间，把宠物的 HP/饱食/快乐/亲密度/经验**五维**状态从抽象数字转化为「它在小屋里活着」的可视化场景，并承载精灵乐园最高价值的「饥饿 → 倒下 → 救援」情感闭环。

### 1.2 解决的问题

| 问题 | 现状 | 小屋提供的解 |
|---|---|---|
| 宠物状态无归属感 | pet.js 把 HP/饱食/快乐压成数字条，宠物像「悬浮的卡片」 | 给宠物一个物理空间，状态变化映射到场景视觉 |
| 衰减机制无视觉反馈 | 需求规格书 §5.3 要求「每小时饱食 -2/快乐 -1；饱食 0 时 HP -5」，但 pet.js **无衰减定时器也无 decay API**（[模块清单 B2](../设计/模块清单与接口.md) 已知断裂点） | 小屋作为衰减的可视化载体，进入即结算、离开即暂停的轻量衰减 |
| 倒下/救援无 UI | `PetSystem.revive(hpPercent=50)` 接口已存在但**无任何 UI 调用**，宠物 HP 归零后静默 | 倒下立绘 + 救援 CTA + 复活动画，直接照搬精灵乐园 frame-006 |
| 互动 API 不写 hunger/intimacy/exp | 现有 `feed/play/rest`（pet.js:184/195/206）只改 hp/happiness，**不写饱食/亲密/经验**，与规格书 §5.3 口径脱节 | 改造 feed/play/rest，对齐饱食+25/经验+10 等权威参数 |
| 缺洗澡互动 | 规格 §2.2 明确要求 4 按钮（喂食/玩耍/**洗澡**/治疗），pet.js **无 bath API** | 新增 `PetSystem.bath()` + 洗澡按钮 |
| 家园感空白 | 原始需求明确提到「给宠物一个"家" (Habitat) 区域」（[原始需求](../原始需求/2026-0628-宠物积分银行-原始需求.md)），代码 0 匹配 | 小屋场景 + 家具摆放 + 装饰持久化 |

### 1.3 一句话目标

**在不引入后端、不破坏现有 SPA Tab 架构的前提下，新增 1 个 `home` Tab 与 1 个 `js/home.js` 模块，改造 PetSystem 互动 API、新增 decay/bath API，把宠物状态可视化升级为「家园养成 + 饥饿倒下救援闭环」。**

## 4. 核心玩法设计

### 4.1 五维状态在小屋的可视化映射（EXP 常驻）

| 维度 | 区间 | 小屋视觉反馈 | 触发阈值 |
|---|---|---|---|
| HP | 0–100 | 顶部红色状态条；HP<40 触发「虚弱」文案；HP=0 切倒下立绘 | <40 警告，=0 倒下 |
| 饱食 Hunger | 0–100 | 顶部橙色状态条；饱食<20 切饥饿立绘 + 「我饿了」气泡 | <20 饥饿，=0 HP 开始衰减 |
| 快乐 Happiness | 0–100 | 顶部粉色状态条；快乐<20 触发「不开心」文案 | <20 不开心 |
| 亲密度 Intimacy | 0+ | 顶部紫色状态条；到达进化阈值（50/150/300/600）触发进化提示 | 见 [需求规格书 §5.2](../规格/需求规格书.md) |
| **经验 EXP** | 0–EXP_TABLE[level+1] | **顶部金色状态条（常驻）**；进度 = exp / 下一级所需 | 满经验触发升级 |

> **EXP 不再降级到次级面板**（吸收阻断修正④ + 用户决策「5 维常驻」）。顶部状态胶囊固定 5 格。
>
> **占位策略**：在 PetSystem.state 扩展 `hunger`/`intimacy` 字段上线前，饱食/亲密两格显示占位「--」（CSS 灰色），避免渲染 NaN。扩展完成后自动填充真实值。

### 4.2 自然衰减机制（落地精灵乐园 L3，P0 化）

**权威参数**（[需求规格书 §5.3](../规格/需求规格书.md)）：

| 动作 | 效果 |
|---|---|
| 自然衰减（每小时） | 饱食 -2、快乐 -1 |
| 饱食为 0 时 | HP -5/小时 |

**本方案落地策略——「进入小屋即结算」**（折衷设计，封装为 `PetSystem.decay()`）：

```javascript
// PetSystem.decay() —— 新增 API，吸收阻断修正①（单一归属 + 防篡改）
function decay() {
    const now = Math.floor(Date.now() / 1000);  // Unix 秒
    const ts = state.last_home_ts;               // 单一归属：PetSystem.state
    // 守卫：null / NaN / 时间篡改（未来时间）一律 hours=0
    let hours = 0;
    if (ts && !isNaN(ts) && ts > 0 && ts <= now) {
        hours = Math.max(0, Math.floor((now - ts) / 3600));
    }
    if (hours >= 1) {
        state.hunger     = Math.max(0, state.hunger - hours * 2);
        state.happiness  = Math.max(0, state.happiness - hours * 1);
        if (state.hunger <= 0) {
            state.hp = Math.max(0, state.hp - hours * 5);
        }
    }
    state.last_home_ts = now;                     // 结算后即更新，防二次结算
    save();
    return { hours, applied: hours >= 1 };
}
```

**last_home_ts 单一归属（阻断修正①）**：
- 字段归 `PetSystem.state.last_home_ts`，**不**复制到 `petbank_home_state`；
- **写入方**：`HomeSystem` 在进入小屋（`renderUI` 触发 `decay()` 后）与离开小屋（`switchPage` 切走时）统一经 PetSystem 写入；
- **读取方**：仅 `PetSystem.decay()` 与 PetSystem.load()；
- 守卫三连：`if (!ts) hours=0` + `if (ts>now) hours=0`（防时间篡改/系统时钟回拨）+ `Math.max(0, ...)`（防负）。

**为何不用 `setInterval` 全局定时器**：
1. SPA 切 Tab 时其他模块已在跑定时器（探索 30s、番茄钟等），再加一个会加重单线程负担；
2. 离线（关浏览器）期间定时器不工作，按时间戳结算更符合「养成」语义；
3. 与精灵乐园「长时间不喂会倒下」的体验一致（借鉴分析 §4.4）。

**衰减触发点**：
- 进入小屋 Tab（`switchPage('home')`）时 `HomeSystem.renderUI()` 调用 `PetSystem.decay()` 结算一次；
- 离开小屋 Tab 时 `PetSystem` 写入 `last_home_ts = now`（由 home.js 的 `onLeave` 钩子触发，或 switchPage 切走前调用 `PetSystem.markHomeExit()`）；
- 应用启动 `init()` 时若检测到 `petbank_pet` 已存在且 `last_home_ts` 存在，补结算一次（防止用户长期不开小屋 Tab，HP 持续衰减至倒下需救援）。

### 4.3 倒下-救援闭环（精灵乐园最高价值借鉴 L1/L5/L6/L7）

```
                ┌─────────────────────────────┐
                │  HP 归零（state.hp === 0）   │
                └──────────────┬──────────────┘
                               ▼
            ┌──────────────────────────────────┐
            │  小屋立绘切换为「倒下态」         │
            │  （灰色滤镜 + 旋转 90° + ZZZ 气泡）│
            │  顶部 HP 条置 0 + 红色警告闪烁     │
            │  互动栏禁用，弹出「救援」CTA 按钮  │
            │  ⚠️ 探索入口禁用（见下方策略）     │
            └──────────────┬───────────────────┘
                           │ 用户点击救援
                           ▼
            ┌──────────────────────────────────┐
            │  调用 PetSystem.revive(50)        │
            │  （接口已存在，js/pet.js:215）     │
            │  HP 恢复至 max_hp * 50%           │
            │  触发复活动画（复用进化 keyframes）│
            │  庆祝弹窗「宠物被救活了！」        │
            │  探索入口解除禁用                 │
            └──────────────────────────────────┘
```

**视觉状态机**（照搬 [借鉴分析 §6 状态反馈](../参考/精灵乐园设计借鉴分析.md)）：

| 状态 | 触发条件 | 立绘 | 文案 |
|---|---|---|---|
| 正常 | HP>40 且 饱食≥20 | 默认立绘 | 无 |
| 虚弱 | HP<40 且 HP>0 | 默认立绘 + 红色滤镜 | 「我好虚弱……」 |
| 饥饿 | 饱食<20 且 HP>0 | 颓废立绘 | 「我饿了」 |
| 倒下 | HP=0 | 倒下立绘（旋转+灰） | 「宠物倒下了，点击救援！」 |

> 饥饿与虚弱可叠加，优先级：倒下 > 饥饿 > 虚弱。

**救援阻断探索策略（裁决 F4 落地，对齐精灵乐园 frame-006 强制救援）**：
- HP=0（倒下态）时，**探索 Tab 入口禁用变灰** + 文案「宠物倒下，请先去小屋救援」；
- 探索模块 `renderExplorePage()` 入口加守卫：`if (PetSystem.getState().hp <= 0) { 提示并阻止 }`；
- 复活（`revive` 成功）后自动解除禁用，无需手动刷新；
- **理由**：精灵乐园 frame-006 设计为倒下强制救援，本项目对齐此心智流，避免「宠物倒着去探索」的违和与数值叠加 BUG（探索战败 HP -50% 叠加在 0 上无意义）。

### 4.4 互动 API 改造（P0，对齐规格 §5.3）

**现状**（js/pet.js）：
- `feed(foodItem)`（184）：只 `heal(hp)` + `happiness+5`，**不写 hunger/exp**，与规格 §5.3「饱食+25/经验+10」脱节；
- `play()`（195）：`happiness+15 / hp-5`，不写 intimacy/exp；
- `rest()`（206）：`heal(30% max_hp) / happiness-5`，不写 intimacy；
- **无 `bath()`**。

**改造后口径**（全部对齐规格书 §5.3，单位：单次互动）：

| 互动按钮 | 映射 API | 改造点 | 权威参数 |
|---|---|---|---|
| 🍕 喂食 | `PetSystem.feed()` (改造) | **改写为 hunger+25 / exp+10 / 积分-10**；HP 回复改为次要项（道具自带 hp 则额外回复） | 饱食+25、经验+10、积分-10（[§5.3](../规格/需求规格书.md)） |
| 🎾 玩耍 | `PetSystem.play()` (改造) | 新增 intimacy+5 / exp+5；保留 hp-5 | 快乐+15、HP-5、亲密+5（新增） |
| 🛁 洗澡 | `PetSystem.bath()` (**新 API**) | 见 §4.5 | 清洁度+30、快乐+5（新增设计） |
| 💤 治疗/休息 | `PetSystem.rest()` (改造) | 重命名语义为「治疗」按钮，新增 intimacy+2；保留 heal 30% max_hp | 恢复 30% HP（[§5.3 衍生](../规格/需求规格书.md)） |

> **治疗 vs 独立 heal 的裁决**（吸收阻断修正⑤）：本方案 4 按钮映射为「喂食→feed / 玩耍→play / 洗澡→bath / 治疗→rest」。不新增独立 `heal()` API，理由：① pet.js 已有内部 `heal()` 函数（被 feed/revive 复用），公开 API 再叫 heal 会混淆；② 规格 §2.2 文案是「治疗」，语义对应「回血+休养」，与现有 `rest()` 的「恢复 30% HP」天然吻合。按钮文字改「治疗」、底层调 rest。

**feed 改造伪代码**（回归点见 §9.4）：

```javascript
function feed(foodItem) {
    if (!state.species) return { success: false, msg: '请先选择一只宠物' };
    if (state.hp <= 0) return { success: false, msg: '宠物需要先复活' };
    // 积分扣除（积分系统 API）
    const costOk = PointsSystem.spend(10);
    if (!costOk) return { success: false, msg: '积分不足（需 10 分）' };
    state.hunger = Math.min(100, state.hunger + 25);   // 新增：饱食+25
    addExp(10);                                         // 新增：经验+10
    if (foodItem?.effect?.hp) heal(foodItem.effect.hp); // 道具额外回 HP（保留兼容）
    state.happiness = Math.min(100, state.happiness + 5);
    save();
    return { success: true, msg: '喂食成功！饱食+25 经验+10' };
}
```

### 4.5 bath 新 API 设计（阻断修正⑤）

**语义定义**：洗澡 = 清洁护理，提升**清洁度（cleanliness）**与少量**快乐**。

**为何引入 cleanliness**：规格 §2.2 五维状态是 HP/饱食/快乐/亲密/经验，清洁度不在五维之内，但「洗澡」若只加快乐会与 play 重叠。折衷：cleanliness 作为**隐藏状态**（不进顶部五维条，仅在小屋角落显示一个小图标），其作用是：
- 避免「脏」debuff：长时间不洗澡 → cleanliness<20 → 立绘加灰滤镜 + 「我脏脏」气泡（类比饥饿预警）；
- bath 互动可解除该 debuff。

**bath 签名**（与 feed/play 一致的 IIFE 风格）：

```javascript
function bath() {
    if (!state.species) return { success: false, msg: '请先选择一只宠物' };
    if (state.hp <= 0) return { success: false, msg: '宠物需要先复活' };
    state.cleanliness = Math.min(100, (state.cleanliness ?? 50) + 30);  // 清洁度+30
    state.happiness   = Math.min(100, state.happiness + 5);             // 快乐+5
    save();
    return { success: true, msg: '洗得香喷喷！清洁度+30' };
}
// 挂载到 PetSystem 返回对象的 bath: bath
```

**cleanliness 衰减**：每小时 -1（与饱食/快乐同批结算，加入 `decay()`），归零不扣 HP（仅视觉 debuff）。

**4 按钮映射最终表**（吸收阻断修正⑤）：

| 按钮 | API | 状态影响 | 积分/道具 | cooldown |
|---|---|---|---|---|
| 🍕 喂食 | `feed(foodItem?)` | 饱食+25 / 经验+10 / (HP+道具值) | 积分-10 或消耗道具 | 无 |
| 🎾 玩耍 | `play()` | 快乐+15 / HP-5 / 亲密+5 / 经验+5 | 无 | 无 |
| 🛁 洗澡 | `bath()` (新) | 清洁度+30 / 快乐+5 | 无 | 无 |
| 💤 治疗 | `rest()` | HP+30%max / 亲密+2 | 无 | 无 |

```javascript
function switchPage(page) {
    document.querySelectorAll('.page').forEach(p => p.classList.remove('active'));
    const target = document.getElementById(`page-${page}`);
    if (target) target.classList.add('active');
    document.querySelectorAll('.nav-tab').forEach(t => t.classList.remove('active'));
    const tab = document.querySelector(`.nav-tab[data-page="${page}"]`);
    if (tab) tab.classList.add('active');
    if (page === 'pet') renderPetPage();
    if (page === 'explore') void renderExplorePage();
    if (page === 'mathpk') MathPKGame.renderUI('math-pk-container');
    if (page === 'inventory') renderInventoryPage();
    if (page === 'card' && window.CardCollection) CardCollection.renderUI('card-collection-container');
    if (page === 'shop' && window.ShopSystem) ShopSystem.renderUI('shop-ui');
    if (page === 'tools' && window.ToolboxSystem) ToolboxSystem.renderUI('tools-ui');
    // ★ 新增 home 分支（阻断修正②）
    if (page === 'home' && window.HomeSystem) {
        window.HomeSystem.renderUI('home-container');   // 内含 PetSystem.decay()
    } else if (window.HomeSystem && window.HomeSystem._lastPage === 'home') {
        // 离开 home：写 last_home_ts
        if (window.PetSystem && typeof window.PetSystem.markHomeExit === 'function') {
            window.PetSystem.markHomeExit();
        }
    }
    if (window.HomeSystem) window.HomeSystem._lastPage = page;
}
```

### 9.4 feed 改造的 inventory 回归（阻断修正③，重点风险）

**现状调用链**（js/inventory.js:79-106 `useItem`）：
- useItem 对消耗品**不调** `PetSystem.feed()`，而是直接调 `PetSystem.heal(hp)` + `PetSystem.addExp(exp)`（inventory.js:99-100）；
- 仅 `data.effect.revive` 时调 `PetSystem.revive()`（inventory.js:97）。

**因此 feed 改造的影响范围**：
- ✅ **不破坏** inventory useItem 链路（useItem 本就不调 feed）；
- ⚠️ **但需回归**：home.js 的「喂食」按钮调 `PetSystem.feed()` 时，道具分支（`foodItem?.effect?.hp`）与无道具分支（扣 10 分）都要测；
- ⚠️ **需对齐**：inventory useItem 的消耗品也应否写饱食？本方案 P0 **不改 inventory**（保持只加 HP/EXP），饱食写入仅走小屋 feed 按钮，避免连锁改造。规格 §5.3「喂食 饱食+25 或消耗道具」的「或」语义支持此拆分。

**回归测试点**（列入执行计划）：
1. inventory useItem 喂食类道具 → HP/EXP 正常，不报错；
2. home feed 无道具 → 积分 -10、饱食 +25、经验 +10；
3. home feed 带道具 → 积分 -10、饱食 +25、经验 +10、额外 HP；
4. 积分不足 → feed 拒绝并提示。

## 10. 不做清单（明确排除项）

> 本阶段明确排除以下范围，避免范围蔓延。每项标注排除理由。

| 排除项 | 排除理由 |
|---|---|
| 多房间 / 多楼层小屋 | 超出 [需求规格书 §2.12](../规格/需求规格书.md)「至少 1 个可摆放家具」的验收要求 |
| 社交拜访他人小屋 | 本项目是单机单设备（[技术架构 §7.2](../设计/技术架构.md)），无多用户数据 |
| 家具自由拖拽摆放 | 本阶段仅固定槽位（5 个），降低实现复杂度；自由拖拽排后续迭代 |
| 家具数值加成 | 避免与装备系统（PetSystem.equip）职责重叠，家具纯装饰 + 互动入口 |
| 复杂经营数值（租金/电费/水电气） | 超出原始需求「家园感」定位，且会增加衰减负担 |
| 自定义小屋主题切换 | 本阶段固定 `cozy_night`，主题切换排 P2+ |
| 宠物自主 AI 行为 | 本阶段立绘按状态静态切换，不做走动/睡觉动画序列 |
| 全局 setInterval 衰减定时器 | 用「进入小屋即结算」替代，避免后台耗电与刷新遗漏（见 §4.2） |
| 多宠物同屋 | 本项目「单人养成为主」（[需求规格书 §1.1](../规格/需求规格书.md)），同时只展示当前宠物 |
| 小屋内购（真实付费） | 本项目积分为课堂虚拟分，无真实货币（[需求规格书 §1.1](../规格/需求规格书.md)） |
| cleanliness 进顶部五维条 | 五维已固定为 HP/饱食/快乐/亲密/经验（规格 §2.2），cleanliness 作隐藏状态仅角落小图标（§4.5） |
| 独立 heal API | 治疗按钮复用 rest()（§4.4），避免与 pet.js 内部 heal() 函数混淆 |
| inventory useItem 改造写饱食 | P0 保持 inventory 只加 HP/EXP，饱食写入仅走小屋 feed（§9.4） |
