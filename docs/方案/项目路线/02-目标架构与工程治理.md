# 目标架构与工程治理

> 目标是渐进收口当前 Vanilla JS，不是立即迁移 React/Vue/TypeScript。当前事实见 `AGENTS.md` 和 [架构总览](../../../docs_project/ARCHITECTURE.md)。

## 目标分层

```text
App Shell / Router
        |
Feature Controllers
        |
Domain Services
DailyState | RewardBridge | LearningAttempt | ProfileService
        |
State Gateway
StorageRegistry | LocalRepository | Migration helpers
        |
Optional SQLite Gateway（账号/家庭/孩子管理已接入；状态快照自动同步待接入）
        |
Node.js API -> SQLite / migrations / revision snapshots
```

现有 `window.*` API 先作为兼容 facade 保留。每次只把一条规则抽成可测试服务，再迁移调用点；不要同时改路由、存储、CSS 和玩法。

## 状态治理

新增机器可读 `storage-registry.json`，每个 key 至少声明：

- `owner`
- `scope`: `learner | parent | device | auth | cloud-config`
- `schemaVersion`
- `profileScoped`
- `cloudSync`
- `retention`
- `migration`
- `legacyKeys`

第一阶段只扫描和报告未登记 key、多个 owner、scope 缺失、文档漂移和潜在敏感数据；不要立刻启用白名单，以免遗漏旧用户数据。覆盖率和 Profile round-trip 达标后，再让导出/未来云快照使用孩子领域显式白名单。

## 每日状态与时间

- 业务日统一由 `PetBankDailyState.localDate()` 产生 `YYYY-MM-DD`。
- 旧 `toLocaleDateString()`/`toDateString()` 只读迁移，不新增同类格式。
- 即时时间为毫秒，审计时间为 ISO，宠物衰减秒单位只封装在 `PetSystem`。
- 每日合同带 `date + profileId`，积分余额不因日切重置。
- 跨页异步用 activation token/AbortController；timer 不能承担业务结算。

## 奖励与积分

统一奖励事件最小结构：

```js
{
  eventId,
  profileId,
  source,
  sourceId,
  rewards,
  occurredAt,
  policyVersion
}
```

`CoreRewardService`/`GameRewardReceipts` 负责合法性和去重，既有 `addGrowthPoints` 等 API 负责兼容落值。后续再演进为：

```text
grant / spend / adjust
  -> idempotent receipt
  -> balance projection
  -> history / feedback
```

不要把消费、家长调整、游戏奖励和任务奖励混成一个只允许正数的接口。重复事件、重复 iframe 消息、多 tab 和刷新必须至多生效一次。

## 学习结果

定义最小 `LearningAttempt`：

```text
learnerId, skillId, contentId, attemptedAt, correct,
hintLevel, errorTag, sessionId, contentVersion
```

先把现有短时 `mastered` 解释为 provisional，再通过跨 1/3/7 天或异题型表现记录 retained。先在汉字或英语一条链路试点；复习调度器稳定前不做首页智能推荐。

## SQLite 后端边界

当前后端已经提供：

- 配置校验和生产数据目录约束。
- SQLite WAL、外键、busy timeout。
- 追加迁移、`GET /api/v1/health`、认证/refresh、家庭/邀请码、孩子和 revision 快照 API。

未来若启动跨设备项目，下一步是把现有快照 API 接入 Profile 启动恢复、切换前 push、切换后 pull、outbox/幂等/冲突和恢复验证，再考虑社交和 PK。前端不能在这些生命周期和权限测试完成前引入“同步成功”状态。

## 模块拆分顺序

1. `app.js` -> `daily-task-service`。
2. 奖励路径 -> `reward-bridge`。
3. 裸 localStorage -> `storage-gateway`，先兼容旧 key。
4. `learn-center.js` -> `learning-session`、`review-scheduler`、`content-repository`。
5. 最后才考虑 route/controller 或类型化改造。

每一步都要：纯函数测试 -> 单模块 contract -> 旧 facade 回归 -> 浏览器核心流程；失败可以回退到旧 facade。

## CI/CD 门禁

门禁分三层：

1. 快速门禁：`node --check`、Pages workflow contract、静态路由、route base、Profile 隔离、遗留账号运行时。
2. 业务门禁：每日状态、奖励 receipt、英语 scope、核心反馈、宠物/探索/商城专项测试。
3. 发布门禁：Pages 制品组装、路由入口、资源白名单、体积/未引用报告和必要的 Playwright 流程。

依赖本地 HTTP 服务的全量回归不塞进 Pages fast gate；它由本地/发布前 runbook 执行。
