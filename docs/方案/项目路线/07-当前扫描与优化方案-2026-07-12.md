### 仍然真实存在的核心问题

1. `localStorage` 仍是散落的直接访问；Profile 依赖“全部 `petbank_*` 动态快照”，没有机器可读 registry、schema 和显式快照白名单。
2. 奖励 receipt 已覆盖主要新路径，但旧玩法、商店、宝箱和部分 iframe 桥仍保留直接余额调整/兼容 fallback，经济规则没有一个可审计的唯一入口。
3. 日期合同已统一到每日任务，但宝箱旧键、遛弯、竞技场和部分学习逻辑仍有历史日期格式；不能把“有共享日状态”误认为“全仓库时间已统一”。
4. `app.js` 约 4,411 行、`learn-center.js` 约 3,656 行，UI、状态、存储和领域规则混合；缺乏清晰边界，但不适合立即全量重写。
5. 学习 `mastered` 仍主要表达短时连续答对，尚未形成跨日保持、提示依赖、错误标签和异题型迁移的统一 `LearningAttempt`/复习合同。
6. 自托管账号/家庭/孩子/快照基础链路已实现；当前缺口是离线 outbox、后台重试、多端合并、细粒度同步状态和生产 canary，不应重复建设认证 API。
7. 工程文档曾经把 Supabase、旧模块、旧行号和旧测试列表写成当前事实；这次更新后仍要以定期扫描防止再次漂移。

## 2. 优化目标

目标不是换框架，而是把现有静态应用从“能跑的全局脚本集合”收敛成四个可验证边界：

```text
页面壳/路由
  -> feature controller（只编排 UI 和生命周期）
  -> domain service（积分、日状态、学习尝试、奖励）
  -> state gateway（localStorage + migration + registry）
  -> optional SQLite gateway（账号/家庭/孩子已接入；快照生命周期待接入）
```

每一层都必须有稳定输入/输出、失败语义和最小 contract test。保留 `window.*` 兼容 facade，先收口规则再考虑 ES module、TypeScript 或打包工具。

## 3. 分阶段方案

### P0-A：建立 Storage Registry，只审计不接管运行时

文件范围：新增 `docs_project/data-contracts/storage-registry.json` 和扫描/报告脚本，首期不改业务读写。

每个 key 声明：`owner`、`scope`、`schemaVersion`、`profileScoped`、`cloudSync`、`retention`、`migration`、`legacyKeys`。先覆盖当前 key 家族，再报告：未登记 key、多个 owner、profile scope 缺失、文档与代码不一致。

验收：扫描结果可重复；当前所有 `petbank_*` key 都有 owner 或进入明确的 legacy allowlist；不改变用户现有数据。

### P0-B：收口奖励入口

文件范围：`js/app.js`、`js/core-reward-service.js`、`js/runtime-loader.js`、`js/math-pk.js`、`js/treasure.js`、游戏桥接和对应测试。

规则：所有“增加成长分/宠物经验/物品”的新调用先提交 `{eventId, profileId, source, sourceId, rewards, occurredAt, policyVersion}`；receipt 成功后再落值。消费、家长调整和兼容回滚分为 `spend`/`adjust`，不能塞进 grant。

验收：刷新、重复消息、重复按钮、重载和两个 tab 的重复事件都至多生效一次；每个来源能查到 accepted/duplicate/rejected 原因；旧 fallback 只在核心 API 缺失时触发并有测试。

### P0-C：统一时间入口，保留旧值迁移

文件范围：`js/app.js`、`js/treasure.js`、`js/walk.js`、`js/card-arena-ui.js`、`js/learn-center.js` 和时间测试。

先复用 `PetBankDailyState.localDate()`，再为遛弯/竞技场/学习提供只读 date helper；旧 `toLocaleDateString()`/`toDateString()` 只做兼容读取，新增数据一律 `YYYY-MM-DD`。审计时间用 ISO，局内时间用毫秒，宠物衰减继续封装在 Unix 秒 API 内。

验收：同日/跨日、旧格式迁移、时区切换边界、profile 切换和 timer 取消都有纯逻辑测试。

### P1-A：拆出可测试领域服务

顺序固定为：

1. `daily-task-service`：从 `app.js` 抽出日期、任务集合、日宝箱和迁移。
2. `reward-bridge`：统一核心奖励、游戏 receipt 和旧 facade。
3. `storage-gateway`：集中 parse/write、schema fallback 和写入错误。
4. `learning-session`/`review-scheduler`：从 `learn-center.js` 抽出尝试记录和复习决策。

每次只迁移一个入口，保留旧 `window.*` API，纯逻辑先测，浏览器测试最后补。不要同时拆 `app.js`、CSS 和路由。

### P1-B：让学习结果可解释

定义最小 `LearningAttempt`：`learnerId、skillId、contentId、attemptedAt、correct、hintLevel、errorTag、sessionId、contentVersion`。把 `mastered` 暂时改称/映射为短时 `provisional`，跨 1/3/7 天或异题型达到条件后再记录 `retained`。先汉字/英语选一条链路试点，再扩数学。

验收：能回答“为什么推荐这道题”，并能计算 1 日/7 日保持率；不以积分、在线时长或总答题数代替学习效果。

### P1-C：完善自托管快照同步可靠性（产品确认跨设备后执行）

不重复建设已经存在的认证、家庭、孩子和 revision 快照 API。下一步执行：离线 outbox -> 重试/退避 -> 多端冲突合并 -> 细粒度状态提示 -> canary/恢复测试 -> 社交/PK。现有启动恢复、切换前 push、切换后 pull 和页面隐藏上传必须保持本地降级、权限测试和回滚手册。

### P2：工程治理和体验收口

- 依据真实修改频率拆 `learn-center.css`/`style.css`，先抽变量和页面边界，不做大面积视觉重写。
- 为根目录补最小 Node 工具清单、`node --check` 批量门禁、JSON/schema 检查和 secret scan；不急于引入构建链。
- 为 Pages 制品输出 route/asset manifest、总大小、最大文件和未引用资源报告。
- 统一 iframe session/source/seq、Modal、语音取消、reduced-motion、键盘和移动端验收。

## 4. 推荐执行顺序

```text
Storage Registry 报告 ─┬─> Profile/快照白名单
                       └─> 奖励入口审计 ─> GameResult/PointLedger 兼容层

共享时间入口 ──────────> daily-task-service

上述三项稳定 ─────────> learning-session/review-scheduler
                       └─> app.js / learn-center.js 渐进拆分

产品确认需要跨设备 ────> 现有 SQLite API + Profile 快照生命周期/outbox/冲突恢复
```

第一迭代只做 P0-A/P0-B/P0-C 的报告、合同和测试夹具；第二迭代再改运行时。这样能先证明边界，不把数据迁移、奖励经济和模块拆分绑成一个不可回滚的大改动。

## 5. 交付门槛

每个工作包必须同时交付：

- 代码/文档边界和不做什么。
- 兼容迁移和回滚策略。

- 与当前回归 runner/Pages fast gate 的接线决定。
- `git diff` 中没有把临时资产、密钥、数据库或用户未提交文件带进来的证据。

发布前仍按 [testing-and-release.md](../../../docs_project/runbooks/testing-and-release.md) 执行；任何全量回归失败都不能用“只是浏览器环境”直接忽略，必须区分环境前置和代码断言。
