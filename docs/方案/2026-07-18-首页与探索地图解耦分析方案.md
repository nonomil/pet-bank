# 首页与探索地图解耦分析方案

> 状态：历史方案，已被首页轮播与常用入口方案取代。当前首页不再保留森林地图宿主；森林 12 节点地图位于 `/app/explore/forest` 独立页面。本文用于记录此前的结构问题与迁移过程，不得作为当前首页 DOM 契约。

> 日期：2026-07-18
>
> 范围：首页、原森林螺旋路线、三世界像素故事地图、森林场景对话舞台
>
> 结论：当前问题不是一个 CSS 小错，而是“旧标签页状态 + 多代首页结构嵌套 + 探索容器整页覆写 + 样式所有权不清”叠加造成。

## 执行状态（2026-07-18）

此前版本的页面解耦已落地；随后首页信息架构继续收敛为轮播与常用入口：首页不再保留森林地图宿主，探索 Tab 独立承载三世界 60 节点和 20 个侦探支线，森林地图迁移到 `/app/explore/forest`，森林对话使用 `#explorationStageRoot`，故事地图使用 `#pixelStoryMapHost`。

最终验收以本方案第 9 章的验证记录为准；未通过或因环境超时的测试不得被描述为已通过。

## 1. 用户看到的四种画面分别来自哪里

| 画面 | 实际来源 | 判断 |
| --- | --- | --- |
| 旧首页先闪现 | 浏览器标签页保留了修改前的 DOM；未重新加载时仍有 `#homePixelWorldMapSlot`，且没有 `app-booting/app-ready` | 版本状态问题，不是当前 `7000` 服务指向临时目录 |
| 三栏磨砂首页 | 当前 `index.html` 的 `#page-map > .home-demo-workspace` | 正式首页的新外壳 |
| 首页下方森林地图 | 新首页内部又包裹 `.home-demo-original-content`，其中继续保留旧首页的 `#homeExplorePanel`、成长入口、轮播和仓库 | 真实的结构嵌套问题 |
| 探索页时而卡片地图、时而森林地图或对话页 | `#page-explore` 是空容器，`js/app.js` 根据全局状态反复改写 `innerHTML` | 真实的页面所有权问题 |

本轮核对 `http://127.0.0.1:7000/index.html` 与工作树内容一致：响应包含 `app-booting`，不包含 `homePixelWorldMapSlot`，包含 `sceneGridMap`。刷新浏览器后，首页同样满足：`app-ready=true`、旧首页像素插槽为 0、临时 Demo 引用为 0、森林节点为 12。因此，`7000` 服务并未加载 `tmp/pixel-home-template-demo/`；旧画面主要来自未刷新的标签页。

## 2. 当前结构为什么容易杂糅

### 2.1 首页是“新首页包着旧首页”

当前首页的主要层级是：

```text
#page-map
└─ .home-demo-workspace                 新三栏首页
   ├─ .home-demo-sidebar                新左栏
   ├─ .home-demo-main                   新中栏
   │  ├─ 今日主线
   │  ├─ 学习/绘本/游戏卡片
   │  ├─ 三世界入口卡片
   │  └─ .home-demo-original-content    旧首页整体被再次嵌入
   │     ├─ #homeExplorePanel           原森林螺旋路线
   │     ├─ #childJourneyHero
   │     ├─ .child-journey-launcher
   │     ├─ #showcase
   │     ├─ #childJourneyFeedback
   │     └─ #treasureWarehouseCard
   └─ .home-demo-right-rail              新右栏
```

这不是单纯的视觉重复。新旧两套区域都表达“今日任务、下一步、探索入口、成长状态”，并由 `renderAll()` 同时更新。结果是：

- 信息层级重复，首页越往下越像另一个页面。
- 原森林地图受到新首页网格宽度约束，卡片、节点和路线显得拥挤。
- `.home-demo-*`、`.child-journey-*`、`.home-explore-*`、`.map-*` 四套样式同时作用。
- 后续修改者很难判断某个元素是兼容层、正式层还是临时 Demo 残留。

### 2.2 探索页没有稳定 DOM，三个视图争用同一容器

`index.html` 只声明空的 `<div class="page" id="page-explore"></div>`。运行时存在三种整页写入：

1. `renderPixelStoryExplorePage()` 把容器改写为 `#pixelStoryShell` 和 `#pixelStoryMapContainer`。
2. `ensureExploreMapShell()` 把容器改写为 `#exploreLegacyContainer`、`#sceneGrid`、侦探容器。
3. `ExplorationDetail.show()` 把容器改写为 `.galgame-stage`；退出时再调用上层函数恢复地图。

因此页面不是“地图组件切换状态”，而是“一个全局容器被不同模块轮流接管”。`activeExploreSceneId`、`ExplorationDetail.isActive()`、DOM 是否存在以及 `dataset.mode` 一起决定下一次渲染，任何一处状态滞后都可能恢复错误的页面。

### 2.3 路由只知道 `explore`，不知道探索内部视图

`switchPage('explore')` 只能表达“进入探索页”，不能表达：

- 显示三世界故事地图；
- 打开原森林场景；
- 显示森林对话舞台；
- 从对话返回首页森林路线；
- 从像素章节返回原来的世界和分页。

这些信息目前由全局变量和 DOM 形状隐式推断。浏览器前进/后退、深层 URL、刷新恢复和异步加载都没有统一的视图状态契约。

### 2.4 CSS 同时保留了多代结构

静态检索发现：

- 首页旧模式、森林地图、首页像素地图相关选择器约 26 处。
- `#page-explore`、像素故事路由和故事壳相关选择器约 19 处。
- 与首页/探索容器所有权直接相关的命中共约 58 处。

其中部分选择器对应已经删除的 `homePixelWorldMapSlot` 和模式切换 UI。它们未必立即产生错误，但会形成“看不见的第二套规则”，让同一个组件在首页和探索页得到不同尺寸、背景和层级。

### 2.5 测试覆盖了功能路径，但还没有覆盖页面所有权

本轮已通过：

```text
node scripts/test-explore-mode-contract.mjs
node scripts/test-home-explore-layout.mjs
node scripts/test-exploration-entry-browser.mjs
node scripts/test-home-browser.mjs
node scripts/test-pixel-story-map-layout-browser.mjs
```

已有测试证明当前工作树的临时行为可用：

- 首页保留 12 个森林路线节点。
- 探索页显示三个故事世界，不显示森林路线节点。
- 森林场景进入后可退出到故事卡片地图。
- 移动端没有横向滚动，浏览器错误为 0。

但测试仍有三个空白：

- 没有断言首页只能存在一套“今日任务/下一步/成长进度”信息架构。
- 没有禁止 `#page-explore.innerHTML = ...` 的所有权破坏。
- 深层打开 `/app/explore` 时，`DOMContentLoaded` 后会先看到空背景，直到全局初始化完成才出现故事地图；没有稳定的加载骨架断言。

## 3. 目标结构

本方案不新增顶栏入口，也不改变“首页保留森林路线、探索 Tab 展示卡片地图”的产品语义。只重新划分 DOM 和模块所有权。

```text
#page-map                             首页唯一页面
├─ #homeDashboardSidebarHost          左侧任务摘要
├─ #homeDashboardMainHost             中间今日主线与入口卡片
├─ #homeForestRouteHost               原森林螺旋路线，独立组件
└─ #homeDashboardProgressHost         右侧成长进度

#page-explore                         故事漫游唯一页面
├─ #pixelStoryMapHost                 三世界卡片式地图
├─ #pixelStoryChapterHost             像素章节对话，默认隐藏
└─ #exploreLoadingState               深层入口加载骨架

#explorationStageRoot                 原森林场景独立舞台
└─ .galgame-stage                     不再覆写 #page-explore
```

所有权规则：

| 宿主 | 唯一写入者 | 允许内容 | 禁止内容 |
| --- | --- | --- | --- |
| `#page-map` | 首页编排器 | 三栏首页和森林路线宿主 | 像素故事地图、对话舞台 |
| `#homeForestRouteHost` | 森林路线组件 | 12 节点螺旋地图 | 三世界 Tab、侦探奖励、故事对白 |
| `#page-explore` | 故事探索页编排器 | 故事地图、像素章节、加载态 | 原森林螺旋节点 |
| `#explorationStageRoot` | `ExplorationDetail` | 原森林场景对白、选择、战斗衔接 | 首页和故事地图整体 HTML |

## 4. 状态与返回规则

使用显式视图状态替代“看 DOM 猜状态”：

```js
{
  page: 'map' | 'explore',
  view: 'home' | 'story-map' | 'pixel-chapter' | 'forest-scene',
  returnTarget: 'home-forest-route' | 'story-map',
  worldId: null | 'sci-fi' | 'forest' | 'block' | 'detective',
  chapterId: null | string,
  sceneId: null | string
}
```

返回规则固定为：

- 首页森林节点 → `forest-scene` → 返回 `home-forest-route`。
- 探索卡片节点 → `pixel-chapter` → 返回相同 `worldId` 和分页。
- 顶栏“探索” → 永远进入 `story-map`，不继承上一次森林场景的 DOM。
- 浏览器刷新 `/app/explore` → 先显示探索加载骨架，再进入 `story-map`。
- 离开探索相关页面 → 统一停止语音、计时器和未完成动画，不靠旧容器被覆盖来清理。

## 5. CSS 拆分建议

| 文件 | 职责 |
| --- | --- |
| `css/home-dashboard.css` | 首页三栏、磨砂面板、响应式布局 |
| `css/forest-route-map.css` | 原森林螺旋地图、节点、路线、解锁态 |
| `css/pixel-story.css` | 三世界卡片地图与像素章节 |
| `css/exploration-stage.css` | 原森林对话舞台、选项、战斗衔接 |
| `css/style.css` | 只保留通用壳、导航、基础 token 和尚未迁移模块 |

拆分时先复制到专属文件并通过截图回归，再删除旧选择器。禁止一次性重命名全部类名；先按宿主作用域收口，再做机械清理。

## 6. 实施优先级

### P0：先阻止继续混写

- 为首页、森林路线、故事地图和对话舞台建立固定宿主。
- 增加测试，禁止任何探索模块直接改写 `#page-explore.innerHTML`。
- 为深层 `/app/explore` 增加可见加载骨架。
- 把返回目标作为显式参数传给场景详情。

### P1：再拆编排和样式

- 从 `app.js` 移出首页视图拼装和探索页视图切换。
- 把森林路线 CSS、首页 CSS 从 `style.css` 分离。
- 删除已经没有 DOM 对应物的首页像素地图和旧模式切换样式。

### P2：最后收口路由和视觉回归

- 为故事世界、分页和章节增加可恢复的 URL/历史状态。
- 增加桌面、平板、手机三组稳定截图基线。
- 运行全量回归与 Pages 制品门禁，再更新发布文档。

## 7. 完成标准

- 首次打开和刷新均不会出现旧首页壳或空白探索页。
- 首页 DOM 中只有一套任务、主线和成长进度信息架构。
- 首页森林路线固定为 12 节点螺旋地图，探索页内森林路线节点数量固定为 0。
- 探索页固定显示三世界卡片地图和侦探支线入口。
- 进入任一森林场景不会删除故事地图 DOM；退出能回到明确来源。
- 代码中不存在 `pageExplore.innerHTML = ...`。
- 删除无对应 DOM 的旧选择器后，桌面和手机截图无视觉回退。
- 聚焦测试、全量回归、Pages 制品和深层路由测试全部通过。
