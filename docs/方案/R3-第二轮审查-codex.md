# R3 第二轮审查（Codex）

审查对象：`docs/方案/2026-06-30-R3-数学场景化-方案.md`  
范围：只读审查，不改代码、不改 stories。

## ✅通过

- **总体方向与 R3 主目标一致**：方案稿明确采用“数据改动为主 + `genMathQuestion` 小改”的收口方式，符合需求对 R3“只改数据 + 少量字段，不改引擎核心”的约束。证据：[需求 §3 R3](../plans/2026-06-30-叙事收口与文案优化-需求.md:63) 、[方案 2.1/3](./2026-06-30-R3-数学场景化-方案.md:20)。
- **5 章题型主线与设计稿大体一致**：方案稿的章节题型分布 `ch1 arithmetic / ch2-3 word / ch4 arithmetic / ch5 logic` 与设计稿 §3.3 一致。证据：[设计稿 §3.3](../plans/2026-06-30-叙事收口与文案优化-设计稿.md:74) 、[方案 2.2](./2026-06-30-R3-数学场景化-方案.md:36)。
- **运行时改法总体可行**：当前 `genMathQuestion` 已按 `arithmetic / word / logic` 分流，`showNextEvent` math 分支也已单点渲染；方案只是在这两处加 event 优先题与样式判断，改动面可控。证据：[js/exploration-detail.js](../../js/exploration-detail.js:247) 、[js/exploration-detail.js](../../js/exploration-detail.js:345)。
- **12 题基础质量大体合格**：方案表中 12 题均有场景对象、答案均不超过 50、都提供了包含正确答案的 4 选项，整体满足儿童题面基础边界。证据：[方案 2.3](./2026-06-30-R3-数学场景化-方案.md:46)。

## ⚠️问题

- **`chapter_skill` 写入位置与设计稿/需求冲突**：设计稿与需求都指向“每个场景 JSON 顶层元数据增加 `chapter_skill`”，但方案稿把它写进了 math 事件示例，并在改动范围中写成“每个 math 事件加 `chapter_skill`”。这会导致元数据语义漂移，也不利于后续按场景做审查、配图、统计。证据：[设计稿 §3.3/§4](../plans/2026-06-30-叙事收口与文案优化-设计稿.md:90) 、[设计稿数据结构](../plans/2026-06-30-叙事收口与文案优化-设计稿.md:135) 、[需求 R3(a)](../plans/2026-06-30-叙事收口与文案优化-需求.md:70) 、[方案 2.4/3](./2026-06-30-R3-数学场景化-方案.md:72) 。
- **第 3 章固定题里混入乘法语义，和教育目标口径不稳**：设计稿与需求把第 2-3 章都定义为 `word` 型低龄应用题，其中第 3 章强调“买卖 / 分配 / 剩余”；但方案稿 `waterfall` 用了“3 片荷叶上各有 2 颗露珠”，本质是乘法分组计数，更接近第 4 章“乘除与机关”。这不影响 `mathType` 字段，但会让章节认知路径变模糊。证据：[设计稿 §3.3](../plans/2026-06-30-叙事收口与文案优化-设计稿.md:82) 、[需求 R3(a)](../plans/2026-06-30-叙事收口与文案优化-需求.md:75) 、[方案题表 waterfall](./2026-06-30-R3-数学场景化-方案.md:55)。
- **方案稿与需求示例存在 2 处题型/题面分歧，执行前需冻结口径**：需求示例里 `candy.math` 是“2 串草莓糖每串 3 颗”的乘法应用，`stargarden.math` 是 `2,5,8,11,?`；方案稿改成了 `candy` 减法剩余题，且把等差规律放到 `space`，`stargarden` 改成斐波那契。方案稿内部自洽，但和需求示例并不一致，若不先冻结会在 plan/task 阶段反复返工。证据：[需求 R3(b)](../plans/2026-06-30-叙事收口与文案优化-需求.md:81) 、[方案 2.3 candy/space/stargarden](./2026-06-30-R3-数学场景化-方案.md:53)。
- **“零回归”表述略过了 `word` 回退的既有前提**：当前 `genMathQuestion` 只有在 `CMATH_POOL` 已加载时才会走 `word` 题库，否则会落回算式题；方案稿说“无 `question` 时回退 arithmetic/word-CMATH/logic”，方向正确，但未显式注明 `word` 回退依赖 CMATH 池可用。这不是新回归点，但应在实施说明里写清楚，避免误以为 `word` 一定稳定回到应用题。证据：[js/exploration-detail.js](../../js/exploration-detail.js:247) 、[方案 2.1/4](./2026-06-30-R3-数学场景化-方案.md:20)。
- **配音与文本映射风险只被隐含提到，未转成执行约束**：当前 `VoiceSystem` 依赖 `#galgameText` 文本变化做自动播报，新增 `question` 会直接改变播报文本；方案稿提到“配音友好”，但没有把“math 事件 `id` 不改、文本改后需补语音映射”写成明确执行项。证据：[data/stories/README.md](../stories/README.md:23) 、[当前 math 事件 id 现状](../../data/stories/forest.json:24) 、[方案风险](./2026-06-30-R3-数学场景化-方案.md:121)。

## 💡建议

- **先修正文档口径，再下发 task**：把 `chapter_skill` 明确为“场景顶层字段”，math 事件只补 `question / answer / options / text`。这是一条必须先收口的结构约束。
- **把第 3 章固定题统一成加减应用/分配语义**：尤其建议重写 `waterfall`，避免“3 组每组 2 个”这种乘法感过强的题面；保持第 4 章再集中进入乘除/机关。
- **在方案里显式锁定“事件 id 不改”**：保留 `forest.math2` 这类 ID，只增字段不改键名，降低配音映射、审查引用、后续 diff 成本。证据：[当前 JSON 结构](../../data/stories/forest.json:24)。
- **把渲染约束写成一句实现备注**：`useWordStyle = !!event.question || event.mathType === 'word'`，意图更清晰；同时说明“固定长题面一律走 `.galgame-word`，仅裸算式保留 `.galgame-math`”。证据：[当前 math 渲染](../../js/exploration-detail.js:349) 、[方案渲染段](./2026-06-30-R3-数学场景化-方案.md:95)。
- **保留 CMATH 题库要写成验收项，不只写边界**：建议明确验收为“无 `question` 的 `word` 事件在 CMATH 池可用时仍抽 CMATH，池不可用时行为与当前一致”，这样 plan/task 更容易验证。证据：[方案验收 4](./2026-06-30-R3-数学场景化-方案.md:116) 、[当前 CMATH 分流](../../js/exploration-detail.js:249)。

## 总体结论

**需修订后执行。**

当前方案的主思路是对的，运行时改法也足够轻，已经接近可直接进入 plan/task。阻塞点主要有 2 个：

1. `chapter_skill` 的写入层级必须先统一到“场景顶层元数据”。
2. 第 3 章固定题要避免提前滑向乘法章目标。

这两点修完后，方案可以进入执行；其余项属于实施备注和验收补强，不构成阻塞。
