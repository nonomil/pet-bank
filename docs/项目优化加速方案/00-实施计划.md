# 项目优化加速实施计划

> **For Codex：** 执行本计划时按阶段读取并执行，每个阶段完成后运行对应验收命令，再进入下一阶段。

**Goal:** 将 PetBank 从“一个包含全部功能和资源的静态 SPA”逐步治理为“共享核心 + 按玩法发布的网页应用”，把首屏和单玩法首次进入控制在可接受范围。

**Architecture:** 保留 Vanilla JS、同源 localStorage、Profile 快照和现有 Node/SQLite API。先通过 manifest 和 runtime loader 收口资源，再将主站、学习中心和游戏中心拆为独立入口/制品；大型公开媒体使用版本化对象存储和 CDN，动态 API 保持同源或独立域名且不缓存。

**Tech Stack:** Vanilla JS、静态 HTML/CSS、GitHub Pages 或 Nginx、COS/OSS + CDN、Node.js、Playwright、Node test。

---

## 执行顺序

### Task 1: 建立可重复基线

**Files:**
- Read: `index.html`
- Read: `js/runtime-loader.js`
- Read: `scripts/assemble-pages-artifact.mjs`
- Read: `scripts/audit-pages-artifact-size.mjs`
- Create: `scripts/measure-page-load.mjs`（若现有测试入口不足）

**Steps:**
1. 记录首屏 HTML、公共 JS/CSS、首屏图片、资源请求数和 `transferSize`。
2. 记录 `today`、`learn`、`playground`、`explore` 的首次进入资源。
3. 记录发布制品总量、目录分布、扩展名分布、Top 30 大文件和重复组。
4. 将结果写入 `docs/项目优化加速方案/01-基线与指标.md` 和 `07-执行记录.md`。

**Exit:** 同一命令可以在本地 HTTP 服务和组装制品上重复得到报告；不能用仓库总大小代替页面指标。

### Task 2: 首屏低风险减负

**Files:**
- Modify: `index.html`
- Modify: `css/style.css`（只处理首屏阻塞和隐藏页规则）
- Test: `scripts/test-static-route-entries.mjs`

**Steps:**
1. 将隐藏页面的非关键 `<img>` 改为进入页面后设置 `src` 或使用可靠的 lazy loading。
2. 为首屏之外图片补齐尺寸、低优先级和失败占位。
3. 保持主站首页可见入口的图片视觉不变。
4. 运行静态路由、页面壳层和浏览器首页测试。

**Exit:** 根首页不再下载隐藏玩法卡片和非当前页面的大图；首屏图片预算下降，页面无新增 404。

### Task 3: 按玩法建立发布 manifest

**Files:**
- Modify: `scripts/assemble-pages-artifact.mjs`
- Create/Modify: `scripts/runtime-asset-manifests/`
- Modify: `data/learn/**/manifest.json`
- Test: `scripts/test-pages-artifact-size-audit.mjs`

**Steps:**
1. 以浏览器 Network 请求和代码入口为证据登记资源，不按目录名猜测。
2. 学习机、像素探险、打字防线、Minecraft 词汇、像素故事分别建立清单。
3. HTTP/HTTPS 只发布 JSON 正式数据；`file://` 回退脚本不进入默认发布路径，除非合同明确要求。
4. 音频按章节/词包发布，禁止整目录自动带入。
5. 让制品审计输出每个 manifest 的文件数和体积。

**Exit:** 每个玩法可以独立说明“入口脚本、数据、图片、音频、回退资源”；总制品满足当前阶段预算，缺资源会在构建期失败。

### Task 4: 运行时和缓存优化

**Files:**
- Modify: `js/runtime-loader.js`
- Modify: `js/app.js`
- Modify: `js/inventory.js` / `js/treasure.js`（如确认重复数据请求）
- Test: `scripts/test-runtime-loader-retry.mjs` and browser smoke tests

**Steps:**
1. 页面 bundle 按并行安全边界加载，保留依赖脚本的顺序约束。
2. 公共 JSON 使用单一缓存 Promise，避免同一页面重复请求。
3. 只预取用户下一步高概率进入的页面，不预取整类玩法。
4. 生产资源使用版本目录和长期缓存；HTML 使用短缓存。

**Exit:** 无重复公共 JSON 请求；页面切换仍满足 lifecycle token、奖励和 Profile 约束。

### Task 5: CDN/对象存储灰度

**Files:**
- Modify: `js/runtime-loader.js`
- Modify: `js/page-router.js`
- Modify: `scripts/assemble-pages-artifact.mjs`
- Create: `ops/static-release/` deployment notes if needed

**Steps:**
1. 先只把大图片、音频和大型公开 JSON 上传对象存储。
2. 使用 `v<release>/` 版本目录，不覆盖已发布资源。
3. 验证 MIME、CORS、Range、缓存命中、404 和回滚。
4. 通过配置切换同源基址/CDN 基址，默认仍可本地运行。

**Exit:** CDN 故障时同源 fallback 可用；API、token、Profile 数据没有公共缓存；上一版本可恢复。

### Task 6: 应用入口拆分与 PWA

**Files:**
- Create: `app/main/`, `app/learn/`, `app/arcade/` entry shells
- Modify: `scripts/assemble-pages-artifact.mjs`
- Create: `manifest.webmanifest`, `service-worker.js`（在前述指标稳定后）

**Steps:**
1. 先按同一域名路径拆制品，不立即拆账号和数据协议。
2. 主站保留任务、积分、宠物和家长主线。
3. 学习中心和游戏中心拥有自己的 HTML、CSS、JS 和资源 manifest。
4. PWA 只缓存核心壳和用户主动选择的学习/游戏包，不缓存全部资源。

**Exit:** 三个入口可独立发布和回滚；Profile、积分、奖励 receipt 和离线状态仍兼容。

### Task 7: 平台决策复核

**Decision rule:**

- 网页 + CDN + PWA：默认路线。
- 微信小程序：只有微信内分发成为主要入口，并接受重写页面/存储/资源适配时启动。
- Android APK：先用 PWA/TWA/Capacitor 验证需求；只有设备能力或应用商店需求成立才原生重写。
