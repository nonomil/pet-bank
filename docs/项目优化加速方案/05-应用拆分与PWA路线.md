# 应用拆分与 PWA 路线

## 推荐产品边界

### 主站 `/app/`

今日任务、积分、宠物、小屋、遛弯、商城、背包、家长设置和成长复盘。这里承担每天打开的核心循环。

### 学习中心 `/learn/`

学习包、学习单、打印、英语、中文识字、词库和学习音频。学习资料按 pack/module 进入资源 manifest。

### 游戏中心 `/arcade/`

数学 PK、汉字游戏、打字防线、像素探险、卡牌对战和独立小游戏。单个玩法可以独立回滚和观测。

### 已独立项目

Anki Minecraft 词卡工作台和自托管 Node/SQLite 后端继续保持独立，不重新并入主站。

## 拆分原则

- 先拆入口和制品，再拆代码仓库。
- 三个应用保持同一主域名路径，继续共享 `localStorage`。
- 共享核心只提供 Profile、积分、奖励、时间、资源基址和事件契约。
- 不把三个应用的 UI 和大型内容再次打包成一个公共 bundle。
- 不为每个小游戏创建独立后端或微服务。

## PWA

PWA 应在资源分包稳定后实施：

1. Web App Manifest 提供 Android 桌面安装入口。
2. Service Worker 只缓存核心壳、公共 JS/CSS 和用户选择的 pack。
3. 图片、音频和大词库采用用户主动下载，不默认缓存全部内容。
4. Profile 和奖励仍由现有 localStorage/API 契约负责。
5. 缓存版本与 release 版本绑定，升级失败可回退。

## 小程序决策

只有满足以下条件才启动小程序项目：

- 微信内访问成为主要入口。
- 接受页面、存储、音频、iframe 和资源加载的重写。
- 已有稳定的后端 API 和登录授权方案。
- 有明确的小程序专属分发或支付需求。

小程序不能作为当前首屏资源问题的替代修复。

## Android 决策

先验证 PWA/TWA/Capacitor。原生 Android 只有在需要系统通知、文件管理、设备能力、商店分发或完整离线包时才立项。WebView 包装本身不会减少图片、音频和 JSON 的体积。
