## 2. P0 缺口（必须先修，阻断稳定演示）

> P0 定义：阻断「稳定演示一整套成长-探索-消费流程」的缺口。评估报告原话：「真正稳定串起来的只有任务积分、宠物基础养成和背包主链路」「切入核心页时会遇到运行时错误」。

### P0-1 宠物小屋 / 家园 + 衰减闭环 + 救援 + 四互动按钮

> ⚠️ **范围扩大（2026-06-29）**：见 [change-request-001](../方案/change-request-001-宠物小屋P0范围扩大.md)。原 P1-1（衰减闭环）、P1-2（倒下 / 救援）及 feed/play/rest 改造、bath 新 API、EXP 5 维常驻全部并入本条 P0。

- **现象**：当前 Tab 体系里有「宠物」页（宠物详情、喂食 / 玩耍 / 休息 / 装备），但没有独立的「宠物小屋 / 家园」空间。原始需求明确写：「这个文档，还有很多地方，没有实现，宠物小屋，宝箱，卡片，遛弯，互动，对战……」。同时 `js/pet.js` 缺 hunger / intimacy 自然衰减、缺倒下 / 救援状态机，互动按钮未含洗澡。
- **根因**：需求侧已点名家园，但前期优先铺开主链路页面与基础模块，未把「家园承载」作为独立模块率先立项；`js/pet.js` 最初仅承载宠物实体状态，未承载「空间 / 摆设」，也未实现时间驱动的衰减 tick 与健康状态机。
- **影响**：这是评估报告认定的「与原始需求最明显的一块差距」；缺少家园承载会让养成-探索-消费循环缺少「归宿感」落点。缺少衰减 + 救援会让养成缺少「需要持续关怀」的张力，守护力反馈链路断裂。
- **涉及文件**：
  - 新增 `js/home.js`（建议）或扩展 `js/pet.js`
  - `index.html`（新增 Tab + 容器节点 + 四互动按钮 + 顶部 5 维状态条）
  - `js/app.js`（`init()` 注入 HomeSystem 初始化 + 注册 home tab 到 nav 数组 + switchPage case）
  - `js/pet.js`（hunger / intimacy / last_home_ts 字段 + decay API + 倒下 / 救援状态机 + feed/play/rest/bath 改造）
  - `js/inventory.js`（useItem 回归：feed 改造后调用链依赖）
  - 数据 / 资产：可能需 `data/furniture.json`、`assets/home/`
- **修复方向**：
  1. 家园 MVP：一个固定布局的小屋 + 宠物精灵站立 + 2-3 个可摆放家具槽位；
  2. **衰减闭环**：扩展宠物状态 schema 加 `hunger`、`intimacy`、`last_home_ts`；`last_home_ts` 单一归属 `PetSystem.state.last_home_ts`（home.js 离开时写），加 `if null skip` 守卫 + `hours = max(0, floor(...))` 防时间篡改；在线 setInterval + 离线按小时结算（v2：每小时饱食 -2 / 快乐 -1）；
  3. **倒下 / 救援**：状态机加「健康 → 倒下 → 救援 → 健康」分支，倒下时显示救援 CTA + 复活动画，承接 `data/items.json` 的 `ancient_bandage`、`revive_potion` 等治疗类物品；
  4. **四互动按钮改造**：feed / play / rest 改造为写 hunger / intimacy / exp（feed 口径「饱食+25 / 经验+10 / 积分-10」对齐规格书 §5.3，并附 inventory useItem 回归测试点）；**新增 bath（洗澡）API** 作为第 4 互动按钮（与喂食 / 玩耍 / 洗澡 / 治疗四按钮映射，治疗 = rest 还是独立 heal 见方案裁决）；
  5. **EXP 5 维常驻**：顶部状态条 HP / 饱食 / 快乐 / 亲密 / 经验 5 维常驻（不降级到次级面板），饱食 / 亲密无数据源时显示占位；
  6. 与商店打通：家具作为独立装饰目录，购买后写入小屋 ownership；本阶段不引入 buff、不并入背包物品表；
  7. 与 [../设计/模块清单与接口.md](../设计/模块清单与接口.md) 中宠物状态字段对接。
- **当前代码状态**：**已落地**。`home.js` 已存在并接入启动流程；`pet.js` 已包含 `bath()`、`decay()`、`markHomeExit()`、`revive()`；当前剩余缺口转为 **商店家具联动（AC-19 / AC-20）**、长期病症与更深层家园玩法。
- **优先级**：P0（评估 §5 第一项；2026-06-29 范围扩大）。
- **验收**：
  1. 小屋 Tab 可打开、宠物可见、至少 1 件家具可摆放并持久化；
  2. home tab 在 app.js nav 数组注册 + switchPage case 可达（列为 AC）；
  3. 在线挂机 1 小时后 hunger -2、intimacy -1；离线补算正确；`last_home_ts` 为 null 时 skip 不报错；
  4. hunger=0 持续触发倒下，倒下后救援 CTA + 复活动画可用，治疗类物品可恢复；
  5. feed 改造后口径「饱食+25 / 经验+10 / 积分-10」生效，inventory useItem 回归通过；
  6. bath API 可用，洗澡后对应维度（清洁度 / 快乐）变化；
  7. 顶部状态条 5 维常驻显示。

### P0-2 CardCollection.init() 启动接线回归跟踪

- **现象**：评估时 `js/card-collection.js` 的图鉴模块功能存在，但 `init()` 没有接到 `app.js` 的启动流程，导致模块不生效。
- **根因**：模块作者写了独立的 `init`，但启动装配遗漏。
- **影响**：图鉴首次渲染 / 数据加载 / 收集进度不可靠，进入图鉴 Tab 可能显示空或未初始化状态。
- **涉及文件**：`js/app.js`（init 装配）、`js/card-collection.js`。
- **修复方向**：在 `app.js` 的 `init()` 中按「存在性 + 类型守卫」装配，例如 `if (window.CardCollection && typeof window.CardCollection.init === 'function') window.CardCollection.init();`。
- **当前代码状态**：**已补丁接入**（核实于 `js/app.js:1145-1147`）。建议保留为回归用例，避免回退。
- **优先级**：P0。
- **验收**：冷启动后图鉴 Tab 数据已加载，控制台无 `CardCollection is not defined` 类报错。

### P0-3 ToolboxSystem.init() 启动接线回归跟踪

- **现象**：评估时工具箱 UI 能打开，但持久化 `init()` 没有被调用，导致刷新后状态丢失。
- **根因**：同 P0-2，启动装配遗漏。
- **影响**：随机点名 / 番茄钟的会话状态在刷新后丢失，体验不可用。
- **涉及文件**：`js/app.js`、`js/tools.js`。
- **修复方向**：在 `init()` 中调用 `ToolboxSystem.init()`。
- **当前代码状态**：**已补丁接入**（核实于 `js/app.js:1148-1150`）。建议保留为回归用例。
- **优先级**：P0。
- **验收**：刷新后番茄钟累计时长 / 随机点名历史能持久化。

### P0-4 数学 PK 积分回写回归跟踪

- **现象**：评估时数学 PK 游戏本体可玩，但得分没有回写到全局总成长分账本，积分链路断裂。
- **根因**：结算逻辑只写了本地最高分（`localStorage`），未调用全局积分写入 API。
- **影响**：玩家在数学 PK 赚的积分不进入「六大成长维度 / 积分账本」主链路，破坏「学习 → 成长」的核心循环。
- **涉及文件**：`js/math-pk.js`（`_endGame` 结算）、`js/app.js`（`addGrowthPoints` 入口）。
- **修复方向**：结算时调用 `window.addGrowthPoints(earnedPoints)`，并 fallback 到 `window.totalPoints += earnedPoints` + `saveAppState()` + `updateStats()`。
- **当前代码状态**：**已补丁回写**（核实于 `js/math-pk.js:363-370`，走 `window.addGrowthPoints` 主链路）。建议保留为回归用例。
- **优先级**：P0。
- **验收**：完成一局数学 PK 后，主页总成长分 / 相关维度可见增长且刷新后持久。

### P0-5 遛弯奖励 item id 对齐回归跟踪

- **现象**：评估时遛弯 (`js/walk.js`) 的事件会投放不存在的 item id，奖励链路报错。
- **根因**：事件表里的物品 id 没有和 `data/items.json` 对齐（拼写错 / 占位 id）。
- **影响**：`InventorySystem.addItem` 拿到未知 id 会静默失败或抛错，玩家看不到奖励，HP / EXP 之外道具奖励链路断裂。
- **涉及文件**：`js/walk.js`、`data/items.json`。
- **修复方向**：把遛弯事件里所有道具奖励改为 `items.json` 中真实存在的 id。
- **当前代码状态**：**已对齐**（核实于 `js/walk.js:22-33`，使用 `toy_ball`、`chest_fragment`、`yarn`、`bone`，均在 items.json 中存在）。建议保留为回归用例。
- **优先级**：P0。
- **验收**：遛弯触发道具奖励后，背包内对应物品数量 +1，控制台无报错。

### P0-6 宝箱任务计数写入回归跟踪

- **现象**：评估时日常宝箱领取条件依赖 `petbank_tasks_completed_today`，但这个计数没有写入点，导致条件永远不满足。
- **根因**：任务完成回调里没有同步写 `localStorage.setItem('petbank_tasks_completed_today', ...)`。
- **影响**：日常宝箱无法正确触发 / 领取，玩家完成任务的奖励闭环断了。
- **涉及文件**：`js/app.js`（`toggleTask` / `saveAppState` 等任务变更点）、`js/treasure.js`（读取方）。
- **修复方向**：在所有任务完成 / 取消的写入点同步更新该计数。
- **当前代码状态**：**已多处写入**（核实于 `js/app.js:135/149/300`，`js/treasure.js:43` 读取）。建议保留为回归用例。
- **优先级**：P0。
- **验收**：勾选 N 个任务后，`petbank_tasks_completed_today` 等于 N，日常宝箱条件可达成。

### P0-7 积分回写主链路一致性（系统性）

- **现象**：评估报告 §4 原话：「探索、图鉴、工具箱、宝箱、数学 PK 这些模块都已经存在，但彼此之间仍有不少接线问题」。这是 P0-2 ~ P0-6 的系统性归纳。
- **根因**：模块化推进过快，启动装配 / 积分写入 / 物品 id 三类「接线契约」没有统一守卫。
- **影响**：任何一处接线断裂都会让对应模块的积分 / 奖励不进入主链路，破坏演示闭环。
- **涉及文件**：`js/app.js`（启动装配中枢）、各模块 `*.js`、`data/*.json`。
- **修复方向**：
  1. 建立「启动装配清单」：所有模块 `init` 必须在 `app.js:init()` 中显式调用，建议加类型守卫；
  2. 建立「积分写入单一入口」：所有模块一律走 `window.addGrowthPoints(delta)`，禁止各自直接改 `totalPoints`；
  3. 建立「物品 id 校验」：所有奖励投放点在 `addItem` 前校验 id 存在性，未知 id 记 warn 不抛错；
  4. 加冒烟测试覆盖 12 个功能模块冷启动。
- **当前代码状态**：P0-2/3/4/5/6 多数已补丁，但「系统性守卫」（启动清单 / 单一积分入口 / id 校验 / 冒烟测试）尚未固化。**这条仍是真正待办**。
- **优先级**：P0（与 P0-1 并列 M0 最高优先）。
- **验收**：见 M0 验收标准。

## 3. P1 缺口（养成深度与体验）

> P1 定义：不阻断演示，但决定「守护力 / 战斗」是否够深，影响长期留存与参考方案对齐。
>
> 📌 **2026-06-29 变更**：原 P1-1（衰减闭环）、P1-2（倒下 / 救援）已并入 P0-1（见 [change-request-001](../方案/change-request-001-宠物小屋P0范围扩大.md)）。本节仅保留战斗相关 P1 项（P1-3 / P1-4）。

### P1-3 战斗缺技能面板

- **现象**：评估 §3 原话：「战斗系统目前能跑通探索 → 遭遇 → 结算，但技能面板……还可以再补」。
- **根因**：战斗引擎当前是基础回合，没有暴露技能选择 UI。
- **影响**：战斗深度不足，装备 / 技能成长线无法体现。
- **涉及文件**：`js/exploration.js`（或独立 `js/battle.js`）、`index.html`（技能槽 UI）、与宠物等级 / 装备联动。
- **修复方向**：定义技能 schema（名称 / 消耗 / 效果 / 解锁等级），渲染技能槽，回合内可选技能。
- **优先级**：P1。
- **验收**：战斗中可选至少 2 个技能，技能消耗与效果生效。

### P1-4 战斗缺道具快捷栏

- **现象**：同 P1-3 来源，战斗中无法快捷使用背包道具（如药水）。
- **根因**：战斗 UI 与背包系统未在战斗态打通。
- **影响**：`potion_small` / `potion_large` 等消耗品在战斗中无法使用，价值降低。
- **涉及文件**：`js/exploration.js`、`js/inventory.js`（战斗态使用入口）、`index.html`。
- **修复方向**：战斗 UI 加快捷栏，绑定背包消耗品类，使用后调 `InventorySystem.useItem`。
- **优先级**：P1。
- **验收**：战斗中可从快捷栏使用药水并回血。

## 4. P2 缺口（打磨）

> P2 定义：体验收口、对齐性、口径统一类问题，不影响主流程跑通。

### P2-0 商店家具联动（纯装饰版，推荐下一步）

- **现象**：宠物小屋已经有基础家具、摆放槽位和 `HomeSystem.addFurniture()`，但商店里还没有“家园装饰”分类，买不到家具，也就无法形成“消费 -> 小屋展示”的闭环。

## 4.5 测试发现（solution-loop 自动化回归）

> 本节收录 solution-loop 阶段自动化测试（playwright / 冒烟）暴露、已独立修复的缺陷，作为回归追溯与防回退记录，不归入 P0/P1/P2 主干。

### TD-1 首页成长地图连线层级 bug ✅ 已修复

> 来源：[change-request-002](../方案/change-request-002-首页地图连线层级bug.md)。solution-loop 阶段 11 final gate 的 playwright 测试发现，DeepSeek V4 Pro 独立审查定位根因。

- **模块**：探索（`js/exploration.js`，`#page-map` 首页成长地图）
- **现象**：`.map-route-line` polyline 路线连线**完全不可见**。不是语法（points 已在 CR-001 修复）、不是几何（`getTotalLength`=392）、不是 stroke 样式（强制 `stroke=red stroke-width=8 opacity=1` 仍不可见）。
- **根因**：SVG 层级被节点覆盖。1）`.map-scene-node` 有 `transform: translate(-50%,-50%)` 创建独立 stacking context；2）DOM 顺序 `board.innerHTML = buildRouteSvg() + nodes` → 节点后插入绘制在 SVG 之上；3）节点 `--node-size:198px` 在 530×980 board 上几何覆盖 polyline；4）即便给 SVG `z-index:9999` 也无法穿透同父级后续兄弟的 transform 层。
- **修复**：方案 B（主）翻转 DOM 顺序，SVG 后插入（`exploration.js:178`）；方案 A（兜底）`.map-route-svg` 显式 `z-index:5`（`style.css:285-293`）。SVG 已有 `pointer-events:none`，置顶不挡节点点击。
- **状态**：✅ 已修复（CR-002），playwright 验证通过（强制红色连线可见、几何回归、`svgZ="5"` 断言生效、console 仅剩 1 个无关 404）。
- **视觉 trade-off**（已知可接受）：连线绘制在节点卡片之上（穿过节点中心），这是地图路线常见呈现。若后续设计要求连线在节点之下露出，需缩小 `--node-size`（方案 C，本期不做）。
- **涉及文件**：`js/exploration.js:178`（DOM 顺序）、`js/exploration.js:107-119`（buildRouteSvg，CR-001 points 修复）、`css/style.css:285-293`（z-index:5）、`css/style.css:293-307`（stroke 规范化，CR-001）。
- **引用**：[../方案/change-request-002-首页地图连线层级bug.md](../方案/change-request-002-首页地图连线层级bug.md)
