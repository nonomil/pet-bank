# Design to Code（设计图还原代码 + 素材拆分）

把"一张图"变成可用的产出——可运行的前端代码，或者独立的透明PNG/分层PSD素材。核心原则是**反复对比、工程化保证结构可控**，不是生成一次就交差。

## 先判断走哪条分支

拿到图片后先判断用户真正想要什么，两条分支可以独立触发，也可以先拆图再还原代码：

- **分支A：设计图还原代码**——用户想要页面/组件的前端实现（React/Vue/HTML）。走下面"分支A"流程。
- **分支B：素材拆分**——用户想要把一张多联图/合成图（比如GPT Image一次性生成的多个UI元素拼在一张图里）拆成独立的透明PNG，或进一步合成可编辑的分层PSD。走下面"分支B"流程。
- 两者可以串联：先用分支B把图拆成干净的图层素材，再用分支A把这些素材还原成代码（比图片直接截图还原更精确，因为背景、卡片、图标已经是独立透明PNG，可以直接当作background-image/icon资源）。

---

## 分支A：设计图还原代码

```
[网页端 GPT Image 生图] -> 下载图片 -> [CC 读图+还原代码] -> [CC 截图自查] -> 对比原图微调 -> 交付
```

### A1. 拿到设计图

如果用户还没有图，先明确来源，不要自己瞎猜。按"够用就停，别用重方案"的原则从上往下选：

**方式A：CC直接调API生图（最省心，适合频繁用）**
如果用户已经/愿意配置 OpenAI 兼容的 API Key（官方、OpenRouter、或302.AI等中转），可以直接在CC内一句话生图落盘，无需切出去网页端。
- 检查项目里是否已经装了 gpt-image-2 类的生图 skill（社区开源方案如 `garden-skills` 的 `gpt-image-2` skill、`VolientDuan/skills`）；没有的话可以引导用户去装，或者自己写一个最小化脚本（用 bash_tool 调 `curl` 到 `/v1/images/generations` 端点）。
- 这类 skill 通常三种模式自适应：有Key就全自动出图；没Key但有内置生图工具（如Codex环境）就走内置；都没有就退化成"帮你写一段结构化prompt，你自己去网页端粘贴"。
- 生图落盘后直接把路径交给下面 A2 处理，不需要用户手动下载。

**方式B：browser-act接管浏览器，操作网页版生图（省token，复用已登录账号）**
- 推荐开源项目 `browser-act/skills`（MIT协议），核心是 `browser-act` skill：复用本地Chrome登录态（Real Chrome Control模式），打开网页版ChatGPT，输入生图prompt，等待出图，截图/下载结果。
- 优势：不会把整页HTML塞给模型消耗context，浏览器层先做等待、截图、提取，只把干净结果交给CC。
- 如果这个动作以后要反复用，用配套的 `browser-act-skill-forge` 固化成专属skill。
- 只是偶尔看一眼网页/截个图，CC自带的 Chrome集成（Claude in Chrome扩展）也够用。

**方式C：用户自己去网页端生图，手动交图（最简单，偶尔用一次够了）**
- 生图prompt建议包含：页面类型（落地页/仪表盘/表单/App截图）、风格关键词、必须出现的具体元素、配色偏好。
- 提醒用户：GPT Image 2 对**文字渲染和UI截图级还原**特别强，适合直接生成"看起来像真实UI"的参考图。

**方式D：Figma设计稿**：优先建议用 Figma MCP（还原精度可达95%+，比读图片更准），跳过生图环节直接进代码还原。先检查是否已连接 Figma MCP。

**方式E：手机截图/竞品截图/手绘草图**：直接可用，跳到 A2。

> **选型建议**：偶尔用一次 → 方式C最简单；要反复批量生图/接入自动化流程 → 方式A（API）；不想开API账单、想用自己账号网页额度 → 方式B（browser-act）。三者不冲突，可按场景切换。

拿到图片路径后（Windows用户粘贴失败可提醒用 AiImgPaste，或让用户直接发文件路径），进入 A2。

### A2. 读图分析，先说人话再动手

看图后先用文字列出以下几点，跟用户确认没理解错再写代码（除非用户已明确要求直接开工）：
- 整体布局结构（几栏、导航位置、卡片/表单/列表等主结构）
- 配色（主色、辅助色、背景色，尽量取近似hex值）
- 字体层级（标题/正文/说明文字的大小关系）
- 关键交互元素（按钮、输入框、图标位置）
- 技术栈确认：React/Vue/纯HTML？是否用 Tailwind？（用户之前提过偏好就直接用，不用再确认）

### A3. 生成代码

- 优先语义化HTML结构，不要一股脑div套div。
- 颜色、间距量化成具体数值（参考设计图比例估算），而不是"大概"。
- 如果项目里已有设计系统/组件库（检查相关SKILL或已装的 `frontend-design` 插件），复用现有规范。
- 复杂页面分区块写：先搭骨架（布局网格），再填充各区块细节，方便中途review。
- 如果分支B已经把背景/卡片/图标拆成了独立透明PNG，优先直接引用这些素材文件（更精确），而不是重新用CSS画。

### A4. 自查与迭代（最容易被跳过，但最重要）

1. 有 Visualizer/浏览器截图能力时，把生成页面截图和原图并排对比。
2. 主动指出差异点："导航栏间距比原图紧""按钮颜色深了一点"，不要等用户来挑错。
3. 根据反馈针对性微调，而不是整体重写。

### A5. 边界与提醒

- 设计图里有明显品牌Logo、真实人物照片、受版权保护的图案，提醒用户注意授权问题，不代为下载或搬运受版权保护的原始素材。
- 图片信息不足以判断细节（hover态、响应式断点）时直接问用户，不要瞎编。
- 不承诺100%像素级还原——如实告知这是"视觉近似还原"，Figma MCP路线才是数据级还原。

## 分支B：素材拆分（多联图裁切 / 透明PNG / 分层PSD）

适用场景：GPT Image等模型一次性生成了"一张图里塞了好几个UI元素"（比如背景+头像+多张卡片+徽章），用户需要拆成独立素材以便复用或导入Photoshop/Figma继续编辑。

### B1. 选工具，规则裁切优先

按"够用就停"原则：
- 规则网格切图（元素排列整齐、间距固定）→ 在线 Ezgif Sprite Cutter 或直接用规则坐标裁切最快。
- 游戏开发工作流、需要打包管理 → TexturePacker 的 Split Sheet。
- 本地批处理、需要处理透明背景/连通区域识别 → 本地 Python 脚本（ImageMagick/OpenCV/PIL），这是本技能默认走的路线，见 `references/asset-splitting.md`。

### B2. 拆出独立透明PNG

用 `references/asset-splitting.md` 里的 `split_asset_sheet.py` 脚本 + 坐标配置文件，把原图按坐标裁切、去棋盘格背景、trim、留边距，输出一组独立命名的PNG。

**关键提醒**：AI生成的"透明背景"有时其实是**画出来的棋盘格图案**，不是真alpha透明，脚本需要显式处理（`--remove-checkerboard`）。更稳的根治方法是重新生图时在prompt里明确要求 `transparent background, no checkerboard pattern`。

### B3.（可选）合成分层PSD

如果用户需要在Photoshop里继续编辑，不要指望图像模型"直接生成PSD"——那样图层、位置、命名都不可控。正确做法分两步：
1. 先按 B2 拆出干净的透明PNG。
2. 再用脚本按坐标、层级把PNG写入PSD（`references/asset-splitting.md` 里的 `create_layered_psd.py` + `psd_manifest_example.json`）。

这样Photoshop打开后是真正独立、可移动/隐藏/缩放的像素图层，图层名、位置、顺序都可控。

**需要向用户说明的边界**：PSD里的元素是像素图层，不是Photoshop原生文字图层；卡片里的中文/图标/数字文字不能直接在PSD里改文字内容。如果用户要"文字也能直接改"，最稳的方法是卡片底板和图标做PNG图层，文字另外在Photoshop/Figma/App里用文本层重新排版。

### B4. 推荐固定流程（给用户的长期建议）

```
整图参考 → 拆成透明PNG多联图 → 自动裁切 → manifest定坐标 → 生成PSD → Photoshop/Figma微调
```

比单纯提示AI"帮我生成PSD"稳定得多。以后做同类UI素材（尤其是儿童App UI这类多元素合成图），可以直接复用这套流程。

详细脚本、命令、prompt模板见 `references/asset-splitting.md`。
