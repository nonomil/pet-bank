# 2026-07-05 静态站性能与部署优化建议

## 目标

基于当前线上三套可访问版本的实测结果，给出一份可直接执行的网页性能优化与部署建议，重点解决以下问题：

1. `summer-bank.nonom.top` 的 `cloud-config.local.js` 返回成 HTML。
2. GitHub Pages 版本的 `cloud-config.local.js` 返回 404。
3. 首页依赖资源过多，首屏加载链路偏重。
4. `data/pets.json` 过大，当前是全量首屏加载。
5. 明确后续应继续走静态托管/CDN，还是继续走 VPS + Tunnel。

## 结论摘要

### 1. 这个项目本质上是静态前端应用

当前站点以 `index.html + css + js + json + assets` 组成，页面主体由前端脚本驱动，不依赖服务器端实时渲染。它更准确的归类是：

- 静态站
- 单页前端应用
- 带可选云配置和云端同步能力的静态应用

这意味着它更适合部署到：

- 静态托管平台
- CDN 边缘缓存平台

而不是把静态站继续当成“源站穿透服务”来发布。

### 2. 当前性能瓶颈不主要在“源站国家”，而在“前端负担 + 发布链路”

本次从当前机器实测：

- `https://nonomil.github.io/pet-bank/`
  - 首页平均总耗时约 `1.124s`
- `https://summer-bank.nonom.top/`
  - 首页平均总耗时约 `1.912s`
- `https://pet-bank.nonom.top/`
  - 首页平均总耗时约 `2.351s`

而三者首页 HTML 内容完全一致：

- 标题一致
- HTML 大小一致
- 内容哈希一致

这说明当前“国内源站 + Cloudflare Tunnel”并没有带来你期待的用户侧速度收益，反而是更标准的静态托管链路更占优势。

### 3. 当前最值得优先改的不是“换个源站”，而是“减首屏负担 + 修配置 + 调缓存”

优先级建议：

1. 修配置文件加载和发布策略
2. 减少首页首屏必须加载的 CSS / JS / JSON
3. 调整静态资源缓存策略
4. 再迁移到更适合静态站的平台

## 直接证据

### 证据 A：首页头部直接依赖外部运行时和多份样式

见 [index.html](</G:/StudyCode/宠物积分系统/index.html:8>) 到 [index.html](</G:/StudyCode/宠物积分系统/index.html:21>)：

- 运行时加载 `cdn.tailwindcss.com`
- 运行时加载 `unpkg.com/lucide`
- 运行时加载 Google Fonts
- 同时串了 10 份 CSS

这会带来几个问题：

- 首屏请求数偏多
- 外部第三方域名阻塞更明显
- `cdn.tailwindcss.com` 官方本身就不建议生产环境这样使用

### 证据 B：首页一次性加载过多脚本

见 [index.html](</G:/StudyCode/宠物积分系统/index.html:990>) 到 [index.html](</G:/StudyCode/宠物积分系统/index.html:1023>)：

- 首页底部一次性加载大量业务脚本
- 包括探索、卡牌、汉字、游乐场、家庭社交、云同步、语音等多个子系统
- 首页打开时，即使用户根本没点对应 tab，也已经把这些脚本纳入首轮加载与执行

这会直接拉高：

- 首屏解析时间
- JS 执行时间
- `domcontentloaded` 完成时间

### 证据 C：全量宠物库在首页阶段加载

见 [js/pet.js](</G:/StudyCode/宠物积分系统/js/pet.js:530>) 到 [js/pet.js](</G:/StudyCode/宠物积分系统/js/pet.js:541>)，以及 [index.html](</G:/StudyCode/宠物积分系统/index.html:991>) 到 [index.html](</G:/StudyCode/宠物积分系统/index.html:995>)：

- `PetSystem.loadPetDB()` 在首页启动时直接执行
- 它会请求 `data/pets.json`
- 当前 [data/pets.json](</G:/StudyCode/宠物积分系统/data/pets.json>) 大小约 `447379 bytes`
- 数据总量 `238` 种宠物，`20` 个系列

这对首页来说偏重。首页其实通常只需要：

- 当前宠物
- 少量推荐宠物
- 图鉴入口摘要

没有必要在首屏就把完整宠物库全量灌进来。

### 证据 D：云配置加载策略需要重新收口

见 [js/cloud-config-loader.js](</G:/StudyCode/宠物积分系统/js/cloud-config-loader.js:4>) 到 [js/cloud-config-loader.js](</G:/StudyCode/宠物积分系统/js/cloud-config-loader.js:54>)：

- 设计上已经区分了：
  - `js/cloud-config.public.js`
  - `cloud-config.local.js`
- 且按逻辑，`cloud-config.local.js` 只应该在：
  - `file:`
  - `localhost`
  - `127.0.0.1`
  时尝试加载

但线上实测暴露出两类问题：

- `summer-bank` 上它返回成了 HTML
- GitHub Pages 上它是 404

这说明至少有一个问题成立：

1. 线上部署的代码版本和当前仓库代码并不完全一致。
2. 某些发布平台的回退规则把这个路径错误地回退到 `index.html`。
3. 生产环境仍然在请求本地开发配置文件。

## 修改建议

## P0：先修线上错误配置和发布规则

### 建议 1：生产环境不再依赖 `cloud-config.local.js`

保留 [cloud-config.local.js](</G:/StudyCode/宠物积分系统/cloud-config.local.js>) 只作为本地开发用途。

生产环境建议只走下面两种之一：

- 方案 A：使用 `js/cloud-config.public.js`
- 方案 B：在部署平台注入运行时配置，再写入 `window.__PETBANK_CLOUD_CONFIG__`

要求：

- 线上公共域名访问时，不应再请求根目录 `cloud-config.local.js`
- 如果平台必须做 SPA 回退，必须排除：
  - `*.js`
  - `*.css`
  - `*.json`
  - `*.webp`
  - `assets/*`
  - `data/*`

否则会再次出现“本该返回 JS，却回成 HTML”的问题。

### 建议 2：核对当前线上是否仍在跑旧 loader

当前仓库代码里，`cloud-config.local.js` 只会在本地环境尝试读取。既然线上仍然请求了它，建议先核对：

- 实际部署的 `js/cloud-config-loader.js` 是否与仓库一致
- 平台构建是否用了旧产物
- 是否有 Service Worker / 旧缓存 / 历史构建残留

## P1：减首页负担

### 建议 3：去掉生产环境运行时 Tailwind CDN

把 [index.html](</G:/StudyCode/宠物积分系统/index.html:8>) 的：

- `https://cdn.tailwindcss.com`

改成构建后本地 CSS。

原因：

- 运行时 Tailwind 会增加首屏网络请求和执行成本
- 官方不建议生产环境这样用
- 它对中国网络环境更容易成为波动点

建议做法：

- 用构建脚本把需要的 Tailwind 样式编译进本地 CSS
- 最终只保留本地样式文件

### 建议 4：拆分首页与非首页脚本

当前首页一次性加载了大量脚本。应改为：

- 首页必需：
  - `pet.js`
  - `home.js`
  - `app.js`
  - 必要的最小工具函数
- 按需加载：
  - `exploration.js`
  - `exploration-detail.js`
  - `card-arena.js`
  - `card-arena-ui.js`
  - `hanzi-game.js`
  - `learn-center.js`
  - `voice.js`
  - 云同步 / 家庭社交相关脚本

建议实现方式：

- 用户首次点击某个 tab 时再 `import()` 对应模块
- 或用显式 `loadExploreModules()`、`loadPetModules()` 之类的懒加载入口

目标：

- 首页不再默认执行所有子系统脚本
- 首屏只保留“看得到就会用到”的逻辑

### 建议 5：减少首页 CSS 数量

当前头部串了多份 CSS。建议拆成两层：

- `critical/home.css`
  - 首页和全局骨架必须样式
- feature css
  - 游乐场、汉字、卡牌、探索等模块在首次进入时再加载

如果暂时不做完整构建，也建议先把：

- 纯首页必要样式
- 非首页样式

分层处理，避免首屏吞下所有页面样式。

## P1：减数据负担

### 建议 6：把 `pets.json` 拆成摘要与详情

当前 `data/pets.json` 对首页来说太大。建议拆成：

- `data/pets-summary.json`
  - `id`
  - `name`
  - `emoji`
  - `rarity`
  - `series`
  - 默认缩略图
- `data/pets-detail/*.json`
  - 某个系列或某个来源的详情

首页只加载摘要。

真正进入：

- 宠物页
- 图鉴页
- 卡牌或战斗详情页

时，再去加载对应详情分片。

### 建议 7：故事数据和语音数据继续延后

从网络抓取结果看，GitHub Pages 版本在首页还会继续拉取多份：

- `data/stories/*.json`
- 语音相关资源

这些更适合在进入：

- 探索
- 故事
- 语音播放相关页面

时再拉。

首页不应把这些作为首轮必须数据。

## P2：调缓存与发布策略

### 建议 8：静态资源改成可缓存

当前某些线上版本对首页、JS、CSS、图片用了：

- `no-cache`
- `no-store`
- `must-revalidate`

这对静态站不友好。

建议：

- `index.html`
  - `max-age=60` 到 `600`
- 带版本号或文件指纹的 JS / CSS / 图片
  - `max-age=31536000, immutable`
- JSON 数据
  - 如果不是实时变化，至少 `max-age=600` 或更高

你现在 GitHub Pages 比 Tunnel 版本更快，一个重要原因就是它的静态缓存策略更合理。

### 建议 9：发布链路从“隧道”切到“静态托管”

对于这个项目，公开站点推荐优先级：

1. EdgeOne Pages / EdgeOne 静态托管
2. 帽子云静态托管
3. GitHub Pages
4. VPS + Tunnel

原因：

- 这是静态前端应用
- 静态托管平台更适合 CDN 分发
- Tunnel 更适合把内部服务穿出来，不适合当公开静态站最佳链路

## 部署建议

## 推荐部署方案

### 方案 A：公开演示版

适合：

- 官网
- 演示站
- 家长/孩子直接访问的页面

推荐平台：

- EdgeOne
- 帽子云
- GitHub Pages

要求：

- 不依赖 `cloud-config.local.js`
- 静态资源全量托管
- 用 `js/cloud-config.public.js` 或构建时注入配置

### 方案 B：私有/调试版

适合：

- 本地测试
- 内部联调
- 开发环境

可继续保留：

- `localhost`
- 本地静态服务器
- 本地 `cloud-config.local.js`

### 不推荐的公开发布方案

不建议继续把“上海腾讯云 + Cloudflare Tunnel”作为这个静态首页的主要公开链路。

原因：

- 不是静态站的最优发布模型
- 会引入额外转发链路
- 无法天然获得静态托管平台的边缘缓存收益

## 建议的实施顺序

1. 修 `cloud-config.local.js` 线上请求问题
2. 从首页移除运行时 Tailwind CDN
3. 把首页脚本拆成“必需 + 按需加载”
4. 把 `pets.json` 改成摘要 + 分片详情
5. 调整缓存策略
6. 迁移到 EdgeOne 或帽子云做公开静态托管

## 验收指标

调整后至少验证这些指标：

1. 首页请求数明显下降
2. 首页不再请求 `cloud-config.local.js`
3. 首页不再首轮加载全部功能模块脚本
4. 首页不再首轮加载完整 `pets.json`
5. 首屏总耗时比当前版本下降
6. `js/app.js` 和关键静态资源命中缓存
