# docs 资料库

`docs/` 保存产品方案、需求原文、调研、专题设计、阶段记录和验收证据。它是历史与决策资料库，不是当前代码契约。

## 先读哪里

1. [工程文档索引](../docs_project/README.md)：当前架构、模块、数据契约、运行和发布验证。
2. [当前实现快照](../docs_project/CURRENT-STATE.md)：当前代码能证明的能力边界。
3. [文档生命周期](./文档生命周期.md)：目录职责、状态标签和归档规则。
4. [产品路线](方案/项目路线/README.md)：仍在版本控制中的长期规划。

## 目录职责

| 目录 | 内容 | 阅读定位 |
| --- | --- | --- |
| `plans/` | 当前仍需执行或复核的设计/实施计划 | 活跃工作 |
| `归档/` | 已完成计划、旧进度、旧总览和交接包 | 历史追溯 |
| `需求/` | 原始需求和规格 | 要解决什么问题 |
| `参考/` | 参考资料、外部案例和截图 | 借鉴什么 |
| `方案/` | 产品决策、叙事、路线和玩法专题 | 为什么做、怎么设计 |
| `工程/` | 架构、工程路线、网页优化和自托管 | 怎么实现和交付 |
| `资源/` | 生图提示词和素材工作流 | 怎么生产资源 |
| `进度/`、`发布/`、`变更/` | 阶段变化、发布证据和变更记录 | 做过什么 |

## 维护边界

- `docs_project/` 的当前事实优先于 `docs/` 中的旧方案和旧行号。
- 活跃计划完成后，先把结论回填到工程文档，再移入 `docs/归档/`。
- 参考截图默认不删除；新测试截图优先写入 `tmp/test-artifacts/`。
- 本目录默认是本地工作材料，是否纳入版本控制由根目录 `.gitignore` 和发布边界决定。
