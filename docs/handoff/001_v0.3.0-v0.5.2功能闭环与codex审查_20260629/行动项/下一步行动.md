## codex 审查建议点（6 项，按优先级）

### P0 · 架构一致性
- **模块边界**：home.js（宠物小屋+家具+气泡+背景+进化进度）职责是否过重？需拆分？
- **localStorage 键约定**：petbank_* 业务键 + petbank_profiles_meta/active/profile_data_{id} 元数据/快照 + petbank_home_* 等。命名是否一致？有无冲突？
- **数据流**：积分（petbank_points）多处读写（app/exploration/shop/home），单一真相源？
- **模块加载顺序**：index.html script 引入（pet→...→profiles→app），依赖是否正确？

### P0 · 回归风险
- **home.js 多轮叠加**：P0（v0.3.7）+ 家具（v0.3.8）+ 气泡/背景 + 进化进度 多轮叠加，renderUI 是否臃肿/冲突？
- **exploration.js 战斗改动**：battleTurn skill/item/CD/defend 改动（v0.3.9）是否影响原有 attack/flee/掉落？
- **app.js 多处接入**：init ensureDefault/render + 动作映射 + 卡片 rarity + 战斗 UI，是否有未清理的死代码/旧分支？

### P1 · 多孩子 swap 边界
- **快照时序**：switchTo 先 snapshot(current) 再 restore(target)，若中途异常（reload 前）数据状态？
- **删除当前档**：remove(active) 用 _swapTo(target, skipSaveCurrent=true)，当前数据作废不存。审查：skipSaveCurrent 路径正确性？
- **老数据迁移幂等**：ensureDefault 首次建 p_default，业务键原地不动。审查：多次调用/active 失效/ meta 损坏的恢复？
- **profile_data 快照膨胀**：每次 switch 存全量业务键快照，多 profile 多快照，localStorage 容量（5-10MB）风险？

### P1 · 战斗技能时序
- **CD tick 的 cdStartedThisTurn**：用技能当回合跳过 tick。审查：连续技能/道具后 tick/防御态下 tick 的边界？
- **defending 一次性清除**：敌人反击 takeDamage(applyDefend) 减伤后清。审查：defend 后未受击（敌人 miss？）/连续 defend？
- **道具消耗回合**：useItemInBattle 触发 battleTurn({type:'item'}) 敌人反击。审查：道具致死（敌人反击 KO）/道具 CD tick？
- **flee_chance 0.3 对齐**：代码硬编码 0.3，combat.json 配置 0.3。审查：是否真读配置（非硬编码）？

### P2 · 奖励权重平衡
- **盲盒 EV（期望值）**：普通 20 分盲盒 EV = 0.5×(5-15 返利均10) + 0.5×道具价值。豪华 50 分 EV。审查：EV 是否低于/高于价格（玩家亏/赚）？
- **卡片 rarity 权重梯度**：common 50/legendary 1。审查：legendary 是否过罕（图鉴难集）/过易？
- **shop.js 硬编码奖池 vs items.json rarity**：4 层奖池硬编码，与 items.json rarity 字段可能漂移。审查：是否应动态读 items.json rarity？

### P2 · 文档与代码对齐
- **CHANGELOG v0.4.0 段 rebase 丢失**：v0.4.0 commit 的 CHANGELOG 段在 rebase 时丢失（已在 v0.5.0 commit 补回）。审查：CHANGELOG 完整性？
- **设计稿边界 vs 实现**：商店家具（v0.3.8 设计稿「不做 buff/拖拽/多房间」）+ 战斗（v0.3.9「不做 MP/技能树」）实现是否真守边界？
- **147 口径**：文档统一 147，代码 pet.js 注释 147，pets.json total 147。审查：全链路一致？
