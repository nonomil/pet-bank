# 商店家具联动设计稿（纯装饰版）

> **ID**：`DESIGN-2026-06-29-01`
>
> **状态**：边界已确认，待实施
>
> **目标**：补齐“积分消费 → 家具购买 → 小屋摆放 → 持久化展示”的消费-家园闭环，同时避免把家具系统做成过重的经营或数值系统。

---

## 1. 背景

当前代码已经具备两块基础能力：

- `js/home.js` 已有 `petbank_home_state`、`petbank_home_furniture`、`placeFurniture()`、`removeFurniture()`、`addFurniture()`。
- `js/shop.js` 已有积分扣减、购买历史、盲盒逻辑，但没有“家园装饰”分类，也没有和小屋 ownership 状态打通。

所以现在的真实缺口不是“没有家具系统”，而是：

1. 家具元数据分散在 `home.js` 内部，没有共享目录。
2. 商店买完东西后，无法把家具写进小屋的已拥有列表。
3. 小屋缺少“已拥有但未摆放”的家具栏展示。

---

## 2. 已确认的产品边界

本轮按“产品级、不过度设计、刚刚好满足需求”的原则，边界固定如下：

- 家具是**纯装饰版**，不提供属性加成，不改 HP / ATK / EXP / 探索收益。
- 家具采用**永久拥有模型**，不是消耗品，不进 `InventorySystem`，不做堆叠数量。
- 同一家具 **不可重复购买**；已拥有后商店按钮禁用并显示“已拥有”。
- 摆放采用**固定槽位**，不做拖拽，不做自由坐标，不做碰撞布局。
- 同一槽位替换家具时，旧家具回到“已拥有未摆放”状态，不会丢失。
- 不做多房间 / 多楼层 / 多主题装修。
- 不做家具 buff、不做租金/清洁费等经营数值。

这意味着本轮目标非常明确：

**把消费闭环打通，而不是把家园玩法做重。**

---

## 3. 备选方案对比

| 方案 | 做法 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| A. 直接在 `shop.js` 里硬编码家具列表 | 商店单独维护家具数组，购买时直接 `HomeSystem.addFurniture(id)` | 改动最少，上手最快 | 家具数据分散在 `shop.js` / `home.js` 两处，后续维护容易漂移 | 不推荐 |
| B. 新增 `data/furniture.json` 共享目录 | 商店和小屋都读取同一份家具数据，ownership 仍写现有 localStorage | 单一事实源、边界清晰、规模刚好 | 需要一次性补一个小型数据加载流程 | **推荐** |
| C. 把家具并入 `data/items.json` | 家具作为新 `type=furniture` 挂到 InventorySystem | 表面上文件更少 | 家具和消耗品/装备语义不同，容易把永久装饰误做成背包道具 | 不推荐 |

### 3.1 选型结论

采用 **方案 B：`data/furniture.json` 共享目录**。

原因：

- 对当前仓库来说，它足够专业，但不过度。
- 能避免 `home.js` 和 `shop.js` 各自维护一份家具表。
- 不会把家具错误塞进 `InventorySystem`，保持“消耗品 / 装备”和“永久装饰”的职责分离。

---

## 4. 用户闭环

### 4.1 商店侧

1. 用户进入商店，看到新分类“家园装饰”。
2. 每张家具卡显示：图标、名称、描述、价格、适用槽位。
3. 若未拥有，点击“购买”：
   - 扣除积分
   - 写入 `petbank_home_furniture`
   - 写购买历史
   - 按钮变成“已拥有”
4. 若已拥有，则按钮禁用，不允许重复购买。

### 4.2 小屋侧

1. 用户切到小屋，看到“已拥有家具栏”。
2. 家具栏显示“已拥有但未摆放”的家具。
3. 先点选家具，再点可兼容槽位完成摆放。
4. 若槽位已有家具，则新家具顶替进去，旧家具自动回到未摆放栏。
5. 刷新页面后，ownership 与摆放结果都保留。

---

## 5. 家具目录与数据模型

### 5.1 共享目录

新增：`data/furniture.json`

推荐结构：

```json
{
  "version": "1.0",
  "defaults": ["food_bowl", "bath_tub"],
  "furniture": [
    {
      "id": "food_bowl",
      "name": "食盆",
      "icon": "🥣",
      "price": 0,
      "slotType": "floor",
      "defaultOwned": true,
      "description": "默认家具，作为喂食区域视觉锚点"
    }
  ]
}
```

### 5.2 首批家具清单

| 家具 id | 名称 | 获取方式 | 价格 | 槽位类型 | 备注 |
|---|---|---|---|---|---|
| `food_bowl` | 食盆 | 默认拥有 | 0 | `floor` | 保留现有默认家具 |
| `bath_tub` | 浴缸 | 默认拥有 | 0 | `corner` | 保留现有默认家具 |
| `cozy_rug` | 温暖地毯 | 商店购买 | 35 | `floor` | 地面装饰 |
| `soft_cushion` | 软垫 | 商店购买 | 45 | `floor` | 地面装饰 |
| `toy_box` | 玩具箱 | 商店购买 | 55 | `floor` | 地面装饰 |
| `night_lamp` | 夜灯 | 商店购买 | 60 | `corner` | 角落装饰 |
| `wall_frame` | 墙饰相框 | 商店购买 | 40 | `backdrop` | 背景装饰 |
| `star_mobile` | 星星挂饰 | 商店购买 | 70 | `backdrop` | 背景装饰 |

### 5.3 槽位模型

当前小屋保留 5 个固定位置：

| 槽位 id | 槽位类型 | 说明 |
|---|---|---|
| `center_left` | `floor` | 中左地面 |
| `center_right` | `floor` | 中右地面 |
| `corner_left` | `corner` | 左角落 |
| `corner_right` | `corner` | 右角落 |
| `back` | `backdrop` | 背景层 |

> 词汇上保留 `surface / corner / backdrop / floor` 四种家具类型，但本期只启用 `floor / corner / backdrop`。`surface` 作为后续扩展预留，不提前实现。

### 5.4 持久化约定

沿用现有 localStorage 键，不新增额外命名空间：

- `petbank_home_furniture`
  语义：已拥有家具 id 数组
- `petbank_home_state`
  语义：小屋摆放状态

示例：

```json
{
  "slots": {
    "center_left": "cozy_rug",
    "center_right": null,
    "corner_left": "bath_tub",
    "back": "wall_frame",
    "corner_right": null
  },
  "theme": "cozy_night"
}
```

### 5.5 ownership 规则

- ownership 是**去重集合**，不是数量库存。
- “已拥有未摆放”是**派生状态**，不单独存表：
  `ownedIds - placedIds = unplacedIds`
- 默认家具必须始终存在于 ownership 集合中。
- 删除摆放只会清空 `slots[slotId]`，不会删 ownership。

## 6. 界面范围

### 6.1 商店

新增一个“家园装饰”板块，放在盲盒和奖励兑换之后或之间均可，但要和现有购买区分清楚。

每张卡至少包含：

- 图标
- 名称
- 价格
- 描述
- 槽位类型标签
- 按钮状态：`购买` / `已拥有`

### 6.2 小屋

当前小屋底部已有槽位展示，本轮增加：

- 已拥有未摆放家具栏
- 当前选中家具的高亮状态
- 槽位兼容提示
- “移除”或“收回”操作

交互方式固定为：

**点击家具 -> 点击槽位**

不做：

- 拖拽
- 长按编辑
- 网格坐标系统

---

## 7. 关键规则

### 7.1 购买规则

- 购买前先检查积分是否足够。
- 购买前先检查是否已拥有。
- 购买成功后只调用 `HomeSystem.addFurniture(id)`，不直接写 `petbank_home_*`。
- 商店只负责“卖”和“扣分”，小屋系统负责“拥有态与摆放态”。

### 7.2 摆放规则

- 只能摆到兼容槽位类型。
- 不兼容槽位不响应或显示禁用态。
- 同槽位替换时，新家具进入槽位，旧家具回到未摆放栏。
- 不允许把未拥有家具摆进小屋。

### 7.3 失败与降级

- `furniture.json` 加载失败时：
  - 商店隐藏“家园装饰”板块或显示加载失败提示
  - 小屋保留默认两件家具，避免白屏
- localStorage 数据异常时：
  - 用默认 ownership + 默认 slots 重建
  - 自动补回 `food_bowl` / `bath_tub`

## 9. 验收标准

1. 商店出现“家园装饰”分类，至少展示 6 个可购买家具。
2. 购买家具会正确扣分，并写入 `petbank_home_furniture`。
3. 已拥有家具不可重复购买。
4. 小屋可看到已拥有未摆放家具，并能放入兼容槽位。
5. 槽位替换后旧家具不会丢失，而是回到未摆放栏。
6. 刷新页面后，ownership 与摆放状态都能恢复。
7. 全流程不引入 buff、不经过背包、不新增后端依赖。

---

## 10. 明确不做

- 家具属性加成 / 探索 buff
- 家具放入背包、可出售、可分解
- 家具拖拽摆放
- 多房间 / 多主题 / 多楼层
- 家具成就、装修评分、房间清洁经营

本轮设计的价值不在“玩法扩张”，而在于：

**把已经存在的积分、小屋、商店三块内容真正闭成一个最小但完整的产品回路。**
