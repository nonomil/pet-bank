# 家庭账号与家长端边界设计

> 状态：已确认设计，等待实施计划执行
>
> 参考：`docs/家庭账号社交体系/`、`docs/家长区与管理接口/`、`docs/参考/banchong账号机制分析与多孩子方案.md`

## 1. 目标

解决两个直接问题：

1. 家长端目前只是孩子端 SPA 的一个换壳页面，孩子端导航、孩子切换器和家长入口仍然容易同时暴露。
2. 家长新增孩子需要进入设置、使用浏览器 `prompt`，再手动决定是否切换和同步，认知负担过高。

本次改造建立清晰的产品边界：

```text
/app/*
  孩子主应用：今日、学习、宠物、探索、游乐场

/parent/*
  家长管理域：登录、家庭、孩子档案、成长记录、规则、工具、设置
```

## 2. 参考结论

班宠乐园 1 和班宠乐园 2 的共同经验不是照搬课堂模型，而是信息架构分层：

- 高频主流程保持短而清楚。
- 家长管理使用独立入口和管理壳。
- 账号、规则、危险操作继续向二级页面收纳。
- 工具页只放低频轻工具，批量管理和覆盖操作不与孩子玩法并列。

本项目仍然采用“家长账号 -> 家庭 -> 孩子档案”的家庭模型，不引入教师、班级、学生三层模型。

## 3. 产品架构

### 3.1 孩子端

规范路径为 `/app/*`。孩子端只展示：

- 孩子自己的成长首页、积分、学习、宠物、探索和游乐场。
- 当前孩子的简短状态信息。
- 不展示家长账号、家庭邀请码、Supabase 配置、诊断、数据导入导出和规则编辑。

孩子端保留本地 profile 机制。孩子不是独立登录账号，而是家庭下的成长档案；这是降低账号数量和登录复杂度的关键。

### 3.2 家长端

规范路径为 `/parent/*`。家长端使用独立管理壳，首屏以“家庭概览”为主，提供三个高频动作：

- 选择孩子并查看今日成长。
- 管理孩子档案。
- 管理家庭账号和协作关系。

账号、家庭同步、好友边界、规则模板、工具和高级操作进入对应二级页面。家长端不复用孩子端顶部主导航，只复用业务模块和数据服务。

### 3.3 未登录守卫

- 访问 `/parent/*` 时先检查当前 Supabase session。
- 已登录：进入请求的家长页面。
- 未登录：进入 `/parent/login`，登录成功后回到原始目标路径。
- 访问 `/app/*` 不要求家长登录，继续支持本地离线模式。
- 本地模式下家长域可以保留本机管理入口，但要明确这是本机入口隔离，不等价于服务端安全；跨设备家庭数据必须经过 Supabase session 和 RLS。

本次不把前端隐藏当成安全权限。真实数据权限继续由 Supabase RLS、Edge Functions 和家庭成员关系决定。

## 4. 账号模型

### 4.1 登录方式

家长端以手机号 + 密码作为主登录方式，以匹配参考项目和家长使用习惯；现有邮箱账号保留兼容：

- 包含 `@` 的输入按邮箱登录。
- 纯手机号按手机号登录。
- 注册界面默认展示手机号字段，保留邮箱兼容入口或兼容旧账号，不要求用户理解内部 auth provider。
- 注册邀请码、家庭邀请码的职责不变：前者允许创建家长账号，后者把已注册家长加入家庭。

真实手机号、密码和 Supabase 密钥不能进入仓库、测试夹具或文档示例。

### 4.2 家长与孩子的关系

```text
Supabase auth.users
  -> public.accounts
  -> public.households
  -> public.household_members
  -> public.child_profiles
```

本地 `ProfileManager` 继续负责孩子端档案切换；登录家长账号后，云端恢复和同步负责把 `local_profile_id` 映射到 `child_profiles.id`。

## 5. “加孩子”流程

家长端“孩子档案”页使用单一创建入口，不让用户直接接触底层同步概念。

### 5.1 创建向导

1. 点击“添加孩子”。
2. 在对话框中填写孩子昵称。
3. 选择一个默认头像。
4. 确认创建。
5. 自动生成本地 profile 并立即选中。
6. 已登录且家庭已准备好时自动同步到云端；失败时保留本地档案并给出可理解的重试提示。

创建成功后不再弹出第二个“是否立即切换”确认。添加孩子的默认结果就是进入新孩子档案。

### 5.2 档案卡片

每个孩子卡片只显示：头像、昵称、当前孩子状态、最近同步状态和进入按钮。改名、删除、云端覆盖等低频/高风险动作收进卡片的操作菜单，并保留二次确认。

### 5.3 兼容现有本地数据

- 继续使用 `js/profiles.js` 的现有快照机制。
- 不改变既有 `petbank_*` 业务键和 profile 元数据合同。
- 首次进入新家长端时，旧的 `p_default` 档案继续保留。
- 云端同步失败不能回滚本地创建，也不能清空现有孩子数据。

## 6. 页面与路由合同

### 6.1 家长路径

```text
/parent                 家庭概览
/parent/login           家长登录 / 注册
/parent/children        孩子档案
/parent/family          家庭成员、邀请码、好友边界
/parent/records         成长记录 / 家庭复盘
/parent/tools           轻工具
/parent/settings        家长设置首页
/parent/settings/account   家长账号与恢复方式
/parent/settings/rules      积分规则模板
/parent/settings/advanced  备份、诊断、覆盖类操作
```

现有 `/settings/*` 和旧 `/parent/settings/*` 路径保留兼容跳转，避免旧书签失效；新入口和内部链接统一使用 `/parent/*`。

### 6.2 孩子路径

继续以 `/app/*` 为规范路径，保留 `/today`、`/learn`、`/pet` 等旧路径兼容映射。孩子端页面不再通过常驻按钮进入家长域；家长入口可由家长专用入口或登录页提供，不放进孩子主导航。

## 7. 组件边界

本次优先复用现有模块，不重写业务玩法：

| 责任 | 现有模块 | 调整方向 |
| --- | --- | --- |
| 路由与壳 | `js/app.js`, `index.html`, `css/style.css` | 输出 app/parent shell，增加 parent auth guard |
| 家长认证 | `js/auth.js` | phone/email 双模式，独立登录页挂载，保留注册邀请码 |
| 本地孩子档案 | `js/profiles.js` | 提供向导式 create API 和卡片状态，不再依赖 prompt |
| 家庭同步 | `js/household.js`, `js/cloud-restore.js`, `js/profile-sync.js` | 创建后自动尝试同步，保留显式恢复/覆盖操作 |
| 家长页面 | `index.html` 中现有 parent/settings 页面 | 重新分配到 parent 路由和专属管理壳 |
| 业务玩法 | `js/app.js` 及各玩法模块 | 只消费当前 profile，不参与家长壳判断 |

## 8. 错误处理与权限边界

- 没有 Supabase 配置：孩子端照常运行；家长端显示“本机管理模式”并解释能力边界。
- session 过期：家长页回到登录页，带回原始目标路径。
- 登录失败：显示可理解的错误，不暴露 Supabase 原始堆栈或 token。
- 新建孩子云端同步失败：本地创建成功，显示“已保存在本机，稍后重试同步”。
- 删除或云端覆盖：二次确认，并明确影响范围；覆盖操作继续收纳到高级区。
- 前端 route guard 只负责产品入口隔离，不能替代数据库 RLS。

## 9. 验证标准

### 静态合同

- `/app/*` 不包含家长账号、家庭云端、诊断和数据管理入口。
- `/parent/*` 使用家长管理壳，不输出孩子端主导航。
- 未登录访问 `/parent/children` 会被导向 `/parent/login`，登录后回原路径。
- 旧 `/settings/*` 路径仍能进入兼容页面。
- 添加孩子流程不再调用浏览器 `prompt`。

### 浏览器流程

1. 本地模式打开 `/app`，孩子主流程不被账号配置打断。
2. 未登录打开 `/parent`，看到家长登录页而非孩子页面。
3. 登录后打开 `/parent/children`，添加孩子一次完成创建和切换。
4. 云端可用时，新孩子自动同步；云端不可用时，本地档案仍保留。
5. 退出家长账号后，家长页回登录态，孩子端本地数据仍可用。
6. 旧账号和旧孩子档案不因登录方式或壳切换丢失。

## 10. 非目标

- 本次不拆分全部物理 HTML 文件，不引入前端框架。
- 本次不重写 Supabase 家庭/孩子表模型，不新增教师/班级模型。
- 本次不把孩子改成独立手机号账号。
- 本次不做短信验证码、找回密码和完整账号安全中心；先保持 Supabase 现有能力。
- 本次不开放广场式社交、聊天或实时在线。
