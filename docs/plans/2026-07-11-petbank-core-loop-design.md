# 宠物积分核心循环设计

## 目标

把现有的积分、奖励、宠物经验、宠物小屋互动串成一条孩子能理解的日常闭环：

`完成行动 -> 获得奖励 -> 宠物成长/状态变化 -> 小屋给出下一步照料 -> 继续行动`

本阶段不重做现有数值，不改词库和游戏规则，不引入每日积分封顶；重点是统一奖励语义、补齐成长反馈和建立每日照料目标。

## 当前问题

- `addGrowthPoints`、`PetSystem.addExp`、商店购买、宝箱奖励和游戏奖励分别维护，来源难以追踪。
- 同样是“奖励”，成长积分、宠物经验、亲密度、道具和家具没有统一展示口径。
- 任务/游戏完成后通常只看到积分变化，宠物的经验、等级、进化和亲密反馈不稳定。
- 小屋已经有喂食、玩耍、洗澡、休息，但没有“今天照料了什么”和“下一步做什么”的持续目标。
- 游戏已有 `GameRewardReceipts` 去重，但不能覆盖任务、宝箱和小屋奖励事件。

## 设计选择

采用“轻量奖励事件层 + 现有模块适配”的渐进方案。

### 奖励事件契约

每次可产生奖励的行为都可转换为以下事件：

```js
{
  eventId: "task:2026-07-11:task-id",
  profileId: "local",
  source: "task|game|chest|home|shop|system",
  sourceId: "具体任务/游戏/动作 id",
  rewards: [
    { type: "growth_points", amount: 10 },
    { type: "pet_exp", amount: 5 },
    { type: "intimacy", amount: 2 },
    { type: "item", itemId: "...", amount: 1 }
  ],
  occurredAt: "ISO-8601",
  policyVersion: "core-loop-v1"
}
```

事件层负责校验、去重、落盘和返回结果，不负责决定所有业务数值。第一阶段适配时保留现有积分数值，避免历史存档迁移风险。

### 奖励分层

- `growth_points`：账户成长分，可用于家庭奖励、商店、家具和背景。
- `pet_exp`：宠物等级和进化进度。
- `intimacy`：持续照料和互动关系。
- `item` / `furniture` / `theme`：收集和装扮，不与积分混淆。

### 小屋每日照料

每天生成 3 个固定照料目标，默认覆盖喂食、玩耍、清洁/休息中的三个动作。目标状态按本地日期保存，重复进入页面不会重复计数。完成目标后显示“今日照料完成”，并给出回到学习或游戏的下一步入口。

目标只记录行为完成，不额外发放新的积分，避免第一阶段引入经济膨胀；后续可在数据验证后增加小额非货币反馈。

## 数据流

1. 任务、游戏、宝箱或小屋动作调用奖励事件服务。
2. 服务先按 `eventId + profileId` 查重；已领取则返回 `duplicate`，不重复发放。
3. 新事件写入 `localStorage`，并调用现有积分/经验/亲密度 API。
4. 返回标准化结果：已发放奖励、宠物变化、升级/进化标记、重复标记。
5. 页面使用统一奖励卡片显示“获得了什么”和“下一步建议”。

## 错误和兼容策略

- `localStorage` 不可用时，保留现有直接发放逻辑，不阻断学习流程。
- 老存档没有新字段时按默认值补齐。
- 事件写入失败不得导致积分重复扣除；先完成去重检查，再执行一次性发放并记录结果。
- 不修改已有 `GameRewardReceipts` 的键格式，通过适配层复用其语义。

## 验收标准

- 同一 `eventId` 重复提交只能产生一次奖励。
- 任务、游戏、宝箱至少各有一个适配入口，且保留原有积分值。
- 发放积分或宠物经验后，结果能明确包含宠物经验/等级变化。
- 小屋显示当天照料进度、未完成目标和推荐动作。
- 刷新页面、重复进入小屋、旧存档升级都不会丢失或重复发放。
- 现有静态门禁和网页专项测试继续通过。

## 分阶段范围

- CP1：奖励事件契约、去重和任务/游戏/宝箱适配。
- CP2：统一宠物成长反馈，补齐等级/进化结果和奖励卡片数据。
- CP3：小屋每日照料状态、推荐动作和动作完成反馈。
- CP4：整合页面文案与全量回归，评估后续经济数值调整。
