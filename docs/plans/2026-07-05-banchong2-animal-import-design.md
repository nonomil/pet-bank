# 班宠乐园2 动物导入设计稿

## 1. 背景

当前项目的宠物体系已经按“分馆 -> 主题册 -> 宠物详情”组织完成，主要入口包括：

- `阳光花园馆`
- `奇趣冒险馆`
- `创想课堂馆`
- `方块生态馆`

这次新增内容来自 `班宠乐园2` 的家长端宠物接口。已确认该来源包含 `ANIMAL / PLANT / PERSON` 三类，而本轮只导入 `ANIMAL`。用户已经明确两点：

1. 不新开顶层分馆，继续沿用现有馆系结构。
2. 新的动物子册名称锁定为 `萌爪伙伴册`。

另外，用户已经指出当前网页里仍有部分 `.bmp / .jpg` 一类的默认图在慢速加载，尤其是图鉴场景，因此这轮导入必须把“素材本地化 + 统一转 `.webp`”当成硬约束，而不是顺手优化。

## 2. 本轮目标

把 `班宠乐园2` 的动物宠物导入到当前项目，并让它们在以下链路里完整可用：

- 认养/宠物列表可见
- `奇趣冒险馆` 中可归档浏览
- `萌爪伙伴册` 中可集中查看
- 宠物详情页有成长图、故事、档案字段和技能文案

本轮以“能稳定上线一批完整动物宠物”为目标，不追求同时重做全部旧卡面或重写所有旧馆逻辑。

## 3. 方案比较

### 方案 A：新开一个顶层分馆承接班宠乐园2

优点：

- 来源边界最清楚
- 后续继续扩植物/人物时更自然

问题：

- 会打破当前已经成型的四馆结构
- 用户当前并不想再增加一个大馆
- 这轮只做动物，先开新馆成本偏高

### 方案 B：并入 `奇趣冒险馆`，新增一个主题册承接动物

优点：

- 和当前产品结构最一致
- 对用户来说改动最小
- 后续仍能继续扩同来源的其他子册

问题：

- 需要在内部保留来源区分，不能直接混成旧 `banchong`

### 方案 C：直接并入现有 `萌宠搭档册`

优点：

- 改动最少

问题：

- 会把老来源和新来源混在一起
- 用户已经明确希望这批内容可区分、可独立命名
- 后续补故事、补图时不利于维护

## 4. 最终决定

采用 **方案 B**。

### 顶层结构

- 顶层分馆继续使用 `奇趣冒险馆`
- 新增主题册：`萌爪伙伴册`

### 来源标识

- 新来源使用独立 source id：`banchong2`
- 不与旧 `banchong` 共用 source，避免筛选、图鉴标签和后续维护混淆

### 范围控制

- 只导入 `ANIMAL`
- `PLANT` 和 `PERSON` 先不进入本轮
- 本轮优先做“动物导入 + 图鉴故事 + 本地 webp 素材闭环”

## 5. 信息架构

用户可见结构锁定为：

- `奇趣冒险馆`
- `萌爪伙伴册`

内部数据结构保持“册内可继续细分系列”的能力，但不把系列命名作为本轮必须锁死的产品决策。原因是：

- 源站动物数量较多，最终会按实际导入名单分桶
- 系列名属于内部整理口径，可在实现时微调
- 用户当前只要求先把馆/册结构和名称锁死

因此，本轮设计锁定：

1. 馆级：`奇趣冒险馆`
2. 册级：`萌爪伙伴册`
3. 系列级：保留为实现阶段的内部整理项

## 6. 数据设计

### 6.1 运行时数据

主要更新以下数据层：

- `data/pets.json`
- `data/pokedex-lore-draft.json`

`data/pets.json` 中新增的宠物条目需要满足现有运行时习惯：

- `source: "banchong2"`
- 使用本地素材路径
- 保持 `stages` 数组结构，兼容当前图鉴详情与成长阶段展示

### 6.2 图鉴文案

`data/pokedex-lore-draft.json` 继续沿用现有字段体系，每只新宠至少补齐：

- `codexTitle`
- `subtitle`
- `intro`
- `origin`
- `childhood`
- `school`
- `work`
- `hobby`
- `specialty`
- `ability`
- `story`
- `traits`
- `skills`

文案目标不是“先占位”，而是做到可以直接在图鉴详情页里读通。允许使用模板化起稿，但入库前必须人工收口，避免明显机器味或空话。

## 7. 图片与资产策略

### 7.1 远端资源特点

已确认源站宠物接口会返回：

- `name`
- `category`
- `baseAssetUrl`
- `maxLevel`

动物宠物当前使用 10 级图片资源，路径形态类似：

- `/pet_assets/动物/阿拉斯加/1.png`
- `/pet_assets/动物/阿拉斯加/2.png`
- ...
- `/pet_assets/动物/阿拉斯加/10.png`

### 7.2 本地资产原则

所有导入后的运行时图片都使用本地资源，不直接热链远端地址。

统一要求：

- 最终入库格式为 `.webp`
- 不把 `.bmp / .jpg / .jpeg / .png` 直接暴露到图鉴主展示链路
- 目录独立存放到 `assets/banchong2/...`

推荐目录形态：

- `assets/banchong2/<系列>/<宠物名>-0.webp`
- `assets/banchong2/<系列>/<宠物名>-1.webp`
- `assets/banchong2/<系列>/<宠物名>-2.webp`
- `assets/banchong2/<系列>/<宠物名>-3.webp`
- `assets/banchong2/<系列>/<宠物名>-4.webp`
- `assets/banchong2/<系列>/<宠物名>-5.webp`

### 7.3 图鉴首页卡面

当前图鉴首页优先尝试加载 `assets/cards/composed-v2/<petId>.webp`，但已有 fallback 机制。

因此本轮不强制为每一只新宠额外生产合成卡面；未生成时允许沿用现有 fallback：

- 首页回退到成长图
- 详情页展示 6 阶成长图和故事档案

这样可以先保证内容和性能闭环，再决定是否批量补做精修卡面。

## 8. 10 级源图到 6 阶项目图的兼容方案

这是本轮的关键兼容点。

现有项目里，`banchong` 型宠物普遍按 6 阶展示：

- `0`
- `1`
- `2`
- `3`
- `4`
- `5`

而 `班宠乐园2` 动物素材是 10 级。为了不改动当前整套成长 UI，本轮采用“10 -> 6 的代表级映射”：

- 项目阶段 `0` <- 源等级 `1`
- 项目阶段 `1` <- 源等级 `2`
- 项目阶段 `2` <- 源等级 `4`
- 项目阶段 `3` <- 源等级 `6`
- 项目阶段 `4` <- 源等级 `8`
- 项目阶段 `5` <- 源等级 `10`

这样做的原因：

- 不需要重写现有 6 阶成长体验
- 能保留“初见 -> 成长 -> 满级”的视觉跨度
- 取偶数代表级，规则简单，导入脚本可稳定复用

本轮不做 10 级完整前端展示，也不在详情页额外加 10 级模式切换。

## 9. 运行时接线

需要更新的前端接线主要有：

- `js/card-collection.js`
- `js/app.js`
- `js/pet.js`

具体口径：

### `js/card-collection.js`

- 新增 `banchong2` 的来源标签
- 把 `banchong2` 纳入 `奇趣冒险馆`
- 在 adventure 主题册中新增 `萌爪伙伴册`

### `js/app.js`

- 宠物来源筛选增加 `banchong2`
- 认养/列表页来源显示文案同步更新

### `js/pet.js`

- 现有 `banchong` 媒体继承逻辑要扩成支持 `banchong2`，或重构为“多阶段宠物通用逻辑”
- 确保新来源的 `stages / imageStages / imageStyle` 能被正常识别

## 10. 性能要求

本轮要顺手解决用户已指出的慢图问题，因此性能要求单独列为验收项：

1. 新导入的 `banchong2` 图鉴图片全部为本地 `.webp`
2. 图鉴主链路不再新增 `.bmp / .jpg / .jpeg` 资源引用
3. 宠物详情和图鉴首页继续使用懒加载
4. 若发现现有旧图鉴仍有非 webp 资源，可在实现阶段补一轮格式审计脚本

## 11. 验收标准

1. `奇趣冒险馆` 内出现 `萌爪伙伴册`
2. `班宠乐园2` 动物以独立来源 `banchong2` 可筛选、可展示
3. 每只导入宠物都有 6 阶本地 `.webp` 成长图
4. 每只导入宠物在图鉴详情页都有可读故事和档案字段
5. 图鉴主展示链路不引入新的 `.bmp / .jpg / .jpeg` 资源
6. 未生成合成卡面的宠物也能通过 fallback 正常显示

## 12. 本轮不做什么

本轮不做：

- 新开第五个顶层分馆
- 导入 `PLANT` 和 `PERSON`
- 把源站 10 级成长完整原样搬进前端
- 强制为每只新宠生产 `assets/cards/composed-v2` 卡面
- 一次性重构全部旧来源的图片格式
