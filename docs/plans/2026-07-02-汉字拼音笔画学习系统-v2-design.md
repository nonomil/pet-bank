# 2026-07-02 汉字拼音笔画学习系统 V2 设计稿

> 范围：在现有“汉字答题 + 本地排行榜 + 轻量记忆状态机”基础上，升级成更像课程化启蒙产品的学习系统。
> 目标：提升学习性，而不是继续堆答题题库。

---

## 1. 先说结论

当前汉字游乐场已经能玩，但更像“题卡测验”，还不像“识字课程”。

最值得做的不是继续加更多题，而是把它升级成 4 条主线：

1. `新字课`：认识字、懂字义、听发音、看例句。
2. `今日复习`：按到期时间复习，而不是只做加权随机。
3. `写字台`：看笔顺、临摹、纠错、默写。
4. `拼音站 + 阅读小岛`：把拼音和小阅读接起来，形成“认 -> 拼 -> 写 -> 读”的闭环。

如果要参考“洪恩识字/洪恩拼音”这类产品，我们应该学的是它们的课程结构和记忆闭环，不是照搬 UI。

## 2. 当前版本的真实状态

### 2.1 已经有的能力

- 汉字大厅 / 启蒙关 / HSK1 入口已接通。
- 本地排行榜已按孩子隔离。
- `js/hanzi-progress.js` 已有轻量记忆状态机：`new -> learning -> mastered`。
- `HanziProgress.pickNext()` 已支持“错题优先 + 局内去重”。
- 空等级开始已被守卫，避免直接写入假 `0` 分战绩。

### 2.2 仍然影响学习体验的问题

1. 现在主玩法仍以“选择题/填空题”为主，识字和写字没有拆轨。
2. 当前记忆系统仍是“权重抽题”，不是“到期复习”。
3. 拼音只作为题面信息存在，没有形成独立训练链路。
4. 没有笔顺/书写环节，容易出现“会认不会写”。
5. 阅读没有和已学字绑定，学完后缺少“能读出来”的出口。
6. HSK 题库仍有少量坏例句，需要数据清洗。

### 2.3 本地题库体检

基于 `data/hanzi-hsk.json` 的本地抽查：

- 总题数：`594`
- 空例句：`0`
- 占位/坏例句：`5`
- 空拼音：`0`
- 缺失 modes：`0`

已定位到的典型坏数据：

- `data/hanzi-hsk.json:698` `第第？`
- `data/hanzi-hsk.json:5880` `...其中**假**...`
- `data/hanzi-hsk.json:6312` `太阳从**南**边升起来。`
- `data/hanzi-hsk.json:6344` `...带有**友**的例句...`
- `data/hanzi-hsk.json:7544` `...**息**...`

## 3. 这轮设计参考了什么

### 3.1 可借鉴的产品结构

洪恩官网对“洪恩识字”的描述，核心不是题库量，而是：

- `玩、认、练、写、说` 五步教学法。
- AI 笔画纠错。
- 子集绘本和识字进度联动。
- 复习内容按记忆曲线组织。

洪恩官网对“洪恩拼音”的描述，则强调：

- 完整拼音体系。
- `玩、调、读、说、拼` 的多环节闭环。
- 拼音阅读和应用，而不是只背声母韵母。

这些都说明：好的启蒙产品不是“多做题”，而是把输入、操练、输出和复习编成课程。

### 3.2 适合网页端落地的开源能力

1. `Hanzi Writer`
   - 可直接用 `<script>` 引入。
   - 支持笔顺动画。
   - 支持 `quiz()` 手写练习。
   - 支持 `onMistake / onCorrectStroke / onComplete` 回调，方便接入积分与记忆系统。

2. `Make Me a Hanzi`
   - 提供 9000+ 常用汉字的字典和笔画图形数据。
   - `dictionary.txt` 可取拼音、部件、字义、构形信息。
   - `graphics.txt` 可取每一笔的 SVG path，用来做本地化笔顺资源。

3. `pinyin-pro`
   - 浏览器可直接脚本引入。
   - 支持带音调拼音、数字声调、多音字自动识别。
   - 支持把汉字转成带 `<ruby>` 的拼音 HTML。

4. `ts-fsrs`
   - 可直接用于到期复习调度。
   - 支持 `request_retention`、`next()`、`repeat()` 等标准间隔重复能力。

### 3.3 一个重要判断

网页端“写字”是方便落地的，但要区分两种目标：

1. `给定汉字的笔顺临摹/纠错`
   - 低风险，适合现在做。
   - `Hanzi Writer + Make Me a Hanzi` 足够。

2. `开放式手写识别：孩子随便写一个字，系统猜是什么`
   - 明显更难。
   - 需要额外的手写识别模型或服务。

所以 V2 推荐先做前者，不建议一上来承诺“全自由写字识别”。

## 4. V2 的产品结构

### 4.1 学习主循环

建议把汉字游乐场改造成下面这个闭环：

`新字课 -> 当日练习 -> 写字台 -> 今日复习 -> 阅读小岛`

其中：

- `新字课` 负责引入。
- `当日练习` 负责第一次检索。
- `写字台` 负责把“认”变成“写”。
- `今日复习` 负责长期记忆。
- `阅读小岛` 负责迁移到真实语言场景。

### 4.2 首页入口建议

汉字专区不再只有“开始挑战”，而改成 5 个入口：

1. `今日任务`
2. `新字课`
3. `今日复习`
4. `写字台`
5. `拼音站`

下面再补一个轻量模块：

- `阅读小岛`

这样家长和孩子一眼能知道“今天先学什么”，而不是直接掉进 10 题挑战。

## 5. 各模块怎么设计

### 5.1 新字课

每次只教 `1-3` 个新字，不要一口气塞 10 个。

单字课建议 6 步：

1. 看图识义：图 + 字 + 简短语义。
2. 听发音：拼音 + 语音按钮。
3. 看构形：象形/会意/形声，能讲就讲，讲不明白就不硬讲。
4. 看组词：2-3 个高频词。
5. 看例句：只给低负担短句。
6. 马上小测：认字 1 题、拼音 1 题、选词 1 题。

### 5.2 今日复习

复习不应该再是“随机抽 10 道”，而是：

- 先抽今天到期的卡。
- 再补少量昨天错题。
- 最后才补新字巩固。

建议题型占比：

- 60% 到期复习
- 25% 错题回炉
- 15% 新字轻练

### 5.3 写字台

写字台建议分 4 档难度：

1. `看动画`
2. `描红`
3. `按笔顺写`
4. `隐藏提示默写`

每写完一个字，系统记录：

- 总笔画数
- 错误笔画次数
- 是否需要提示
- 完成时长

这些数据不要和“认字掌握度”混为一谈，而是单独记“书写掌握度”。

### 5.4 拼音站

拼音不要只给在题卡上方显示，而要拆成独立训练：

1. `认读`
   - 声母 / 韵母 / 整体认读
2. `声调`
   - 一二三四声辨认
   - 同音不同调对比
3. `拼读`
   - 声母 + 韵母 + 声调合成音节
4. `汉字映射`
   - 拼音找字
   - 多音字辨音
5. `拼音阅读`
   - 带注音短句
   - 点击字可听读

### 5.5 阅读小岛

推荐做“子集阅读”，不是直接丢整段材料。

每篇短文控制为：

- `80%` 熟字
- `20%` 新字/拓展字

孩子点击任意字时，可展开：

- 拼音
- 词义
- 已学/未学状态
- 是否加入复习

## 6. 记忆系统要怎么升级

### 6.1 为什么当前版本还不够

现在的 `HanziProgress` 更像“学习中的加权抽样器”：

- 它能照顾错题。
- 它能避免短时间重复。
- 它能展示 learned/mastered 数量。

但它还没有回答最关键的问题：

`这个字下次应该什么时候复习？`

### 6.2 建议的数据结构

建议把一条学习记录从“一个字一条状态”升级成“一个内容轨道一条状态”。

示例：

```json
{
  "itemId": "hsk1:char:山",
  "track": "writing",
  "state": "review",
  "dueAt": 1762348800000,
  "lastReviewAt": 1762262400000,
  "stability": 3.4,
  "difficulty": 5.6,
  "reps": 4,
  "lapses": 1,
  "lastRating": "good"
}
```

推荐至少拆 4 条轨道：

1. `recognition` 认字
2. `writing` 写字
3. `pinyin` 拼音
4. `reading` 阅读迁移

同一个“山”可以出现 4 条记忆状态，互不污染。

### 6.3 调度方案

优先级建议：

1. 首选 `ts-fsrs`
   - 标准化程度高。
   - 后续容易扩成更完整的复习系统。

2. 如果当前仓库想保持极轻量
   - 先做 `FSRS-lite`
   - 只保留 `dueAt / reps / lapses / difficulty / stability`

不建议继续长期停留在纯权重随机。

### 6.4 评分映射

可以把孩子的表现映射成 4 档：

- `Again`：答错 / 写错很多 / 完全不会
- `Hard`：答对但明显吃力
- `Good`：正常答对
- `Easy`：快速稳定答对

这比“对/错二值”更适合记忆调度。

## 7. 具体到网页端，怎么接笔画

### 7.1 MVP 做法

第一版不需要自己画笔顺引擎，直接接现成能力：

1. `index.html` 引入 `hanzi-writer.min.js`
2. 本地准备字形数据
   - 先用 CDN 跑通
   - 再把常用字包本地化
3. 新建 `js/hanzi-writing.js`
4. 新增页面 `page-hanzi-writing`
5. 接入排行榜和记忆系统

### 7.2 页面结构建议

写字页分成左右两区：

- 左侧：目标字、拼音、词义、动画按钮
- 右侧：书写板

底部保留 4 个操作：

- `看笔顺`
- `开始书写`
- `重写`
- `下一个字`

### 7.3 评价逻辑

书写模式不要只看“是否完成”，还要记录：

- `mistakesOnStroke`
- `totalMistakes`
- `strokesRemaining`
- `usedHint`

其中 `usedHint` 和 `totalMistakes` 可以直接决定本次记忆评分。

## 8. 和现有仓库最贴合的实施顺序

### P0 当前版本补稳

1. 清洗 `data/hanzi-hsk.json` 中已知 5 条坏例句。
2. 补汉字学习回归测试。
3. 文档同步 `HanziProgress` 现状与局限。

### P1 复习系统升级

1. `js/hanzi-progress.js` 升级为 due-based scheduler。
2. 大厅新增 `今日复习` 卡。
3. 会话组卷改成“到期优先”。

### P2 写字台

1. 接 `Hanzi Writer`
2. 新增 `page-hanzi-writing`
3. 书写成绩写入 `writing` 轨道

### P3 拼音站

1. 接 `pinyin-pro`
2. 做声调 / 拼读 / 多音字专项
3. 生成带注音的 `ruby` 阅读材料

### P4 阅读小岛

1. 做熟字率过滤
2. 生成短句/短文
3. 将阅读点击行为回流到复习系统

## 9. 这轮建议的产品边界

建议明确以下边界：

- 不做“另一个大富翁”式第二循环。
- 不做联网社交排行榜。
- 不做开放式全字手写识别首版。
- 不做过重的 AI 实时批改链路首版。

这块最优先的是：

`课程化 + 记忆化 + 可书写 + 可拼读`

而不是继续堆更多玩法壳子。

---

## 10. 我对下一步的建议

如果继续做实现，最合适的顺序是：

1. 先做 `P0` 收口：数据清洗 + 回归测试。
2. 紧接着做 `P1`：把“权重抽题”升级成“到期复习”。
3. 然后做 `P2`：接笔顺与网页写字。

原因很简单：

- 没有稳定题库，后面都在沙地上搭楼。
- 没有到期复习，学习系统就还只是小游戏。
- 没有写字台，“会认不会写”这个核心缺口补不上。
