# PK Brawl Battle Motion Design

> 日期：2026-07-08
> 目标：把数学 PK、宠物卡牌对战、探索遭遇战统一升级成“前冲贴近 -> 命中 -> 受击 -> 回位”的战斗演出，同时保留三套玩法各自的视觉个性。

## 1. 背景

当前三套战斗已经具备：

- 基础命中提示
- 一部分光效层
- 一部分伤害数字和音效
- HP / 回合 / 出招文案

但整体手感仍偏“原地触发”：

- 攻击者大多没有真正离开原位
- 命中更像远程触发，而不是靠近后打中
- 受击方缺少足够明确的“被打到”反应
- 数学 PK、卡牌对战、探索遭遇战之间还没有形成一条统一的动作语言

参考 `docs/数学PK游戏/宝可梦数学大乱斗.md` 中的图，孩子最容易感知到的不是复杂规则，而是：

1. 谁冲过去了
2. 谁真的打中了
3. 被打的一方有没有明显中招
4. 这一回合结束后双方有没有回到台上继续准备下一次交手

## 2. 目标

本轮动效升级的直接目标：

- 三套玩法都必须出现明确的“靠近后出招”
- 命中光效必须作用在受击方，而不是只在攻击方身边出现
- 受击方必须有后仰 / 抖动 / 缩放等受击反应
- 动作结束后双方回位，舞台结构不被打乱

更高层的体验目标：

- 数学 PK 更像“宠物冲过去打中了对方”
- 卡牌对战更像“技能真的撞上了”
- 探索遭遇战更像“故事里的碰撞发生了”，而不是一段抽象日志

## 3. 核心决策

### 3.1 统一动作骨架，允许玩法差异

本轮不为三套玩法各写一套完全独立的引擎，而是统一成一条动作骨架：

1. `起势`
2. `前冲`
3. `接触`
4. `命中`
5. `受击`
6. `回位`

三套玩法共用这条时序，但各自配置：

- 前冲距离
- 缩放幅度
- 旋转 / 弹跳强度
- 命中光颜色
- 命中爆点层数
- 受击抖动强度
- 总时长

### 3.2 不新起一套大引擎

本轮优先扩展现有体系：

- `js/math-pk.js` 里的正式对战动画
- `js/card-arena-ui.js` 里的回合结果演出
- `js/battle-fx.js` + `js/app.js` 的探索遭遇战战斗特效

避免把当前已经存在的 BattleFx、音效事件、契约测试全部推翻重来。

### 3.3 教学可读性优先

尤其是数学 PK 和探索遭遇战，动效升级不能遮挡题目、答案区、血条或下一步按钮。

也就是说：

- 命中闪光应该集中在角色附近
- 中心区域不允许被厚重提示框长时间挡住
- 动效总时长要短，避免拖慢做题节奏

## 4. 统一动作语言

### 4.1 起势

攻击方在动作开始前先有一个短促蓄力：

- 轻微前倾
- 轻微放大
- 身后出现淡淡的蓄势光

目的：

- 让孩子在视觉上知道“谁要出手了”

### 4.2 前冲

攻击方明显离开原位，朝对手方向冲刺：

- 数学 PK：中等距离，偏直线
- 卡牌对战：更远，可带扑击/旋身/弧线
- 探索遭遇战：更短，强调快速接触

前冲时可以附带：

- 冲刺残影
- 短促轨迹线
- 轻微地面投光 / 尘点

### 4.3 接触

攻击动作必须发生在靠近后：

- 不是原位先放技能再瞬间判定
- 而是角色移动到接近中线或对手前侧，再触发攻击姿态

### 4.4 命中

命中必须作用在受击方身上，至少包含：

- 一团中心爆点
- 一层受击闪光
- 一圈短促扩散或碎闪

这层是参考图中“火花/爆亮”的核心体验。

### 4.5 受击

受击方需要至少两种反应叠加：

- 轻微后仰
- 短抖动
- 瞬时缩放
- 血条同步震动或脉冲

### 4.6 回位

攻击方退回原位，受击方稳定回正，准备进入下一轮。

这样每回合都有“冲出去再回来”的完整节奏。

## 5. 三套玩法的差异化

### 5.1 数学 PK

定位：最接近参考图。

重点：

- 明显前冲
- 命中黄白光球 / 光斑
- 对方吃到短促闪亮和受击抖动

动作风格：

- `前冲距离`：中
- `接触停顿`：短
- `命中亮度`：高
- `整体节奏`：利落，约 600-850ms

不做：

- 复杂全屏爆炸
- 长时间遮挡题面

### 5.2 宠物卡牌对战

定位：三套里最花、最夸张。

重点：

- 普攻：短前冲 + 直接撞击
- 强力技：更长前冲 + 斩光 / 旋击
- 必杀：更明显的轨迹、爆点和回弹

动作风格：

- `前冲距离`：中到大
- `接触停顿`：略长
- `命中层数`：最多
- `整体节奏`：700-1100ms

### 5.3 探索遭遇战

定位：故事战斗，轻量但不能虚。

重点：

- 有明显贴近
- 有命中和受击
- 不打断叙事太久

动作风格：

- `前冲距离`：短到中
- `命中亮度`：中
- `整体节奏`：500-800ms

## 6. 技术设计

### 6.1 统一事件结构

建议在三套玩法内部统一抽象成如下信息：

```js
{
  attacker: 'human' | 'robot' | 'pet' | 'enemy' | 'ally',
  defender: 'robot' | 'human' | 'enemy' | 'pet' | 'opponent',
  motionStyle: 'dash' | 'hop' | 'spin' | 'pounce' | 'claw',
  impactStyle: 'math-burst' | 'card-slash' | 'story-hit',
  strength: 'light' | 'medium' | 'heavy',
  result: 'hit' | 'crit' | 'shield' | 'heal' | 'faint'
}
```

不要求三套玩法立刻完全共享同一个 JS 文件接口，但语义层尽量对齐。

### 6.2 数学 PK

在 `js/math-pk.js` 中继续沿用现有 `_playAttackFx` 方向，但升级为：

- 角色容器显式前冲 / 回位
- 新增命中爆点 overlay
- 受击角色附加短暂受击 class

### 6.3 卡牌对战

在 `js/card-arena-ui.js` 中补一层回合动作演出：

- 根据行动类型给出不同前冲动作
- 命中时把 impact 落在被打卡位上
- 强化当前行动者和受击者的差异

### 6.4 探索遭遇战

扩展 `js/battle-fx.js` 与 `js/app.js`：

- BattleFx 不再只是浮在一层做“符号式特效”
- 同时驱动 pet/monster 区域的位移与受击 class
- 让 `battle-animate` 事件既能触发 overlay，也能触发角色 motion

## 7. 动效约束

### 7.1 低龄友好

- 不做尖锐、惊吓型闪白
- 不做强烈镜头抖动
- 不做长时间屏幕遮挡

### 7.2 可降级

保留 `prefers-reduced-motion: reduce`：

- 缩短或关闭前冲位移
- 关闭旋转与复杂残影
- 保留最小命中闪光和受击反馈

### 7.3 性能

优先使用：

- `transform`
- `opacity`
- 轻量伪元素 / 小 overlay DOM

避免每回合创建过多复杂节点。

## 8. 验收标准

### 8.1 数学 PK

- 正式对战中，答对后宠物必须明显前冲
- 命中光效必须出现在机器人侧
- 机器人抢答成功时，机器人也必须前冲并打到宠物
- 双方动作后回位

### 8.2 卡牌对战

- 普攻与强技至少有两种不同前冲动作
- 命中光效必须作用在受击方卡位
- 受击方血条有同步受击反馈

### 8.3 探索遭遇战

- 宠物攻击和怪物攻击都要有贴近动作
- 受击方必须有清晰受击反应
- 结束后仍不影响故事推进按钮和结算入口

## 9. 本轮不做

- 3D 镜头
- 全屏大爆炸技能演出
- 每只宠物完全独立动作库
- 逐帧骨骼动画系统
- 对战录像回放

## 10. 下一步

按这个设计进入实施时，优先顺序是：

1. 先补契约测试，锁住三套玩法都必须“贴近 + 命中 + 受击”
2. 先改探索 BattleFx 骨架，因为它最接近共用层
3. 再改数学 PK
4. 最后改卡牌对战，把动作花样拉满
