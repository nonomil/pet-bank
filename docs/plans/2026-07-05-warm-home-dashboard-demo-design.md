# 温暖陪伴型首页仪表盘示例工程设计

日期：2026-07-05

## 目标

在 `prj` 下新建一个可直接打开的测试工程，演示“给 Codex 做 UI 前，如何先准备 spec、关键视觉方向、设计规范 md，再落成首页仪表盘”的完整最小闭环。

这个示例工程不追求接入真实后端，而是专注验证一件事：

`温暖陪伴型宠物积分首页` 如何在没有成熟设计系统的前提下，仍然通过结构化输入稳定做出高质量第一屏。

## 设计结论

### 1. 工程形态采用静态前端演示工程

- 放在 `prj/home-dashboard-warm-demo`
- 采用 `index.html + styles.css + app.js + data.js` 的轻量结构
- 不引入额外框架，保证可以直接双击或本地静态预览

这样做的原因是：

- 这次重点是演示 UI 输入包和视觉落地方法，不是框架工程化
- 仓库里已有多处静态前端示例，适合放一个低门槛 demo
- 后续如果要把这套输入包迁移到 React/Vite，再单独升级也不迟

### 2. 页面定位为“家庭陪伴首页”，不是“运营后台”

首页仪表盘的第一感觉应该是：

- 温暖
- 被鼓励
- 有宠物陪伴反馈
- 有今日目标，但不过度 KPI 化

因此信息排序不采用“看板式数据优先”，而采用：

1. 宠物与问候 hero
2. 今日积分与小目标
3. 今日任务卡片
4. 最近奖励与成长反馈
5. 家庭协作入口

### 3. 视觉语言采用“柔和卡片 + 晨光背景 + 圆润层级”

视觉上要刻意避开：

- 生硬后台风
- 纯游戏数值面板风
- 过度玻璃拟态和厚重阴影

拟采用的设计语言：

- 奶油白、蜂蜜金、薄荷绿、湖水蓝作为主辅色
- 大圆角卡片
- 柔和投影和轻渐变背景
- 宠物 hero 卡作为首页情绪锚点
- 所有文案尽量像“陪伴式提醒”，而不是“系统指令”

### 4. 首页信息结构固定为 6 个模块

#### A. 顶部问候栏

- 当前时间问候
- 家庭昵称
- 一句轻提醒

#### B. 宠物 hero 卡

- 宠物名称与状态
- 今日心情
- 连续签到 / 陪伴天数
- 主 CTA

#### C. 今日积分进展

- 今日已得积分
- 距离小目标差多少
- 进度条
- 奖励诱因文案

#### D. 今日任务区

- `3` 张任务卡
- 每张卡有任务标题、说明、积分值、完成状态
- 支持一个“优先任务”高亮

#### E. 最近奖励 / 成长反馈

- 最近解锁的奖励
- 宠物成长变化
- 轻时间线或双栏摘要

#### F. 家庭协作入口

- 家长寄语
- 家庭榜样行为
- 查看全部任务/奖励的次级入口

### 5. 数据采用本地 mock，但结构要接近真实产品

虽然是 demo，也不能只放占位字。

需要准备：

- 家庭用户信息
- 宠物状态信息
- 今日积分摘要
- 任务列表
- 奖励列表
- 成长记录

这些数据将存放在 `data.js`，让示例工程既能展示 UI，也能顺手示范“给 Codex 的真实数据上下文应该长什么样”。

### 6. 示例工程要同时体现“输入包”概念

这个 demo 不是只有页面本身，还应该把输入物料一起放进去：

- `spec.md`
- `design-guidelines.md`
- `codex-prompt.md`

这样用户以后看这个工程时，不只是看到结果，还能看到：

- 这页为什么这样设计
- 如果让 Codex 接手，输入应该怎么写

## 验收标准

- 打开 `index.html` 后能看到完整的温暖陪伴型首页仪表盘
- 页面不是后台风，也不是数值堆叠型游戏面板
- 宠物 hero、今日积分、任务卡、奖励反馈、家庭协作入口层级清楚
- 文案是“陪伴式”的，不是硬命令口吻
- 工程目录里同时包含 `spec.md`、`design-guidelines.md`、`codex-prompt.md`
- 至少有一个基础 smoke test 能校验页面关键结构存在
