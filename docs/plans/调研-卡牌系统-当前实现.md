## 1. 卡片图鉴实现

### 1.1 两级图鉴渲染流程

入口 `CardCollection`（`js/card-collection.js:5` IIFE）。状态：

- `_cards`（已收集 petId 数组，localStorage `petbank_cards`，`:6-7`）
- `_view`（`'category'` 大类卡片 | `'grid'` 瀑布流，`:13`）
- `_selectedSource`（当前选中大类，`:14`）
- `_allSpecies = PetSystem.getAllSpecies()`（`:34`，全部 147 宠物模板）

渲染层级（`renderUI` 调度，`:115`）：

1. **大类卡片** `_renderCategory()`（`:119-152`）：
   - 顶部统计：总收集 `_cards.length / _allSpecies.length`，按稀有度计数（`:122-133`）
   - 4 个大类卡片 `original / banchong / classpet / minecraft`（SOURCE_NAMES/SOURCE_EMOJI，`:16-17`），每个显示进度条 `collected/pets.length`（`:140-148`）
   - 点击 → `setView('grid', src)`（`:143`）

2. **瀑布流** `_renderGrid()`（`:155-172`）：
   - 按 `_selectedSource` 过滤 `_allSpecies`（`:156`）
   - 每个宠物渲染 `.card-item`，class 含 `collected|uncollected` + `card-rarity-{rarity}`（`:164`）
   - 未收集显示 `❓` + `???`，已收集显示 `emoji` + `name`（`:166-167`）
   - 点击 → `showDetail(id)`

3. **详情模态** `showDetail(petId)`（`:178`）：弹 `#cardDetailModal`。

`setView(view, source)`（`:19-23`）切换视图并 re-render。

### 1.2 「未收集卡片半透明看不见」根因【关键问题】

`css/card-collection.css:111-114`：

```css
.card-item.uncollected {
    opacity: 0.4;
    filter: grayscale(1);
}
```

- `opacity: 0.4` → 整体透明度 40%
- `filter: grayscale(1)` → 灰度 100%
- 且未收集时 emoji 统一渲染为 `❓`（`js/card-collection.js:166`），名称为 `???`（`:167`）—— 用户根本看不到宠物本体图/名

对照已收集（`:116-118`）：仅 `background: rgba(255,255,255,0.1)`，无透明/灰度。

**结论**：未收集卡片不是「看不见」（仍占位可点），而是「弱化呈现」（半透明 + 灰 + 占位符❓）。这是设计意图（防剧透），不是 bug。若要改为「看得见但弱化」，调高 opacity（如 0.7）或保留灰度但去 opacity 即可。

> 历史背景：git log 显示 `1c100b4 style(explore): 探索场景卡片不半透明(opacity→grayscale)` —— 探索场景已从 opacity 改为纯 grayscale，但图鉴 `.card-item.uncollected` 仍同时保留 opacity+grayscale，与探索场景策略不一致。这是潜在改进点。

## 2. 战斗数值体系

### 2.1 宠物数值字段（data/pets.json flat[]，147 只）

实际字段清单（13 个）：`id, name, desc, emoji, source, series, rarity, base_hp, base_atk, stages, imageStages, imageStyle, imageUrl`

**已有**：`base_hp`、`base_atk`、`rarity`、`series`、`source`、`stages`、`imageStages`

**缺失**（常见 RPG 字段，当前全无）：`def`(防御)、`spd`(速度)、`skill`(自带技能)、`crit`(暴击)、`mp`(蓝量)。

> 经 grep 全量校验：`def/defense/spd/speed/skill/skills/crit/mp/mana/mdef` 在 147 只宠物中出现次数全为 0。当前所有宠物共用同一套技能（data/skills.json 全局表），靠装备和等级差异化。

### 2.2 等级 → atk/hp 推导（PetSystem state）

`base_atk/base_hp` 是「基础值」。PetSystem 在选宠/升级时把基础值赋给 `state.atk / state.max_hp / state.hp`（升级可能加成，未在本次调研深挖升级公式）。战斗取值：

- `PetSystem.getTotalAtk()`（`js/pet.js:313-318`）：`state.atk + state.weapon.effect.atk`（武器加成）
- `PetSystem.getTotalMaxHp()`（`js/pet.js:392-397`）：`state.max_hp + state.armor.effect.max_hp`（护甲加成）
- `PetSystem.takeDamage(amount, {applyDefend})`（`js/pet.js:188-197`）：若 `defending` 则伤害×0.5 一次性清除
- `PetSystem.heal(amount)`（`:200-203`）

装备槽位（`js/pet.js:66-67,401-420`）：`weapon`(加 atk)、`armor`(加 max_hp)。仅 2 槽，无饰品/鞋子等。

### 2.3 战斗伤害公式（js/exploration.js battleTurn, `:323-444`）

**玩家伤害**：
- 普攻（`:398`）：`playerDmg = max(1, petAtk + rand(0..2) - 1)`，petAtk = `getTotalAtk()`
- 攻击型技能（`:384`）：`playerDmg = max(1, floor(petAtk × skill.multiplier) + rand(0..2))`
- 防御型技能（`:376-381`）：本回合不造伤，设 `defending=true`，下回合受击×0.5

**敌人伤害**（`:348, :437`）：
- 反击（`:437`）：`enemyAtk = max(1, monster.atk + rand(0..1) - 1)` → 经 takeDamage 的 applyDefend 判定
- 逃跑失败惩罚（`:348`）：`damage = max(1, monster.atk - floor(pet.level/2))`（唯一用到 pet.level 减伤处）

**特点**：
- 玩家伤害只看 atk，无 def 减免、无暴击、无闪避
- 敌人伤害也只看 monster.atk（场景配置，来自 data/scenes），玩家 def 不参与（无 def 字段）
- 唯一减伤机制：`defending`（防御技能，×0.5）+ 逃跑失败时的 `pet.level/2`
- 技能 multiplier：强力击 1.8、必杀 3、防御 0（data/skills.json）

### 2.4 怪物数值来源

怪物来自场景配置（`exploration.js:287` `scene.monsters`，字段 `name/hp/atk/exp`），数据在 data/scenes 相关文件（本次未深挖，scene.danger_level 决定等级与 EXP，`:291,316`）。

### 3.3 宠物图资源结构

两套不统一的图片字段（按 source 不同）：

- **original/minecraft**：用 `imageStages`（dict，key 0~4 = egg/seed/idle/mature/ultimate 5 阶段），路径 `assets/pets/poses/{id}_{stage}.webp`；带 `imageStyle`（pvz/minecraft）
- **banchong**：用 `stages`（array，0~5 共 6 阶段），路径 `assets/banchong/{series}/{name}-{n}.webp`；无 imageStages/imageStyle
- **classpet**：stages/imageStages/imageStyle 均为 None（疑似缺图或用 imageUrl）

资源数量：
- `assets/pets/poses/*.webp`：96 张（original+minecraft 透明立绘）
- `assets/banchong/**/*.webp`：492 张（banchong 91 只 × 约 5.4 阶段）

> 字段不统一是图鉴/战斗渲染的潜在复杂度来源（前端需按 source 分支取图）。
