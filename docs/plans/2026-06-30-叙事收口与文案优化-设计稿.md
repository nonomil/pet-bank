# 叙事收口与文案优化设计稿

> **ID**：`DESIGN-2026-06-30-07`
> **状态**：待执行
> **对应需求**：[2026-06-30-叙事收口与文案优化-需求](./2026-06-30-叙事收口与文案优化-需求.md)
> **目标**：在不新增大系统的前提下，把 12 个探索场景收口成“可维护、可扩写、可验收”的轻叙事内容层。

---

## 1. 设计结论

下一阶段不再继续扩新模块，而是集中完成 3 件事：

1. 把探索故事收口成**单一正式来源**。
2. 把 12 场景的数学事件改成**真正场景化**。
3. 把文档、章节目标、运行时实现统一成**同一口径**。

这不是一次“功能扩张”，而是一次“内容治理 + 产品收口”。

---

## 2. 设计边界

### 2.1 保留

- 保留 12 个场景，不扩新场景。
- 保留 5 章路线图和当前地图顺序。
- 保留现有事件结构：`narrate / discover / math / choice / encounter`。
- 保留现有战斗系统、怪物立绘、语音自动播报链路。
- 保留现有数学题型入口：`arithmetic / word / logic`。

### 2.2 不做

- 不引入完整分支剧情引擎。
- 不引入全学科课程体系。
- 不重写数学出题核心算法。
- 不重做探索框架或战斗框架。
- 不新增第 13 个及以后场景。

---

## 3. 推荐方案

## 3.1 故事文件夹成为唯一正式来源

推荐方案：

**以 `data/stories/*.json` 为唯一正式故事源，移除运行时“静默回退到旧文案”的做法。**

具体落法：

- `data/stories/*.json` 存放场景全部正式文本。
- `js/exploration-detail.js` 不再维护完整硬编码 `sceneEvents`。
- 如果故事文件加载失败，探索页直接给出可理解的错误提示，例如：
  `探索故事加载失败，请使用本地 HTTP server 或 GitHub Pages 打开。`

这样做的原因：

- 改哪一份文本会生效，变得明确。
- 不会出现“本地改了 JSON，运行时却被兜底旧文案覆盖”的假象。
- GitHub Pages 与本地 HTTP server 都是正常部署路径，失败时显式报错比静默回退更专业。

## 3.2 结束文案也并入场景 JSON

当前代码里 `SCENE_ENDING` 仍在 `js/exploration-detail.js` 中硬编码。  
这会让故事文本虽然文件化了，但结束感文本仍是第二来源。

推荐把结束文案也并入每个场景 JSON，字段为：

- `ending_text`

这样单个场景故事真正做到“一改即全改”，也方便未来配音和审稿。

## 3.3 每章建立教育目标，而不是扩课程系统

推荐只做**轻量章节教育目标**，不并入整个“幼小衔接”课程平台。

5 章目标如下：

| 章 | 章名 | 教育目标 | 主题型 |
|---|---|---|---|
| 1 | 起点花园 | 数感 / 数量关系 | `arithmetic`（10-20 以内加减） |
| 2 | 森林边界 | 加减应用 | `word`（低龄应用题） |
| 3 | 海边集市 | 生活情境应用题 | `word`（买卖 / 分配） |
| 4 | 高地洞窟 | 乘除与机关题 | `arithmetic`（乘除） |
| 5 | 星空终点 | 规律 / 逻辑 | `logic`（规律 / 等式） |

每个场景 JSON 增加：

- `chapter_skill`

这个字段是元数据，不是新系统。它的价值是让文案、审查和后续补图都能围绕“本章练什么”展开。

## 3.4 数学题必须成为场景的一部分

不是把“应用题”套在场景皮上，而是让题目本身来自场景对象：

- 森林：蘑菇、树叶、岔路
- 海滩：贝壳、螃蟹、珍珠
- 瀑布：露珠、荷叶、青蛙
- 糖果：糖串、蛋糕、饼干
- 雪山：冰晶、冰柱、雪狼脚印
- 星空：星座、星光灯笼、月亮树

推荐写法：

- 数学题的题面先描述场景，再提出数量关系。
- 奖励文案要体现“世界被你推进了”。
- 失败提示也要保留场景语气，不要变成系统口吻。

## 3.5 文案优化只做“去模板化”，不做重剧情

本轮文案优化的重点不是扩剧情，而是减少机械重复：

- `encounter` 不再重复“黑影 / 身影 / 脚步声”。
- `narrate` 不再大量重复“你发现了 / 前方有 / 你决定”。
- `choice` 选后不只给奖励，也给即时环境反馈。

目标是：

**让 12 个场景像 12 个不同地方，而不是同一套模板换皮。**

## 4. 数据结构目标

推荐的场景文件结构：

```json
{
  "id": "forest",
  "name": "神秘森林",
  "emoji": "🌲",
  "chapter": 1,
  "chapter_name": "起点花园",
  "chapter_skill": "数感 / 数量关系",
  "min_level": 1,
  "danger_level": 1,
  "ending_text": "森林里的捣蛋鬼被你吓跑啦！你采到了发光蘑菇，满载而归。",
  "events": [
    { "id": "forest.narrate0", "type": "narrate", "text": "..." },
    { "id": "forest.discover1", "type": "discover", "text": "..." },
    { "id": "forest.math2", "type": "math", "mathType": "arithmetic", "difficulty": "easy", "text": "..." },
    { "id": "forest.choice3", "type": "choice", "text": "...", "options": [] },
    { "id": "forest.encounter4", "type": "encounter", "text": "..." }
  ]
}
```

说明：

- `events` 仍是运行时核心结构。
- `chapter_skill` 和 `ending_text` 是本轮新增元数据。
- 不引入更复杂的状态字段、分支条件、剧情变量。

- 给 12 场景补齐 `chapter_skill`

这是最能降低维护成本的一批。

### 第三批：R4 / R5 文案去模板化 + 死代码清理

- encounter / narrate / choice 去模板化

这是成品感打磨批次。

## 6. 验收标准

1. `data/stories/*.json` 成为唯一正式故事源。
2. 12 个场景均具备 `chapter_skill` 和 `ending_text`。
3. 12 个 `math` 事件都能看出明确场景对象。
4. 章节目标、运行时实现、文档描述三者一致。
5. 不回归路线地图、战斗、怪物立绘、语音自动播报。

如果直接交给 CC，推荐按下面顺序执行：

1. 先读需求文档。
