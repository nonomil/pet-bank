## 1. 背景与目标

### 1.1 参考案例提取
「汉字游乐场」是一个独立的汉语启蒙答题游戏，其博文披露的游戏设计要点：

- **卡片引擎**：每张卡片 = `汉字 + 拼音 + 图片 + 例句` 四要素，按学习进度动态生成，翻牌交互。
- **交互闭环**：答对/错即时动效 + 计分，题目间淡入淡出过渡。
- **本地排行榜**：存最高分 + 最近 10 次战绩，「与自己赛跑」（OPC 精神）。
- **UI 调性**：网格布局、圆角阴影「呼吸感」、弹性适配多屏。

### 1.2 本项目现状（差距）
| 能力 | 现状 | 差距 |
|---|---|---|
| 答题对战引擎 | `math-pk.js` 完整（5 轮竞速/计分/写积分） | 仅数学题库，无汉字 |
| 成绩记录 | 仅 `petbank_math_high_score` 存单个数字 | **无排行榜 UI、无成绩列表、无 profile 隔离** |
| 汉字/语文玩法 | 无 | 完全空白 |
| 多孩子隔离 | `ProfileManager` 已支持 | 排行榜需复用 |

### 1.3 本次目标 / 非目标
**目标**
- G1 补一个**通用本地排行榜**（跨玩法、按孩子隔离），先接入 math-pk，未来汉字等玩法直接复用。
- G2 落地**汉字答题玩法**：四要素卡片、多出题模式、复用 math-pk 结算骨架、接入排行榜。

**非目标（明确不做）**
- ~~N1 HSK 官方词库体系化~~ → **已追加为目标，见 §10**（用户决议 2026-07-02 翻转）
- N2 苹果内购/订阅（本项目静态站，无商业化）
- N3 SQLite（坚持 localStorage + JSON，零依赖）
- N4 真图片资源（用 emoji 代替，后续可接 `docs/GPT生图`）
- N5 多人/联网排行榜（坚持本地单人进步记录）

## 2. 产品设计

### 2.1 排行榜产品形态
- **定位**：单人进步记录，不是多人 PK。「与自己赛跑」——看最佳成绩与近期趋势。
- **入口**：首页新增「🏆 排行榜」卡片 → 独立页 `page-leaderboard`。
- **展示**：按玩法（mathpk / hanzi / …）分 tab；每个玩法显示
  - 🥇 个人最高分（大字）
  - 最近 ≤10 次战绩列表（分数 + 日期 + 对错比等 meta）
  - 进步趋势 sparkline（emoji 八档 `▁▂▃▄▅▆▇█`，零依赖）
- **隔离**：按 `ProfileManager.getActiveId()` 隔离，每个孩子独立榜单；切孩子即切榜单。

### 2.2 汉字答题玩法
- **卡片四要素**：`汉字 char` + `拼音 pinyin` + `图标 emoji`（代图片）+ `例句 example`（`**` 标红目标字）。
- **出题模式**（一卡多用，`modes` 字段声明）：
  - `choose-char-by-pinyin`：看拼音选汉字
  - `choose-pinyin-by-char`：看汉字选拼音
  - `fill-blank`：例句填空（挖掉目标字）
- **流程**：选等级 → 动态出题（翻牌动画）→ 选答 → 即时正误动效 + 计分 → 下一题淡入淡出 → 结算 → 写积分 + 入榜。
- **计分**：答对 +基础分（如 +10），连击加成（可选），结算写 `addGrowthPoints` 归入学习力维度。
- **UI 调性**：卡片网格、圆角阴影「呼吸感」，对齐参考案例。

## 3. 数据模型

### 3.1 排行榜存储（localStorage）
```
key:   petbank_lb_{gameId}_{profileId}      // gameId ∈ {mathpk, hanzi, ...}
value: { "best": Number,
         "recent": [ {score, ts, ...meta} ] }   // recent ≤10，ts=Date.now()
```
gameId 命名规范冻结：`mathpk` / `hanzi`（小写、无空格）。profileId 取 `ProfileManager.getActiveId()`，异常/空兜底 `'default'`。

### 3.2 汉字题库 `data/hanzi-questions.json`
```json
{ "source": "汉字游乐场·汉字题库", "version": "1.0",
  "levels": {
    "1": [ { "char":"山", "pinyin":"shān", "emoji":"⛰️",
             "example":"我们一起去爬**山**。",
             "answer":"山", "opts":["山","水","火","木"],
             "modes":["choose-char-by-pinyin","choose-pinyin-by-char","fill-blank"] } ],
    "2": [], "3": []
  }
}
```
分级桶结构对齐 `data/math-cmath.json`，降低认知成本。levels 1-3 各放 8-12 条种子。

### 3.3 Leaderboard API（P0 冻结，P1 依赖）
```js
window.Leaderboard = {
  record(gameId, score, meta={}),   // 写入，维护 best+recent(≤10)，返回 {best,recent}
  getBest(gameId),                   // Number
  getRecent(gameId, n=10),           // [{score,ts,...meta}]
  renderUI(containerId, gameId, opts),
};
```

## 4. 交互流程

```
答题玩法（math-pk / hanzi 共用骨架）
  开始 → 出题 → 玩家作答 → 判对错 → 即时动效+计分
       →（未达终点）下一题淡入 ─┐
       →（达终点）结算 ─────────┘
            ├ addGrowthPoints(earned)         // 写入成长积分（学习力维度）
            └ Leaderboard.record(gameId, score, {correct,total})  // 入榜

排行榜查看
  首页「🏆 排行榜」→ switchPage('leaderboard')
    → Leaderboard.renderUI(container, 当前tab的gameId)
    → 按 profileId 读取并渲染 best + recent + sparkline
```

## 5. 备选方案与权衡

### 5.1 汉字玩法架构（已选 A）
| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| **A. 新建 hanzi-game.js 复用 math-pk 结算骨架** | UI/题库独立，计分/入榜/写积分复用；不污染 math-pk | 需提取或参照 math-pk 流程 | ✅ **已选** |
| B. 扩展 math-pk，汉字作为题库分类 | 改动最小 | math-pk 是「人机抢答竞速」框架，会扭曲汉字卡片玩法 | ✗ |
| C. 完全独立，不借 math-pk 代码 | 隔离最彻底 | 计分/结算重写，重复代码最多 | ✗ |

### 5.2 卡片「图片」要素
| 方案 | 结论 |
|---|---|
| emoji 代替图片 | ✅ **已选**（零资源成本，静态站友好） |
| SVG/真图 | ✗（成本高，留 N4 远期接 GPT 生图） |

### 5.3 排行榜形态
| 方案 | 结论 |
|---|---|
| 单人本地进步记录（最高分+趋势） | ✅ **已选**（贴合参考案例「与自己赛跑」+ 本项目本地存储调性） |
| 多人/家庭 PK 排名 | ✗（本项目无后端，多设备同步成本高，远期） |

### 5.4 存储
| 方案 | 结论 |
|---|---|
| localStorage + JSON | ✅ **已选**（与项目一致） |
| SQLite（参考案例用） | ✗（引入原生依赖，违背静态站原则） |

## 7. 风险与回滚

| 风险 | 缓解 |
|---|---|
| 排行榜 key 与旧 `petbank_math_high_score` 不兼容 | P0 已做 best-effort 幂等迁移（getBest=0 且旧值>0 时吞入，meta.migrated=true） |
| meta 字段在不同玩法不一致 | renderUI 按 meta 已知字段（correct/total/win）渲染，未知字段忽略；P1 若新增字段补 case |
| 汉字题库种子质量 | 起步 levels 1-3 各 8-12 条，人工可控；后续扩量走数据生成流程 |
| 切孩子时榜单串味 | key 强制带 profileId；切孩子重渲染 |

**回滚**：P0/P1 全部为新增文件 + 小幅接入改动，无破坏性删除；`git restore` 相关文件即可完全回滚。

## 10. HSK 词库扩展（追加需求 · 2026-07-02）

> 用户决议翻转原 N1 非目标。范围：**新 HSK 3.0 的 1+2 级**；粒度：**单字为主 + HSK 词组填空**；生产：**脚本批量 + SiliconFlow API 补单字例句**。

### 10.1 数据源（联网调研已确认）
- [ivankra/hsk30](https://github.com/ivankra/hsk30)：清洗过，含 pinyin/POS/繁体（首选）
- [drkameleon/complete-hsk-vocabulary](https://github.com/drkameleon/complete-hsk-vocabulary)：complete.json 全等级
- HSK 3.0 词量：1 级 ~300-500 词、2 级 ~500-1272 词（以实际词表为准）

### 10.2 生产流程（拆两步，降低阻塞）
1. **脚本（无需 API key，立即可做）**：拉词表 → 提取单字（去重，按 HSK 等级标注）→ 生成**词组填空题**（HSK 词组挖目标字，原文即上下文，免 LLM）+ 单字题骨架（`example` 留空待补）。输出 `data/hanzi-hsk.json`（`levels.hsk1` / `levels.hsk2`）。
2. **SiliconFlow API（需 key）**：仿 [prj/class-pet-garden-1/scripts/batch_generate.py](../prj/class-pet-garden-1/scripts/batch_generate.py) 模式，OpenAI 兼容端点，文本模型（Qwen2.5-72B / DeepSeek-V4），为每个单字补 `example`。抽检后合并。

### 10.3 玩法适配
- hanzi-game 等级选择新增「HSK 1 级 / HSK 2 级」入口（与启蒙关 levels 1-3 并存）。
- 词组填空复用 `fill-blank`；单字题复用 `choose-char-by-pinyin`。

### 10.4 依赖与风险
- **硬依赖**：SiliconFlow API key（来源待 §12-Q3 决议）。
- 风险：词表版本差异 / API 成本（千次级）/ 题库体积（上千题需分页或懒加载）/ 例句质量抽检。
