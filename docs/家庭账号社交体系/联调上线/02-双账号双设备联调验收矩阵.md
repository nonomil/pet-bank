# 双账号双设备联调验收矩阵

> 目标：用真实家长账号、真实家庭关系和至少两台设备，验证“多家长、多孩子、多家庭互动”是否真的闭环。

---

## 1. 推荐联调角色

建议至少准备 3 个家长账号、2 个家庭、3 个孩子：

| 角色 | 所属家庭 | 所在设备 | 用途 |
|---|---|---|---|
| 家长 A1 | 家庭 A | 设备 1 | 创建家庭 A、同步孩子 A-1 / A-2 |
| 家长 A2 | 家庭 A | 设备 2 | 接受家庭邀请码，验证同家庭协作 |
| 家长 B1 | 家庭 B | 设备 3 或浏览器隐身窗口 | 创建家庭 B、同步孩子 B-1、验证跨家庭好友 |

推荐孩子分工：

- 孩子 A-1：主串门 / 主 PK 发起方
- 孩子 A-2：同家庭孩子，用于验证“仅家庭可见 / 仅家庭可 PK”
- 孩子 B-1：跨家庭好友，用于验证好友边界

---

## 2. 验收前固定记录

每次联调开始前先记录：

- Supabase 项目名
- 前端访问地址
- 家长账号邮箱
- 家庭 A / 家庭 B 的 `household_id`
- 孩子 A-1 / A-2 / B-1 的 `child_id`
- 各自好友码

如果后面看到数据异常，没有这组基线会很难定位。

---

## 3. 联调矩阵

| ID | 场景 | 操作摘要 | 预期结果 |
|---|---|---|---|
| `A01` | 注册邀请码校验 | 用 `PARENT-BETA-*` 注册 3 个家长账号 | 无邀请码不能注册；有效邀请码可注册；使用后状态被 claim |
| `A02` | 家庭 A 创建 | 家长 A1 登录并创建家庭 A | 可拿到 `household_id`，家庭摘要正常显示 |
| `A03` | 多孩子同步 | 家长 A1 同步 A-1、A-2 或执行一键同步 | 两个孩子都出现在家庭 A，且各自有云端 `friend_code` |
| `A04` | 同家庭双家长协作 | 家长 A1 签发家庭邀请码，家长 A2 接受 | A2 加入同一家庭，能看到家庭 A 与 A-1 / A-2 |
| `A05` | 跨设备恢复 | 在设备 2 登录 A2，并切到本地不存在的孩子 | 能导入云端孩子壳；必要时恢复最近快照 |
| `B01` | 家庭 B 建立 | 家长 B1 创建家庭 B 并同步 B-1 | 家庭 B 独立存在，不和家庭 A 串档 |
| `B02` | 跨家庭加好友 | 用 A-1 的好友码让 B-1 兑换，或反向兑换 | 双方建立好友关系，可在社交页看到对方 |
| `B03` | 小屋可见性边界 | 把 A-1 `home_visibility` 改为 `private` | 同家庭 A-2 仍可见；好友 B-1 不应看到小屋摘要 |
| `B04` | 串门权限边界 | 把 A-1 `visit_access` 改为 `private` | A-2 仍能串门；B-1 的串门按钮不可用或请求失败 |
| `B05` | PK 权限边界 | 把 A-1 `pk_access` 改为 `private` | A-2 仍能发起 PK；B-1 不应成功发起挑战 |
| `C01` | 轻互动动作 | B-1 对 A-1 分别执行打招呼 / 送礼 / 一起遛弯 | 来访记录可见，活动流新增对应条目 |
| `C02` | 数学异步 PK | A-1 向 B-1 发起数学 PK，B-1 延后作答 | 双方看到同一题组；晚些提交仍能结算胜负 |
| `C03` | 汉字异步 PK | A-1 向 B-1 发起汉字 PK | 同一题组、可异步提交、结果页正常 |
| `C04` | 家庭内对战 | A-1 向 A-2 发起 PK 或串门 | 同家庭孩子即使 access 为 `private` 也可互动 |
| `C05` | 动态流完整性 | 检查 `review` / 动态流页 | 至少出现好友建立、来访、PK 发起、PK 提交、PK 完成 5 类事件 |

## 4. 每条案例建议抓取的证据

为避免“觉得好像通过了”，每个案例至少留一条证据：

| 场景 | 最低证据 |
|---|---|
| 注册邀请码 | `registration_invites` 状态变化截图 |
| 家庭邀请码 | 家庭邀请码弹层 + A2 接受成功截图 |
| 孩子同步 | `friend_code` 显示截图 |
| 跨设备恢复 | 设备 2 登录前后对比截图 |
| 可见性 / 权限边界 | 按钮禁用态或错误提示截图 |
| 数学 / 汉字 PK | 发起页、应战页、结果页 3 张图 |
| 动态流 | 事件时间顺序截图 |

补充建议：

- 跑 `B03` / `B04` / `B05` 时，立刻用 `node scripts/family-social-ops.mjs pair:inspect --child-a <child-a-id> --child-b <child-b-id>` 把“小屋可见 / 可串门 / 可发 PK”三条边界一起导出来，避免只看按钮状态误判为权限已生效。
- 跑完 `C02` / `C03` 后，立刻用 `node scripts/family-social-ops.mjs pk:inspect --match <pk-match-id>` 把题组 ID、题目数、双方成绩、用时和动态流顺序导出来，避免只凭页面观感判断“好像是同一套题”。
- 跑 `A05` 跨设备恢复前，先在两台设备的设置页分别设置清晰的诊断标签，再导出诊断 JSON；随后把两份快照交给 `node scripts/family-social-ops.mjs diagnostics:compare --left-json ./device-1-diagnostics.json --right-json ./device-2-diagnostics.json`，快速确认是不是同家庭、是不是少了云端孩子、恢复前后差异还剩什么。

## 5. 容易误判的点

以下情况很容易让人误以为“通过了”，但其实没有证明目标：

- 只在同一浏览器多个 tab 里测，不算跨设备恢复
- 只测同家庭孩子互相可见，不能证明跨家庭好友边界正确
- 只测发起 PK 成功，不能证明“双方拿到的是同一题组”
- 只看 UI 按钮禁用，不看真实写入结果，不能证明权限真的生效
- 只看前端 toast，不检查动态流，不能证明触发器写入成功

---

## 6. 建议的联调顺序

建议严格按下面顺序跑，避免脏数据干扰判断：

1. `A01` 到 `A05`
2. `B01` 到 `B05`
3. `C01` 到 `C05`

如果 `A05` 跨设备恢复都没通过，就先不要继续测好友和 PK。

补充说明：

- `A05` 现在可以直接在设置页的“云端档案恢复”区手动触发。
- 如果只是本地缺少孩子入口，先点“从云端导入孩子档案”。
- 如果要验证“云端快照覆盖本地旧数据”，再点“用云端覆盖本地数据”。
