# 上线检查与回滚手册

> 目标：在进入小范围内测前，明确什么叫“可以上线”，出了问题先怎么收口。

---

## 1. Go / No-Go 基线

以下 5 条全部满足，才建议进入小范围试运行：

1. 家庭账号社交回归测试基线保持通过
   - 当前参考值：`79 passed`
2. Supabase migrations 与 11 个 Edge Functions 已部署
3. 双家长 / 双家庭 / 双设备联调矩阵全部通过
4. `home_visibility` / `visit_access` / `pk_access` 边界已实测
5. 至少完成 1 轮数学 PK 和 1 轮汉字 PK 的真实闭环

如果第 3 条或第 4 条未完成，不建议对真实家长开放。

---

## 2. 推荐灰度节奏

### 阶段 0：开发联调

- 参与者：开发者自己
- 范围：本地站点 + 1 个 Supabase 项目
- 目标：把最小冒烟和联调矩阵跑通

### 阶段 1：内部双家庭测试

- 参与者：2 个家庭，3 个家长账号
- 范围：登录、家庭邀请、孩子同步、好友、串门、数学 / 汉字 PK
- 目标：验证真实人手操作时不会卡在流程里

### 阶段 2：小规模试运行

- 参与者：5 到 10 个家庭
- 范围：先开放家庭账号、孩子同步、好友串门，再逐步开放 PK
- 目标：观察是否出现数据串档、权限泄露、PK 结算异常

---

## 3. 上线前最后一轮检查

建议按这个顺序逐项勾：

1. 运行自动化基线
2. 确认测试项目与生产项目没有混用
3. 确认注册邀请码仍处于可控发放状态
4. 确认家庭邀请码过期与撤销链路可用
5. 确认小屋背景、好友码、动态流、家庭复盘在移动端也可见
6. 确认至少 1 个“本地无档案 -> 登录恢复云端孩子壳”的案例成功

---

## 4. 软回滚策略

如果问题只发生在云端新能力，而本地单机玩法仍正常，优先用软回滚：

### 方案 A：停止继续发码

- 停止发放新的 `registration_invites`
- 停止把 Supabase 配置提供给新的测试家庭

适用场景：

- 登录体系可用，但担心继续扩大测试范围

### 方案 B：把社交边界全部收紧

执行：

```sql
update public.child_profiles
set home_visibility = 'private',
    visit_access = 'private',
    pk_access = 'private';
```

适用场景：

- 发现跨家庭可见性、串门或 PK 权限有泄漏风险

### 方案 C：撤销待处理邀请码

执行：

```sql
update public.household_invites
set status = 'revoked'
where status = 'pending';

update public.registration_invites
set status = 'revoked'
where status = 'pending';
```

适用场景：

- 希望冻结新增家庭和新增家长加入

---

## 5. 分能力收口建议

不同问题，不必整套功能一起下线：

| 问题类型 | 建议收口方式 |
|---|---|
| 注册邀请码异常 | 停止发放新邀请码，保留已登录家庭 |
| 家庭邀请异常 | 暂停 `issue-household-invite` / `accept-household-invite` 流程，保留已有家庭 |
| 好友串门异常 | 统一把 `visit_access` 收紧为 `private` |
| PK 结算异常 | 先暂停数学 / 汉字 PK，对好友串门保持开放 |
| 动态流异常 | 保留主链路，单独下掉动态摘要展示 |

核心原则：

- 优先保住本地养成与已同步孩子
- 再保住家庭关系
- 最后才是跨家庭互动和 PK

---

## 6. 硬回滚的注意点

当前 schema 已经把家庭、孩子、好友、PK、动态流作为一整套表结构落下去了。

因此不建议在试运行期频繁做“删表级硬回滚”，而建议：

- 通过邀请码、配置注入和权限收紧来止损
- 先冻结入口，再分析具体坏点
- 在修复后继续用同一套项目复测

也就是说，这一轮更适合“关入口、收权限、保数据”，而不是“直接推翻数据库”。

---

## 7. 试运行结束后的沉淀

每完成一轮试运行，至少沉淀下面 4 项：

1. 通过案例数 / 失败案例数
2. 失败案例对应的账号、家庭、孩子 ID
3. 是否需要新增运维脚本或管理页面
4. 哪些能力可以从“试运行”升级为“默认开启”
