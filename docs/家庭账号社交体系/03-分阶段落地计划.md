# 家庭账号社交体系分阶段落地计划

> 当前代码进度：`R1 登录 + 家庭 + 多孩子云端化`、`R2 好友码 + 串门`、`R3 数学异步 PK`、`R4 汉字异步 PK + 家庭复盘/好友战绩摘要 + 云端动态流` 已完成前端与合同层闭环，后续重点转向真机联调与上线验收。

## 1. 总体策略

这次不建议“一口气把完整社交平台做完”，而建议按四层递进：

1. **先立云端身份和家庭模型**
2. **再把本地孩子档案接到云端**
3. **然后做好友与串门**
4. **最后做数学/汉字异步 PK**

这样可以最大化复用现有养成玩法，同时把风险拆开。

---

## 2. 阶段划分

### M0：文档、选型、数据合同

目标：

- 冻结产品模型
- 冻结后端选型
- 冻结核心表结构与权限边界

产出：

- 本专题目录文档
- `docs/plans` 下设计稿、实施计划、任务清单
- 后端选型结论

验收口：

- 能回答“为什么不能继续只用 localStorage”
- 能回答“为什么是 household + child，而不是 adult + pet”
- 能回答“邀请码分几种、分别干什么”

### M1：账号、家庭、多孩子云同步

目标：

- 家长能注册登录
- 家长能创建家庭
- 家庭里能创建多个孩子
- 本地 profile 能导入到云端 child

范围：

- Auth
- household
- household_members
- children
- child_pet_state
- child_home_state
- profile 导入

验收口：

- 两台设备登录同一账号能看到同一个家庭
- 同一个家庭能看到 1 个以上孩子
- 现有 profile 数据能导入，不丢核心宠物进度

### M2：家庭协作与好友关系

目标：

- 主家长能邀请另一位家长加入家庭
- 不同家庭的孩子能建立好友关系
- 好友孩子之间能进入彼此小屋

范围：

- household invite
- friend code
- friendship
- open house view
- visit footprints

验收口：

- 一个家庭里有两个家长账号
- 两个家庭各有一个孩子，并能成为好友
- 好友能看到彼此开放的小屋并留下来访记录

### M3：数学异步 PK

目标：

- 好友孩子之间能发起同题数学 PK
- 双方可先后作答
- 系统可自动结算胜负

范围：

- pk match
- question set freeze
- attempt record
- result settle
- 好友挑战入口

验收口：

- A 先发起，B 晚点作答，仍能正常结算
- 双方看到的是同一组题
- 结果能写进动态或战绩记录

### M4：汉字异步 PK 与动态流

目标：

- 汉字/识字玩法接入同题异步 PK
- 来访、礼物、PK 结果进入动态流

范围：

- hanzi question set
- hanzi attempt
- activity feed
- 简单提醒与结果页

验收口：

- 汉字 PK 能完整闭环
- 家长能看到“谁来串门、谁发来 PK、谁赢了”的摘要流

## 3. 推荐优先级

### 必须按顺序推进

1. `M1 账号与家庭`
2. `M2 好友与串门`
3. `M3 数学 PK`
4. `M4 汉字 PK`

原因：

- 没有家庭与孩子真源，就没有后续关系
- 没有好友关系，就无法做对战目标
- 数学 PK 比汉字 PK 更容易先标准化题组结构

### 可并行的小快修

- `清晨阳光房` / `花园阳台` 背景图已完成接回
- 继续整理 `profiles.js` 的导入接口，让它更适合作为云端同步桥
- 继续把 `leaderboard.js` 的本地榜逻辑抽象成可复用的“战绩视图”

### M5：真实环境联调与上线验收

目标：

- 把当前 schema 与函数部署到真实 Supabase 项目
- 验证双家长、双设备、双家庭的真实联调
- 验证动态流与访问边界的真实写入结果
- 准备灰度与回滚手册

范围：

- Supabase 项目配置
- 注册邀请码种子数据
- 双账号双设备验收矩阵
- go / no-go 与回滚清单

验收口：

- 双家长能在不同设备看到同一家庭
- 本地无档案时可恢复云端孩子壳与最近快照
- `home_visibility` / `visit_access` / `pk_access` 边界真实生效
- 数学 / 汉字异步 PK 与 `activity_feed` 在真实环境闭环

## 6. 风险与控制点

### 6.1 风险：做得太像“平台重构”

控制方式：

- 保留现有前端玩法壳
- 后端只补“身份、关系、同步、结算”
- 不在第一版重写所有模块

### 6.2 风险：一次上太多社交形态

控制方式：

- 先做好友制，不做广场制
- 先做异步，不做实时
- 先做来访和 PK，不做聊天

### 6.3 风险：本地老数据迁移复杂

控制方式：

- 以 `profiles.js` 现有快照结构为导入桥
- 首次导入显式触发，不做暗中覆盖
- 导入后保留本地缓存作为回滚兜底

---

## 7. 需要提前决策的事项

### 7.1 注册方式

建议拆成两阶段：

- 内测 / 第一版：邮箱或免短信方案，先跑通账号和关系模型
- 面向家长公开版：再评估手机号注册或短信 OTP

### 7.2 好友关系方向

建议默认双向确认，避免孩子被陌生家庭直接打扰。

### 7.3 小屋开放程度

建议默认 `friends_only`，不要默认对所有人开放。

---

## 8. 结论

如果只看功能名，这像是“登录系统 + 社交系统 + PK 系统”。

但从落地顺序看，它其实应该被理解成：

1. 先给现有养成产品补上**家庭身份层**
2. 再给孩子补上**好友关系层**
3. 最后把现有数学/汉字玩法接成**异步对战层**

这三层顺序不能反。
