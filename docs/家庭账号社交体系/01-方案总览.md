# 家庭账号社交体系方案总览

## 1. 背景与核心判断

当前项目已经从“单孩子单设备”升级到“同设备多孩子切换”，具体实现落在 `js/profiles.js`。这一步已经证明：**孩子档案隔离** 是真实需求，而且现有产品形态能承接多孩子。

但你现在提出的新目标已经超过“本地多孩子”：

- 多个家长分别注册多个账号
- 每个家庭下可以有 1 个或多个孩子
- 孩子的宠物之间可以串门、互动、送礼物、做 PK
- 数学题可以异步 PK，同一组题、先后作答都能结算
- 汉字/识字也可以做同样的异步 PK

这意味着系统不再只是“一个浏览器里的几个 profile”，而是一个**跨账号、跨家庭、跨设备**的关系网络。

**结论：**
当前纯前端 `localStorage` 架构已经不够，必须引入云端数据真源。

---

## 2. 产品目标

### 2.1 主目标

把“孩子和系统玩”升级成“孩子和别人家的宠物、别人家的孩子玩”，形成持续关系，提升留存。

### 2.2 次目标

- 同一个孩子支持多个家长共同查看和陪伴
- 一个家庭支持多个孩子
- 不把现有养成主链路打碎
- 先做异步社交，不一开始就做实时在线

### 2.3 成功标准

- 家长能注册账号并创建家庭
- 家长能在家庭里新增 1 个或多个孩子
- 家长能邀请另一位家长加入同一家庭
- 孩子能与其他家庭孩子建立好友关系
- 好友之间能串门、留言、送礼、发起数学/汉字 PK
- 同一孩子在不同设备登录后能恢复同一份宠物数据

## 3. 推荐产品模型

推荐把产品主体从“孩子档案”升级为“家庭空间”：

```text
家长账号
  └── 家庭
        ├── 家长成员（主账号 / 协作家长）
        ├── 孩子 A
        │     └── 宠物 A
        ├── 孩子 B
        │     └── 宠物 B
        └── 家庭设置 / 邀请 / 隐私

孩子
  ├── 小屋
  ├── 积分与成长
  ├── 探索 / 背包 / 图鉴
  ├── 数学 PK 记录
  ├── 汉字 PK 记录
  └── 好友关系（其他家庭的孩子）
```

### 3.1 为什么不是“成人账号直连宠物”

因为真正的长期留存主体不是家长，而是孩子。

如果用“家长账号 -> 宠物”直连：

- 爸爸和妈妈看同一个孩子会冲突
- 一个家庭多娃时层级会乱
- 孩子与孩子之间的好友关系会被错误地挂在成人账号上

所以更稳的结构是：

`家长账号 -> 家庭 -> 孩子 -> 宠物`

---

## 4. 角色与权限模型

### 4.1 家长账号

每个家长都有独立登录账号，用于：

- 登录
- 接受邀请
- 查看自己所在家庭
- 操作被授权的孩子

### 4.2 家庭

家庭是数据归属核心，负责承接：

- 多个家长
- 多个孩子
- 统一邀请码 / 邀请链接
- 家庭级隐私设置

### 4.3 孩子

孩子是养成与社交主体，负责承接：

- 宠物
- 小屋
- 道具与成长数据
- 好友关系
- PK 记录

### 4.4 推荐权限级别

| 角色 | 典型身份 | 权限 |
|---|---|---|
| `owner` | 家庭主创建者 | 管理家庭、邀请家长、增删孩子、处理隐私 |
| `guardian` | 爸爸/妈妈/爷爷奶奶等协作家长 | 查看和操作家庭内孩子，不能删家庭 |
| `friend` | 其他家庭的孩子 | 只能访问开放的串门与 PK 表面 |
| `guest` | 未建立关系的外部用户 | 只能看到注册/邀请入口 |

## 5. 邀请码体系

邀请码不要混成一个码，建议拆成三类：

### 5.1 注册邀请码

用途：

- 活动拉新
- 白名单测试
- 渠道归因

特点：

- 在注册页填写
- 决定新账号是否允许进入系统
- 可选，不一定要永久保留

### 5.2 家庭邀请码

用途：

- 邀请另一位家长加入同一家庭

特点：

- 绑定到 `householdId`
- 有有效期
- 只能加家庭成员，不能跨到孩子好友关系

### 5.3 宠物好友码

用途：

- 让两个不同家庭的孩子建立互动关系

特点：

- 绑定到 `childId`
- 可做成短码或分享链接
- 建立成功后形成双向好友关系

---

## 6. 串门与互动设计

推荐优先做**异步互动**，不做实时房间。

### 6.1 串门 MVP

孩子进入好友小屋后可以：

- 看对方的小屋主题与摆设
- 看到对方宠物状态摘要
- 留下一次来访足迹
- 送一个小礼物
- 触发一个互动动作，比如“打招呼”“一起遛弯”“一起玩球”

### 6.2 为什么先异步

- 开发复杂度低
- 不需要在线匹配
- 更适合家庭陪伴场景
- 家长不必担心孩子需要“同时在线”

### 6.3 后续可扩展

- 来访留言板
- 每日互助任务
- 好友亲密度
- 联合挑战

---

## 7. 数学 / 汉字异步 PK 设计

### 7.1 数学 PK

核心不是“同时按抢答”，而是“同一组题，系统可比对”。

推荐结构：

1. 发起方创建一场 PK
2. 系统冻结一组题目 `questionSetId`
3. 先作答的一方提交成绩
4. 后作答的一方在有效期内完成同一组题
5. 系统按正确数、用时、提交时间结算

### 7.2 汉字 PK

与数学 PK 同构，只是题库类型换成：

- 识字选图
- 拼音配字
- 常用词辨认
- HSK 分级题

### 7.3 为什么要“同题组”

如果双方题不一样，PK 就不是公平比较。

所以数学和汉字 PK 都建议使用：

- `match`
- `question_set`
- `attempt`

这三个概念来冻结题目、记录结果、统一结算。

## 8. 推荐技术路线

### 8.1 推荐方案

**静态前端保留 + Supabase 轻后端**

具体组成：

- 前端：继续保留当前 `HTML + CSS + Vanilla JS`
- 认证：Supabase Auth
- 数据库：Supabase Postgres
- 权限：RLS
- 服务逻辑：Supabase Edge Functions

### 8.2 为什么不继续只用 localStorage

因为它做不到：

- 多账号登录
- 多设备同步
- 邀请码校验
- 家庭成员关系
- 跨家庭好友关系
- 异步 PK 共享题组
- 来访记录持久化

### 8.3 为什么不推荐先上完整自建后端

当前仓库是无构建、无运维的静态前端项目。

如果直接改成完整自建后端：

- 会把工程复杂度一下子拉高
- 需要长期运维账号、安全、数据库
- 会拖慢当前产品验证

所以建议先上“带鉴权和数据库的托管后端”，不要第一步就自建整套服务。

## 9. MVP 范围建议

### 9.1 必做

- 家长账号注册 / 登录
- 家庭创建
- 家庭邀请另一位家长
- 家庭下新增多个孩子
- 现有 profile 数据导入为孩子
- 孩子好友码 / 好友关系
- 串门（查看小屋 + 足迹 + 简单互动）
- 数学异步 PK

### 9.2 第二阶段再做

- 汉字异步 PK
- 礼物系统
- 关系链 feed
- 好友榜单
- 家庭消息中心

### 9.3 暂不做

- 实时在线房间
- 语音聊天
- 陌生人广场
- 随机匹配
- 复杂公会 / 班级体系

## 10. 与当前代码的衔接策略

### 10.1 可以保留的主链路

当前这些模块的玩法逻辑大多还能保留：

- `app.js`
- `pet.js`
- `home.js`
- `exploration.js`
- `inventory.js`
- `math-pk.js`
- `hanzi-game.js`

### 10.2 需要新增的一层

要新增一层“云端身份与数据同步”：

- `auth.js`
- `cloud-client.js`
- `household.js`
- `social.js`
- `profile-sync.js`
- `pk-service.js`

### 10.3 已有 `profiles.js` 的新定位

未来不应再把 `profiles.js` 当成终态方案，而应把它降级成：

- 本地多孩子快速切换层
- 老数据导入桥
- 离线缓存兜底

---

## 11. 产品结论

这次的目标本质上是把项目从：

`单机养成工具`

升级成：

`家庭账号驱动的轻社交养成产品`

最关键的判断不是“加一个登录页”，而是：

1. 数据归属要从 `profile` 升级到 `household + child`
2. 社交主体要从“家长”升级为“孩子/宠物”
3. 互动先做异步，再考虑实时
4. 技术上必须有云端真源
