### 1.2 遗漏点（建议补进方案）

**a. 卡牌对战入口位置未定（中等遗漏）**
- 方案 §4.1 仅写「独立页面/入口」，但没说挂哪个 tab。
- 代码事实（`index.html:29-33` + `app.js:485 PAGE_TO_TAB`）：现有 5 个一级 tab = 首页/积分/宠物/探索/更多，**无第 6 个 tab 余量**（底部导航已 5 个，再加会挤）。
- 图鉴卡入口在「宠物」tab（`PAGE_TO_TAB.card='pet'`）。
- **建议明确**：卡牌对战入口放「宠物」tab 下作为子页（`PAGE_TO_TAB.arena='pet'`），与图鉴同 tab 收口养成闭环；或「探索」tab 下（与探索战斗同属战斗域）。方案需在 §4.1 二选一并登记 `PAGE_TO_TAB`。

**b. 奖励循环只有一句话，缺数值与触发点（中等遗漏）**
- §5.2 关卡数据只有 `{points:50, dropCard:"xxx"}` 示意，但未定义：
  - 积分奖励接入哪个系统（`CardCollection.checkSeriesRewards` 走的是系列补全，关卡积分是否进 PetSystem/积分系统？）；
  - `dropCard` 与现有「25% 掉宠物图鉴卡」是否重叠（探索胜利用 `app.js:1193` 的加权池，卡牌对战走关卡 `dropCard` 直给？两套并存会割裂）。
- **建议补**：奖励分三类明确来源——①关卡积分（进积分系统）②掉卡（沿用 25% 加权池，复用 `app.js:1193` 逻辑抽公共函数）③成就/解锁（关卡进度），并在 §3.4 列触发点。

**c. 与图鉴收集的联动未显式（轻量遗漏）**
- §0 目标提「收集的宠物卡组队」但没写**对战掉卡反哺图鉴**这一闭环。
- 现状掉卡是 `CardCollection.addCard(petSpecies.id)`（宠物物种图鉴卡），卡牌对战若掉「敌方宠物卡」天然补图鉴。
- **建议补**：§3.4 明写「卡牌对战胜利掉卡 → 复用 `CardCollection.addCard` → 图鉴解锁/系列奖励」，画进养成闭环图。

## 2. 决策合理性审查

### 2.1 D4（def/spd 新引擎统一接入）—— ⚠️ 方向对，但缺「回归护栏」

- **合理点**：调研已证明 atk/def/spd/技能四维天然适配宝可梦式，def 当前为 0 不接入等于废维。统一接入避免「探索一套公式、对战另一套」的双轨维护。
- **隐患**：D4 推荐「探索+对战都用」= **探索战斗伤害公式要改**。现状探索四条公式（`exploration.js:348/384/398/437`）都不读 def，一旦接入 `atk-def/2`，**已平衡的探索关卡伤害曲线会变**（怪物 atk 量级若大于宠物 def/2，宠物挨打更疼；反之更耐打）。
- **必改**：D4 落地前需补一条约束——「探索迁移后伤害曲线需回归验证（同一怪物同等级，迁移前后伤害偏差 ≤ 阈值）」，否则属于「无声改平衡」。怪物数据（`scenes.json`）无 def/spd 字段，def/2 对怪物侧 = 0，怪物挨打不变；**主要风险是宠物挨打侧**（怪物 atk - 宠物 def/2，宠物变耐打，探索变简单）。

### 2.2 D5（battle-engine 通用重构）—— ✅ 合理，与调研 §6.3 一致

- 调研明确「`battleTurn` 去硬编码 + 抽 combatant」是推荐路径（§6.2/6.3）。
- 风险点在迁移回归（见 §3.1），决策本身合理。
- **补充**：combatant 抽象字段（§5.1）需覆盖探索现状的「怪物无 def/spd」——`scenes.json` 怪物只有 7 字段（id/name/emoji/hp/atk/exp/drops），**combatant 工厂对怪物要给 def=0/spd=0 默认值**，方案 §5.1 已写但应在 §3.2 公式旁注明「怪物 def=0 时退化为纯 atk，与现状完全一致」。

### 2.3 D6（SSR≤1）—— ✅ 合理，但需补「SSR 数量从哪数」

- 合理：防 3 SSR 碾压，鼓励低稀有度上场，符合儿童向。
- **待澄清**：方案没说「SSR 限制是按队伍还是按全局」。建议按**单支队伍 ≤1 SSR**（玩家可有多张 SSR 卡，但单场出战限 1 张），与「鼓励收集但限制滥用」匹配。
- 另需补：玩家若 SSR 卡不足 1 张（新手），是否允许全 R/SR 上场？显然允许，但方案应明写「≤1 是上限非下限」。

## 3. 风险遗漏审查

### 3.1 探索战斗迁移回归风险（方案 §8 已提，但未展开执行级风险）

方案 §8 只说「保证不回归」，但**没列回归检查清单**。以下 6 项是 P0 必须显式验证的回归点：

| # | 回归点 | 现状代码 | 迁移后必须不变 |
|---|--------|----------|----------------|
| R1 | 掉卡 25% 加权池 | `app.js:1193-1201` | 胜利后仍触发 `CardCollection.addCard` |
| R2 | 技能 CD tick（本回合启动的不递减） | `pet.js:388-394` `tickCooldowns(exclude)` | 引擎必须保留 exclude 参数 |
| R3 | 逃跑（0.3 概率 + 失败白嫖一击 + 30% exp） | `exploration.js:338-356` | 逃跑分支整体迁移 |
| R4 | 胜负结算（exp/drops/rare_drops/addWin） | `exploration.js:407-434` | 探索走 PVE 结算钩子，对战走独立钩子 |
| R5 | showEnding（galgame 入口） | `exploration-detail.js`（closeBattleModal 分叉） | galgame 战斗的结束回调不能断 |
| R6 | UI 锁 `battleUILocked`（防连点） | `app.js:1092/1050` | 引擎迁移后锁释放时序不变（调研记的「上次战斗锁残留卡死」bug 已修，别回归） |
| R7 | **敌人反击公式变化** | `exploration.js:437` 现 `max(1, monster.atk + rand(0..1) - 1)` 不读 def | 迁移后变 `max(1, (monster.atk - pet.def/2)...)`——**这是静默改平衡，见 §2.1** |

**R7 是最隐蔽的风险**：方案说「不回归」但 D4 本身要求探索也接入 def，敌人打宠物会变轻（宠物有 def 减伤），探索难度下降。**这不是 bug 是设计选择，但必须在方案里显式声明「探索接入 def 后难度预期下降，可接受/需补偿」**，否则上线后体感「怎么怪物打不动我了」会成投诉。

### 3.2 combatant 抽象通用性 —— ⚠️ 字段差异需补

方案 §5.1 combatant 字段：`{id,name,img,side,hp,maxHp,atk,def,spd,skills,defending}`。

- **宠物侧**：`pet.js` 有 `level/weapon/armor/skills开放列表/cooldowns(闭包非state)`。combatant 是否带 level？显示要 Lv（`app.js:1066`）。**建议补 level 字段**。
- **怪物侧**：`scenes.json` 怪物无 def/spd/skills，只有 emoji 兜底（`app.js:1071` onerror 切 emoji）。**怪物无 skills → 怪物只能普攻？**现状探索怪物就是纯普攻反击（`exploration.js:437`），但卡牌对战的「AI 队伍 3 只宠物/怪物」若用真宠物卡当敌人，敌人就有技能了。**combatant.skills 对探索怪物应是空数组（只普攻），对卡牌对战敌人是非空**——方案需注明这个分叉，否则 AI 贪心规则（§3.4「CD 好优先技能」）对无技能怪物会出错。
- **装备**：宠物有 weapon/armor 加成（`getTotalAtk/Def`），怪物无装备。combatant 的 atk/def 应是「**已聚合的最终值**」（宠物侧调 getTotalAtk 填入），而非裸 base——方案 §5.1 没说聚合时机，建议明写「combatant.atk = 现役宠物的 getTotalAtk()，构造时一次性聚合」。

### 3.3 3v3 换人逻辑 —— ⚠️ 二义未定

方案 §3.3：「当前宠物 HP 归零 → 自动换下一只（**或**玩家手动换人，耗回合）」。

- **二义**：自动换 vs 手动换到底是哪个？调研 §5.2 说「换人占用本回合动作（参考宝可梦）」，但方案 §3.3 用「或」字模糊。
- **建议二选一并写死**：
  - 推荐方案：**HP 归零 = 自动换人（不耗回合，因为玩家无法选择已倒下的）；存活时手动换人 = 耗回合动作（占 4 动作外的第 5 动作「换人」）**。
  - 这样自动换是被动触发，手动换是主动战术，两者不冲突。
- 还需补：**首发按 spd 排序**（§3.3 已写），但「换上的替补是否影响后续 spd 先手判定」？spd 是当前上场宠物的，换人后先手判定用新上场宠物的 spd——方案应明写。

### 3.4 AI 贪心规则 —— ⚠️ 可能太强（ultimate 滥用）

方案 §3.4 AI 规则：「CD 好优先技能」「血低优先防御」「血量低于阈值换人」。

- **太强风险**：「CD 好优先技能」= ultimate(×3) CD5 一好就放，3v3 中 AI 每只都这么干，玩家会被 3 倍伤害压制。儿童向产品 AI 不该这么凶。
- **太弱风险**：纯贪心无预判，玩家学会「防御接反击」就能稳定通关，后期无挑战。
- **建议**：AI 分难度档（调研 §4.2 已提「低关卡 AI 不用技能，高关卡 AI 会预判」），方案 §3.4 应把「难度梯度」从一句话展开成规则表：低关卡 AI 50% 概率普攻/不用 ultimate；高关卡才完整贪心 + 换人。**关卡数据 `arena-stages.json` 应带 `aiLevel` 字段**（方案 §5.2 漏了）。

### 3.5 关卡平衡 + def/spd 缺省 —— ✅ 基本安全，补两点

- def/spd=0 缺省：公式 `max(1, atk-def/2)` def=0 退化为 atk，`pet.js:120-121` 已 `|| 0` 兜底，**安全**。方案 §8 已写。
- spd=0 平手：§3.2「随机/玩家先手」——**建议固定「玩家先手」而非随机**，儿童向产品玩家先手体验更好，且可预测。
- **关卡平衡未给数值**：§5.2 关卡只有结构无数值范围。建议 P2 前先定「首关敌方 atk ≈ 玩家初始宠物 atk，末关 ≈ 2x」之类的曲线锚点，否则关卡设计师无依据。

### 3.6 卡牌对战入口 —— 见 §1.2-a（中等遗漏，非纯风险）

## 4. 可执行性审查

### 4.1 P0→P1→P2 顺序 —— ✅ 合理

依赖链清晰：引擎先行（P0）→ 迁移验证不回归（P0）→ 3v3（P1）→ 闯关（P2）→ PvP 彩蛋（P3）。无循环依赖。

### 4.2 P0 内部缺验证护栏 —— ⚠️ 必补

P0 三件事（引擎 / 探索迁移 / showBattleModal 改读 combatant）**耦合在一次提交里风险过高**。建议 P0 内部拆三步带验收：

| 步 | 内容 | 验收门禁 |
|----|------|----------|
| P0.1 | 建 `battle-engine.js`（combatant + 公式 + 4动作），**探索战斗暂不切** | 引擎单测：def/spd 公式、CD tick、checkFaint |
| P0.2 | 探索 `currentBattle.monster` → combatant，切新引擎 | **回归测试**：§3.1 的 R1-R7 全过；同怪物迁移前后伤害曲线对比 |
| P0.3 | `showBattleModal` 读 combatant（双方立绘/HP） | 探索 1v1 UI 不变 + 战斗日志/浮动伤害/抖动正常 |

**P0.2 是最高风险步**，建议单独 commit + 手测探索全 12 场景至少 3 场（含 galgame 入口场景，验 R5 showEnding）。

### 4.3 缺「战斗动画 dispatch 事件」补回

调研 §6.2 末记：`battle-animate` 事件在 `exploration.js` 无 dispatch（死代码），`handleBattleAnimate`（`app.js:1247`）靠它驱动抖动。**迁移引擎时若不补 dispatch，抖动动画会彻底失效**（现状可能已部分失效）。
- 建议 P0.2 顺手补：引擎执行攻击后 `dispatchEvent('battle-animate', {target})`，复活动画链。方案未提此项，属遗漏。

## 6. 结论

**方案可推进**。核心选型（宝可梦式 3v3 PvE）与调研完全一致，def/spd 接入方向正确，UI/引擎复用判断成立。

**关键风险**：P0.2 探索迁移是静默改平衡（R7）+ 回归面广（R1-R7）+ 动画事件可能失效，是整个方案最高危一步。

**必改项（P0 重点）**：①声明 def 接入对探索难度的方向性影响 ②补 7 项回归检查清单 ③补 battle-animate dispatch ④换人逻辑二义写死。这四项补齐后 P0 可安全动手。

**可选优化**：入口定死、AI 分难度档、combatant 字段补全、奖励循环/图鉴联动展开——这些不影响 P0 启动，但 P1/P2 前需补。
