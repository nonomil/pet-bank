## 二、决策合理性评估

### D1：def + spd 两维（不加 crit/skill）— ✅ 合理

- **支撑**：现 `state` 仅 hp/atk（`js/pet.js:60-62`），战斗公式（`js/exploration.js:398`）只看 atk，无防御/暴击/闪避深度。def+spd 是最小增量、最大收益。
- **不加 crit/skill 合理**：crit 会触发数值膨胀 + 暴击 UI；skill 已有全局技能表（`data/skills.json` 3 技能），加 per-pet skill 改造面过大。两维恰够卡牌展示且 P4 平滑接入战斗。
- **风险**：def/spd 新增后 `state` 对象（`js/pet.js:56-78`）需同步加字段并持久化，方案 §2.3 仅说"暴露 getTotalDef/getTotalSpd"，未明确 state 结构改动（见风险 R1）。

### D2：CSS 先行 + Agnes 增强 — ✅ 合理（强烈推荐保留）

- **支撑**：CSS 边框可立即修复"看不见"问题，不阻塞在 AI 生图（生图 KEY 有效性/透明区质量均有不确定性）。
- **可信度**：现 CSS 已有 `.card-rarity-*` 边框（`css/card-collection.css:142-146`）作为基础，增强为 box-shadow/渐变是渐进式改动，工程量可控。
- **降级策略**（§4.3）明确"P0 CSS 不依赖生图"，验证标准 §6.5 双机制但 §6.2 卡牌式允许"CSS/PNG"——降级路径自洽。

### D3：仅展示不改战斗 — ✅ 合理

- **支撑**：`battleTurn`（`js/exploration.js:323-444`）当前 4 处伤害公式（普攻/技能/反击/逃跑）均不读 def，强行接入需改 4 处 + 怪物配置加 def + 平衡回归测试，风险高。
- **本轮边界清晰**：P1 仅加数据 + PetSystem 暴露 + 卡牌展示，P4 战斗接入列为可选。降低本轮爆炸半径，合理。

### D4：3:4 竖卡（网格 180×240 / 详情 360×480）— ✅ 合理

- **支撑**：调研 §2.1 显示炉石/百闻牌/PVZ 均为竖卡 ~3:4，移动端友好。2x 详情卡（360×480）符合 @2x 渲染惯例。
- **备注**：方案 §3.1 提"大类卡保持现有 160×宽"——需确认大类卡（`_renderCategory`，`js/card-collection.js:119-152`）是否也应卡牌化。方案未明确大类卡的卡牌化范围（见 R3）。

### D5：四角数值（HP左上/SPD右上/DEF左下/ATK右下）— ✅ 合理（有一处微调建议）

- **支撑**：炉石标杆是"ATK左下/HP右下"，方案 §3.4 采取"HP左上/ATK右下"，HP 不在右下。**与调研 §5.1 炉石惯例（HP右下/ATK左下）有出入**，但方案 D5 的"上下排分流（HP+SPD 上，DEF+ATK 下）"逻辑自洽（战斗最关注的 HP/ATK 分置对角），辨识度反而更高。
- **微调**：方案 §3.4 文字写"HP左上/SPD右上/DEF左下/ATK右下"，但调研 §5.2 方案A 图示是"HP左上/SPD右上/DEF左下/ATK右下"——两者一致。但调研 §4.4 CSS 骨架示例用的是"HP左下/ATK右下/DEF左下偏上/SPD右下偏上"（bottom 定位），**两份文档的角标位置代码示例不统一**。实施时需以方案 §3.4 四角为准（见 R4，非阻塞，实现细节）。

## 三、风险评估

### R1【P0】state 对象无 def/spd 字段 — 方案 §2.3 描述不准确
- **现状**：`js/pet.js:56-78` state 对象字段为 species/level/exp/hp/max_hp/atk/wins/...，**无 def/spd**。选宠逻辑（`:107-109`）只映射 `state.max_hp=species.base_hp; state.hp; state.atk=species.base_atk`，**不映射 def/spd**。
- **影响**：方案 §2.3 称"getState() 含 def/spd"——但当前 getState 返回的 state 无此字段。P1 必须同步改：①state 对象加 `def`/`spd` 字段；②选宠逻辑映射 `species.base_def/base_spd`；③load() 兼容旧存档（无 def/spd 时补默认值）。
- **严重度**：中（不影响 P0，但 P1 实施时若漏改会导致 getTotalDef 读 undefined）。方案需补充 state 结构改动步骤。

### R2【P1】classpet 缺图兜底需验证统一性
- **现状核实**：classpet 记录（如 `pets.json:3476-3486` 小猫咪）**完全没有 stages/imageStages/imageStyle/imageUrl 字段**（非 None，是字段缺失）。调研称"40 只"，实测 `"source":"classpet"` 在文件中出现 80 次（嵌套结构 + flat 数组各含）。CardCollection 用 `getAllSpecies()` 返回 flat[]。
- **方案兜底**：§7 称"立绘用 emoji 兜底（同 pet 页方案），角标数值仍显示"。
- **缺口**：方案未说明 **CardCollection 渲染逻辑如何区分"有图/无图"宠物**。当前 `_renderGrid`（`js/card-collection.js:155-172`）只渲染 emoji+name，无图片加载逻辑——P0 卡牌化需新增"有图则显示立绘 PNG，无图则 emoji 兜底"的分支。方案 P0 描述（"四层叠加：边框+立绘+暗角+HTML数值"）隐含所有宠物都有立绘，**与 classpet 无图事实冲突**。
- **建议**：P0 渲染重构需显式加入 source/图字段判空的 emoji 兜底分支，并在方案中点名。

### R3【P1】数值平衡梯度细节需复核
- def/spd 梯度表（§2.2）：common def 2~5 / legendary def 11~15。但调研 §3.1 显示 common atk 均值 6.5、legendary atk 均值 13.2，def 区间（2~5 vs 11~15）与 atk 区间近似同梯度。
- **潜在问题**：def 用 `def×0.5` 减伤（§2.2 P4），legendary def 13 → 减伤 6.5，对 common 怪 atk 3~6 可能导致"减伤后 0 伤害"（普攻公式 `max(1, atk-def*0.5)`）。需确认 P4 接入时是否引入怪物 def 配置同步。**本轮 P4 不做，故不阻塞，但梯度设计需在 P4 前复核。**
- 另：§2.2 legendary 行 hp 写"134~160"（沿用现有），与 epic 行"130~160"重叠区间大，legendary 与 epic 区分度低（仅 6 只 legendary 占 4%，可接受）。

### R4【低】未收集策略是否解决"看不见"根因 — 部分解决
- **根因**（调研 §1.2）：opacity 0.4 + grayscale + emoji 占位符 `❓`/`???` 三重弱化。
- **方案策略**（§3.5）：去 opacity，改 `grayscale(1) brightness(.5)`，但**仍保留 emoji 占位符 ❓/??? + 角标 ?**。
- **评估**：grayscale+brightness 让卡片"轮廓可见"，确实比 opacity 强（解决"融入背景"）。但未收集宠物仍显示 `❓` 而非真实剪影（因为不知道是哪只），**"看不见具体宠物"的根因未完全解决**（这是图鉴的防剧透设计，属产品决策非 bug）。方案选择合理——若要"看见剪影"需改变未收集不剧透的设计前提。
- **结论**：grayscale 统一是正确方向，与探索场景（commit 1c100b4）一致，解决"半透明不可见"。防剧透占位符是合理保留。

### R5【低】配色不一致遗留
- 调研 §1.3/问题3：legendary 配置色 `#f39c12`（pets.json rarity_config:28）vs CSS 边框 `#f1c40f`（css:146）。方案 §3.3 采用 WoW `#FF8000` 橙金（第三套色）。
- **风险**：若 P0 改 CSS 为 #FF8000 但不联动改 pets.json rarity_config，配置色与渲染色又添第三种不一致。**建议 P0 一并统一三处配色**（rarity_config + CSS + 方案 §3.3 表）。

### R6【低】边框图不阻塞可信度
- Agnes KEY 有效性（方案复用 `.tmp/agnes-*.py` 模式）+ PNG 透明区生成质量均有不确定性。但 D2 CSS 降级策略已对冲——**P0 不依赖生图即可交付图鉴可见性修复**，"不阻塞"声明可信。
