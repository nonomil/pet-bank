## 1. banchong 宠物互动机制总结

### 1.1 页面结构（三栏布局）

| 区域 | 内容 |
|------|------|
| 左栏 | 「去谁家串门？」学生列表，每行 = 学生名 + 宠物名 + Lv，点击切换查看不同学生的宠物（豆豆-云朵萨摩耶Lv.3 / 豆豆龙-数码火吉Lv.3 / 测试学生-焦糖小熊猫Lv.3） |
| 中央 | 当前选中宠物大图（600px webp）+ 「云朵萨摩耶 Lv.3 / 95%」（Lv + 进化进度百分比）+ 「19/19」（疑似饱食或互动次数） |
| 右栏 | 小屋引导（"还空着呢"）/ 小屋操作（管理小屋、管理玩具、自定义价格）/ 徽章墙 |

宠物图命名约定：`assets/images/pet/{slug}-{stage}.webp`，如 `cloud-samoyed-2.webp`，`stage` 数字即进化阶段。

### 1.2 核心互动机制（实测）

**结论：banchong 的"互动"重头戏是装饰经济系统，不是点宠出气泡。**

#### A. 装饰经济系统（核心，banchong 强项）
- **管理小屋**：买家具装饰宠物小屋
- **管理玩具**：买玩具给宠物
- **教师自定义价格**：教师可为小屋和玩具分别自定义价格（教育场景的"定价权"）
- 这是 banchong 与"班级积分/教师管理"场景绑定的核心循环：积分 → 购买装饰 → 个性化小屋 → 班级展示

#### B. 进化可视化
- 大图下方常驻显示 `Lv.X / 进度%`，例如 `Lv.3 / 95%` 表示接近下一次进化
- 进化阶段通过 `{slug}-{stage}.webp` 切换立绘
- 进度百分比给玩家"再差一点就进化"的预期驱动

#### C. 社交串门（班级场景）
- 左栏多学生列表，点击即切换到别人的宠物小屋
- "去谁家串门？"暗示有拜访/查看他人小屋装饰的社交玩法
- 强依赖多用户数据共享（班级/教师视角统一）

#### D. 徽章墙
- 右栏「徽章墙：全部 1 →」入口，成就/里程碑展示
- 配合进化与装饰，作为长线收集目标

#### E. 微互动（很轻量）
- 点击宠物：`active:scale-95`（按下缩放）
- hover 宠物：`group-hover:opacity-100` 蒙层（bg-black/60 半透明蒙层，疑似显示"查看"提示）
- **明确：无点击气泡文字**。DOM 中无 `tip` / `bubble` / `speech` 元素，hover 宠物时 newTips 为空

### 1.3 banchong 强项 vs 弱项

| 维度 | banchong | 评价 |
|------|----------|------|
| 装饰经济（家具/玩具/定价） | ✅ 完整闭环 | 强，是核心差异化 |
| 进化可视化（Lv/%） | ✅ 直观 | 强，预期驱动 |
| 社交串门 | ✅ 班级场景 | 强，但强绑教育多用户 |
| 徽章墙 | ✅ 有入口 | 中，长线目标 |
| 点击气泡文案 | ❌ 无 | 弱，互动反馈单薄 |
| 状态系统（HP/饱食/快乐等） | ❌ 未见 | 弱，缺养成深度 |
| 倒下/救援闭环 | ❌ 无 | 弱，无紧迫感机制 |

| banchong 机制 | 本项目现状 | 状态 | 对应模块/文件 |
|---------------|-----------|------|---------------|
| 装饰经济（家具） | 基础家具摆放已实现，仅 2 件默认家具（食盆/浴缸） | 🟡 半成品 | `home.js` `placeFurniture/addFurniture/FURN_META`（home.js:310-321）、`shop.js`（无家具分类） |

| 徽章墙 | 有 `card-collection.js`，但无独立徽章/成就系统 | 🟡 可复用 | `js/card-collection.js` |

| 状态系统（5 维） | HP/饱食/快乐/亲密/经验 5 维全有 | ✅ 已有 | `home.js` renderUI vitHtml（home.js:344-369） |
| 倒下/救援闭环 | 已实现（灰度立绘 + 救援 CTA + 复活动画） | ✅ 已有 | `home.js` `onRescue`（home.js:220-242） |
| 立绘姿态滤镜 | 已实现（hungry/dirty/weak/down 4 态滤镜） | ✅ 已有 | `home.js` `_poseClass`（home.js:149-155） |
| 微互动（点击/hover） | 按钮有 `:active`，但宠物立绘本体无点击/ hover 反馈 | ❌ 缺 | `home.js` `.home-pet-img` |
| 离线衰减补算 | 已实现 | ✅ 已有 | `PetSystem.decay()`（home.js:283-285） |

### 2.2 关键差异结论

- **本项目强项**：状态气泡（4 态文案）、5 维状态系统、倒下-救援闭环、立绘姿态滤镜、离线衰减——养成深度和情感反馈远超 banchong。
- **banchong 强项**：装饰经济（家具/玩具/定价闭环）——这是本项目 P2-0 缺口的直接借鉴对象。
- **场景差异**：本项目是**单孩子家庭积分银行**，banchong 是**多学生班级管理**——串门/教师定价等班级机制不直接适用。

## 3. 合入方案（按优先级）

### P1-A：宠物立绘微互动增强（点击缩放/抖动 + hover 高亮）

**实现思路**：
- 给 `.home-pet-img` 增加点击事件，点击时短暂播放 `bounce` / `wiggle` 动画 + 随机气泡文案（"汪！" / "喵～" / "🥰"）
- hover 时叠加高亮光晕（`filter: brightness(1.1) drop-shadow(0 0 12px rgba(255,255,255,.6))`）+ `cursor: pointer`
- 纯 CSS keyframes + 少量 JS class 切换，零数据改动

**涉及文件**：
- `js/home.js`：`_injectStyles` 加 keyframes（wiggle/bounce）、renderUI 给 `.home-pet-img` 加 `onclick="HomeSystem._petTap()"`、新增 `_petTap()` 切换 class
- 工作量：**0.5 人日**

**对应差距/AC**：补强小屋体验细节，非强制 AC，属 P1 体验优化。

---

### P1-B：进化进度可视化（小屋主视觉显示 Lv + 进化进度条）

**实现思路**：
- 本项目 EXP 条已在右栏（`home.js:364-367`），但缺 banchong 那种"大图下方常驻 Lv + 进化%"的主视觉。
- 在 `.home-stage` 顶部（气泡上方或下方）加一条「Lv.X · 进化 95%」胶囊条，复用已有 `_expPct(s)`（home.js:140）和 `s.level`。
- 进度 ≥ 80% 时高亮闪烁（`animation: pulse`），强化"快进化"预期。

**涉及文件**：
- `js/home.js`：`_injectStyles` 加 `.home-evolve-badge` 样式、renderUI 在 `home-stage` 内 `petImgHtml` 上方插入进度条 HTML
- 工作量：**0.5 人日**

**对应差距/AC**：补强进化感知，非强制 AC，属 P1 体验优化。呼应 [../路线/差距清单与开发路线图.md](../路线/差距清单与开发路线图.md) P0 家园 MVP 验收点 3（进化可见）。

---

### P2-A：装饰经济系统（家具）—— 借鉴 banchong 核心

> 这是 banchong 最值得借鉴的一块，本项目已预留大量接口，落地成本低。

**实现思路**（与路线图 P2-0 完全对齐）：
1. 新增 `data/furniture.json` 作为 `shop.js` / `home.js` 共享家具目录（至少 6 件非默认家具）
2. `shop.js` 新增「家园装饰」分类，复用现有购买流程（扣分 + 写入 ownership）
3. 购买后调 `HomeSystem.addFurniture(furnId)`（home.js:264 已实现）
4. `home.js` 将 `FURN_META`（home.js:310）从硬编码改为读 `data/furniture.json`
5. 小屋新增「已拥有未摆放家具栏」，点击槽位摆放/替换（`placeFurniture` home.js:245 已实现）
6. 同一家具不可重复购买；同槽位替换时旧家具回未摆放栏（需在 `placeFurniture` 加回退逻辑）

**涉及文件**：
- 新增：`data/furniture.json`
- `js/shop.js`：新增家具分类渲染 + 购买逻辑
- `js/home.js`：`FURN_META` 动态化、`placeFurniture` 加替换回退、renderUI 加未摆放栏
- 工作量：**2-3 人日**

**对应差距/AC**：直接对应 **AC-19 / AC-20**（商店家具联动），[../路线/差距清单与开发路线图.md](../路线/差距清单与开发路线图.md) P2-0，[../规格/需求规格书.md](../规格/需求规格书.md) §2.12 / §2.14 "商店家具联动未打通"。

### P2-B：玩具系统（家具的延伸）

**实现思路**：
- 玩具 = 可互动的家具（逗猫棒、毛线球、骨头等），摆放后点击触发"宠物玩耍"短动画（复用 `_petTap` + 跳跃 keyframes）+ 提升 happiness
- 数据上与家具共享 `furniture.json`，但增加 `type: 'toy'` 标记和 `onPlay` 效果字段
- 复用 `PetSystem.play()`（home.js:196 已调用）作为效果入口

**涉及文件**：
- `data/furniture.json`：加 toy 类目
- `js/home.js`：`.home-furn-slot` 点击区分家具/玩具，玩具触发 `onPlay` 动画
- 工作量：**1-2 人日**（依赖 P2-A）

**对应差距/AC**：扩展 AC-19/20，无独立 AC，属 P2 体验深化。

---

### P2-C：徽章墙（成就展示）

**实现思路**：
- 本项目已有 `js/card-collection.js`（图鉴），可新建 `js/badge.js` 或在 card-collection 内加「徽章」分区
- 徽章来源：首次进化、连续喂食 7 天、首次救援、家具收集 3/6/全、探索 N 次等
- 小屋右栏加「徽章墙 →」入口，跳转徽章页

**涉及文件**：
- 新增：`js/badge.js`（或扩展 card-collection.js）、`data/badges.json`
- `js/home.js`：右栏加徽章墙入口卡片
- 工作量：**2 人日**

**对应差距/AC**：补强长线目标，无独立 AC，属 P2 收集玩法。

---

### P3：社交串门（多宠物切换，非班级串门）

**实现思路**：
- **不建议照搬 banchong 班级串门**（需后端多用户数据共享，本项目是单孩子家庭）
- 可简化为**多宠物切换**：本项目支持多只宠物（`pets.json` 147 种），小屋加「我的宠物」切换栏，点击切换当前展示的宠物小屋
- 数据层：localStorage 存多宠物 state 数组，复用现有 `PetSystem.getState` 改为 `getState(petId)`

**涉及文件**：
- `js/pet.js`：多宠物 state 管理
- `js/home.js`：左栏加宠物切换列表
- 工作量：**3-5 人日**（数据模型改动较大）

**对应差距/AC**：无对应 AC，属 P3 长期扩展，需评估是否与 [../规格/需求规格书.md](../规格/需求规格书.md) 单宠物主线冲突。

## 4. 推荐优先级（性价比排序）

> 排序原则：**用户价值 × 实现成本**，价值高且成本低的先做。

| 顺序 | 项目 | 价值 | 成本 | 性价比 | 理由 |
|------|------|------|------|--------|------|
| 1 | **P1-A 微互动增强** | 高 | 极低（0.5 人日） | ⭐⭐⭐⭐⭐ | 纯 CSS + 少量 JS，立即提升宠物"活"的感觉，零数据风险 |
| 2 | **P1-B 进化进度可视化** | 高 | 低（0.5 人日） | ⭐⭐⭐⭐⭐ | 复用已有 `_expPct`，强化进化预期，数据已就绪 |
| 3 | **P2-A 装饰经济（家具）** | 极高 | 中（2-3 人日） | ⭐⭐⭐⭐ | banchong 核心玩法，本项目接口已预留 70%，闭合消费循环，直接对应 AC-19/20 |
| 4 | **P2-B 玩具系统** | 中 | 低（1-2 人日，依赖 P2-A） | ⭐⭐⭐ | 家具的自然延伸，复用 play 逻辑 |
| 5 | **P2-C 徽章墙** | 中 | 中（2 人日） | ⭐⭐⭐ | 长线收集目标，但短期可见性低于装饰 |
| 6 | **P3 多宠物切换** | 中 | 高（3-5 人日） | ⭐⭐ | 场景差异大，数据模型改动深，建议延后 |

**推荐节奏**：
1. **第一周**：P1-A + P1-B（共 1 人日），立即上线，体验跃升
2. **第二、三周**：P2-A 装饰经济（2-3 人日），闭合 AC-19/20，补齐产品核心循环
3. **后续**：P2-B 玩具 → P2-C 徽章墙 → 视产品方向决定 P3

## 5. 不建议照搬的（差异点）

### 5.1 班级社交串门 → 不直接适用
- banchong 是**多学生班级**场景，串门依赖班级内多用户数据共享（教师视角统一管理）
- 本项目是**单孩子家庭**积分银行，无班级概念
- **替代方案**：若要做社交感，简化为「多宠物切换」（见 P3），而非跨用户串门

### 5.2 教师自定义价格 → 简化为家长配置或不做
- banchong 的"教师自定义小屋/玩具价格"是教育场景的定价权玩法（教师可调控积分经济）
- 本项目是家庭场景，家长即最高权限，自定义价格的产品价值低
- **建议**：不做，或简化为「家具统一标价 + 少量稀有家具高价位」即可

### 5.3 无气泡设计 → 不借鉴
- banchong 无点击气泡，互动反馈单薄
- 本项目 `_bubble` 状态气泡（home.js:158-164）反而是强项，4 态文案 + 姿势滤镜的组合反馈比 banchong 更丰富
- **结论**：保持本项目气泡设计，不向 banchong 靠拢

---

## 6. 总结一句话

> banchong 的**装饰经济系统**（家具/玩具/定价闭环）值得本项目 P2-A 直接借鉴，对应已落地的 `HomeSystem.addFurniture` 接口和 AC-19/20；而本项目在**状态气泡、5 维养成、倒下救援**上反而优于 banchong，应保持差异化优势，先做 P1 微互动 + 进度可视化提升即时体验，再做 P2 装饰经济闭合核心循环。
