## 一、案例概要

《我要上清北》是一个儿童闯关益智类 H5 小游戏，由一位教育行业从业者用 Codex 辅助开发。核心理念：**游戏化学习不是给题目加特效，而是重新设计孩子的思考动作**。

参考案例与本项目的相似度极高——两者都是：
- Vanilla H5 儿童学习游戏
- 用 AI（Codex/Claude）辅助迭代开发
- 覆盖数学/语文/英语/科学多学科
- 有积分/商城/装扮/成长报告系统
- 面向儿童+家长双角色

## 二、设计原则逐条对照

### 原则 1: 第一屏不是催学习，是让孩子愿意开始

**案例做法**:
> "刻意避开了传统学习工具的样子。很多学习产品打开以后第一眼就是课程、题库、成绩、排行榜。我希望第一屏先像一个可以进入的世界。"
> 主城页面有角色、积分、今日推荐、四个学科入口。不是"去做题"，而是"你今天可以先破一个数学副本"。

**本项目现状**: ✅ 基本做到

| 信号 | 实现 |
|------|------|
| 首页是探索地图而非课程列表 | 首页展示 12 场景螺旋地图（MAP_LAYOUT），每个节点是一个可进入的冒险场景 |
| 有角色和积分展示 | 顶部导航有宠物展示、积分胶囊（topPoints） |
| 不是任务清单式开场 | 默认页面是 `map`（探索地图），任务在侧栏而非首屏独占 |

**差距**: 首页目前是地图+侧栏任务混合，仍有"任务清单感"。建议后续版本优先展示地图/宠物/今日推荐，任务卡片下沉到二级。

---

### 原则 2: 能力测试不是考试，是起步难度校准

**案例做法**:
> "能力测试在这个项目里的定位不是考试，而是'起步难度校准'。页面里会有四科小测，但文案尽量写成诊断舱、入门预估、推荐起步难度。好的教育产品应该把评价从'审判'变成'导航'。"

**本项目现状**: ⚠️ 一期已改善，完整诊断仍不足

| 信号 | 实现 |
|------|------|
| 难度分级 | 数学PK有 5 档难度（easy20/easy100/medium_mul/medium_mix/hard），按能力选择 |
| 场景等级解锁 | 探索场景按 min_level(1-5) 螺旋推进，从简单到复杂 |
| 推荐理由 | ✅ 数学PK难度配置已补 `fitFor/reason`，大厅和设置页都展示"适合谁/为什么" |
| 诊断性文案 | ⚠️ 仍缺"诊断舱""入门预估"等完整导航型流程 |

**差距**: 一期已补静态推荐理由，但还不是基于历史表现的起步难度校准。孩子能知道每档"适合谁"，系统还不会主动说"今天建议从哪一档开始"。

---

### 原则 3: 难度=更适合，不=更难

**案例做法**:
> "难度不应该只代表'更难'，还应该代表'更适合'。如果今天状态一般，可以从入门开始；如果已经稳定，再去挑战更高难度。这和传统刷题很不一样。"

**本项目现状**: ⚠️ 有基础，缺引导

| 信号 | 实现 |
|------|------|
| 多档难度 | 数学PK 5 档难度，探索 5 级场景 |
| "今天状态一般"的灵活性 | ✅ 孩子随时可切难度（math-pk.js:1135），不受锁限制 |
| 乘法启程节奏放缓 | ✅ `f9b13773` commit 专门为乘法初学做了节奏优化 |
| 难度推荐文案 | ❌ 无"今天建议选入门"或"上次在这里卡住了，建议先练习" |

**差距**: 难度切换是自由的（好），但没有基于历史数据的个性化推荐。

---

### 原则 4: 战斗页——真正的游戏化是重写思考动作

**案例做法**:
> "如果只是把选择题换个皮肤，那其实还是刷题。以数学副本为例：系统不会只问'15+8等于多少'。它会把几只小怪放在场上，每只小怪头顶有一道题，然后给出一个目标，比如'击中答案小于30的小怪''击中偶数答案的小怪'。孩子要先观察，再计算，再判断规则，最后才点击攻击。"

**本项目现状**: ✅ 核心亮点，是做得最好的部分

| 信号 | 实现 |
|------|------|
| 不是简单刷题 | 数学PK要求：先读题→计算→判断规则→选择目标→攻击 |
| 分类/比较/排除/决策 | ✅ 应用题（CMATH）模式需要"先理解文字再计算" |
| 多学科覆盖 | ✅ 数学PK（计算+应用）/ 汉字游戏（选字+填空）/ 英语词汇（看图识词） |
| 探索战斗 | ✅ 探索场景的战斗融入叙事上下文，不是孤立的题目弹窗 |

**本项目在这个维度上甚至超过了参考案例**——参考案例只描述了想法，本项目已经实现了数学PK+探索战斗+汉字游戏+英语词汇四种游戏化学习形态。

---

### 原则 5: 失败结算保护孩子继续尝试的勇气

**案例做法**:
> "失败也要说清楚原因：是时间耗尽，还是没有击破BOSS，还是命中次数不够。接着给出下一次策略。失败不是终点，失败应该变成复盘。"

**本项目现状**: ✅ 反馈闭环一期已落地，报告层仍可深化

| 信号 | 实现 |
|------|------|
| 探索战斗失败 | ✅ 只扣 10% HP，不死亡，有复活动作；失败结算已展示"复盘/下一步" |
| 数学PK失败 | ✅ 显示得分和最高分，可立即再来一局，并在结算页显示"复盘/下一局" |
| 探索数学小题 | ✅ 答错时展示"下一步"提示，优先使用题目 hint，其次使用 explanation |
| 失败原因说明 | ✅ 探索战斗失败会区分撤退失败/敌人反击，并给出休息、救援、防御、回血道具等下一步 |
| 复盘引导 | ✅ 数学PK、探索小题、探索战斗已补轻量复盘；家长报告层的错因归纳尚未实现 |

**差距**: 一期已把数学PK、探索小题和探索战斗从"可再试"推进到"可复盘"。下一步应把这些事件沉淀到错题/战斗原因聚类和成长报告建议中。

---

### 原则 6: 奖励是脚手架，不是拐杖

**案例做法**:
> "积分可以换外观、道具和补给，真实奖品需要家长确认；战斗道具可以提供提示或保护，但不能直接替孩子答题；时装主要满足表达欲，不让数值碾压学习能力。奖励系统最好的位置是'脚手架'，不是'拐杖'。"

**本项目现状**: ✅ 基本做到

| 信号 | 实现 |
|------|------|
| 积分换外观/道具 | ✅ shop.js: ITEMS(8种生活奖励)+BATTLE_ITEMS(5种)+盲盒 |
| 道具不替答题 | ✅ 战斗道具（回血/加攻/炸弹）辅助战斗，不跳过题目 |
| 装扮不影响成绩 | ✅ 宠物 species/装备 影响战斗属性（合理），但外观（imageStages/emoji）不影响 |
| 家长确认 | ❌ 无家长确认环节——孩子直接用积分购买 |

**差距**: 缺少"家长确认"环节。案例中的"真实奖品需要家长确认"在本项目中未实现。积分兑换的自由度完全在孩子端。

---

### 原则 7: 装扮系统——让孩子拥有自己的学习身份

**案例做法**:
> "让孩子选择角色、衣服、鞋子、配饰，不只是为了好看，而是让他觉得'这是我的角色，这是我的成长'。装扮不能喧宾夺主。衣橱和商城是独立路径，不强行打断学习。"

**本项目现状**: ✅ 做到且更丰富

| 信号 | 实现 |
|------|------|
| 角色选择 | ✅ 261 种宠物可选（PetSystem.chooseSpecies） |
| 外观变化 | ✅ 5 阶段进化（蛋→幼崽→成长期→完全体→终极体）+ 宠物卡片 |
| 装扮路径独立 | ✅ 宠物小屋（home.js）是独立页面，不打断学习流程 |
| 小屋家具装扮 | ✅ 5 槽位家具系统 + 8 种背景主题 |

---

### 原则 8: 成长报告是导航，不是审判

**案例做法**:
> "成长报告是给家长看的，但我不希望它变成另一个成绩单。这个页面更想表达的是：今天练了多久，完成了几局，错题集中在哪些知识点，下一轮建议先练什么。"

**本项目现状**: ⚠️ 基础存在，深度不足

| 信号 | 实现 |
|------|------|
| 家长可见的数据 | ✅ FamilyReview（family-review.js）展示孩子的学习进度 |
| 成长贴纸报告 | ✅ app.js 的 `renderGrowthStickerReport()`：热力图+6维度进度条 |
| 学习中心进度 | ✅ learn-center.js 的 pack/module/lesson 三级进度追踪 |
| 错题分析 | ❌ 无错题本——没有"错题集中在哪些知识点"的分析 |
| 下一步建议 | ❌ 无智能推荐——不会说"下一轮建议先练什么" |

**差距**: 有"数据展示"但无"数据解读"。家长能看到完成数和进度条，但得不到 actionable 的建议。

---

### 原则 9: Codex 推进方式

**案例做法**:
> "先让它帮我整理产品文档，再一起拆核心循环；先用 H5 快速做出可玩的原型，再从页面截图里做交互复审；发现首页像工具，就改成主城；发现战斗像点题卡，就强化小怪、BOSS 护盾、连击、技能反馈。"

**本项目现状**: ✅ 完全一致

| 信号 | 实现 |
|------|------|
| 文档→原型→复审→迭代 | docs/ 下有 271 个 markdown 文件，prj/ 下有大量原型和测试 |
| 首页从工具感→世界感 | 首页从简单任务列表→探索地图+宠物小屋+游乐场 |
| 战斗从点题卡→策略决策 | card-arena.js 从 3v3→2v2，技能 CD/防御/换人机制逐步深化 |
| 素材迭代 | robot-rivals 从 v2→v5 迭代 4 版 |
