# 家庭与社交能力

> 当前状态：旧 Supabase/社交运行时已移除。当前保留本机多孩子和离线玩法，同时已接入 VPS 自托管 SQLite 的家长账号、家庭、邀请码、孩子映射和 Profile 快照生命周期；好友、串门、PK、动态流仍未完成。
>
> 方案与实施顺序：[家庭账号社交体系](../../docs/方案/专题/家庭社交/README.md)

## 当前可用能力

- `js/profiles.js`：在本机 `localStorage` 中创建、切换和隔离多个孩子档案。
- `js/walk.js`：单机宠物遛弯路线和互动。
- `js/home.js`：单机宠物小屋、家具和背景主题。
- `js/leaderboard.js`：按本地 profile 隔离的个人成绩记录。
- `js/family-review.js`：只读取 `petbank_guided_feedback_history`，展示本机复盘空态和玩法卡点建议。
- `js/parent-account.js` + `js/self-hosted-api.js`：家长设置页的注册/登录、家庭管理、邀请码和云端孩子映射。

## 当前不可用能力

以下能力不能在前端伪装成已接入：

- 多端自动合并和完整的细粒度同步状态中心；当前基础上传、启动恢复、outbox 和 revision 冲突拒绝已经接入。
- 好友码、串门、动态流和跨家庭 PK。
- 本机复盘不等于家庭同步；它不会读取账号、好友或云端数据。

未完成能力的后端入口统一在 `/api/v1/` 下，由 `prj/petbank-server/` 的 Node.js + SQLite API 承载。前端必须区分“账号/家庭/孩子 API 操作成功”和“孩子本机状态已同步”。

## 产品边界

家长端与孩子端要分成两条路径：

```text
/app/*
  -> 孩子端：本地学习、宠物、小屋和游戏

/parent/*
  -> 家长端：账号、家庭、孩子、同步和后续社交
```

家长端不应把孩子端的玩法控制和本地存储细节暴露为“账号系统”。账号、家庭、孩子映射和基础快照生命周期已完成；下一阶段先做生产验证，再逐项放开社交能力。

## 后续接入顺序

1. 多端同步治理：冲突处理选择、恢复验证和生产 canary。
2. 好友和串门：访问权限、好友码、互动记录。
3. PK 和动态流：独立 migration、权限测试和回滚方案。

在每一步完成前，保留本地离线能力作为明确降级路径，不将未实现的云端动作写入成功状态。
