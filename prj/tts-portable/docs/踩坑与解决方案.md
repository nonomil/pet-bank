# 踩坑与解决方案（核心文档）

> 这是整个 tts-portable 项目**最大的价值**。这 10 个坑都是真实踩过、用时间换来的教训。
>
> 每个坑记录：**现象 / 根因 / 解决方案 / 代码或配置位置 / 如何避免**。遇到 TTS 相关报错，先查这份文档。

## 1. 端口 8765 冲突

**现象**
启动 `tts_server.py` 报 `OSError: [Errno 10048] address already in use`，或服务起不来；即使起来，前端 `/health` 也连不上。

**根因**
8765 是 LLM 代理（如某些 OpenAI API 代理、LM Studio 等）常用端口，极易被占用。开发机上常驻多个本地服务时冲突概率很高。

**解决方案**
- **默认端口已改为 9885**（见 `config.py`）。
- 仍冲突时用环境变量改端口：
  ```bash
  set TTS_PORT=9886   # Windows CMD
  export TTS_PORT=9886 # Git Bash / Linux / Mac
  python tts_server.py
  ```
- 启动脚本 `start.bat` / `start.sh` **已内置端口占用预检测**（检测到占用直接退出并提示改 `TTS_PORT`），其原理片段：
  ```bat
  :: start.bat 片段
  netstat -ano | findstr ":9885 " >nul && (
    echo [warn] 端口 9885 被占用，改用 TTS_PORT 环境变量
    set TTS_PORT=9886
  )
  python tts_server.py
  ```

**代码位置**
`config.py` 第 22 行：`PORT = int(os.environ.get("TTS_PORT", "9885"))`。

**如何避免**
永远不要硬编码端口；启动脚本检测占用；文档里所有示例统一用 9885 而非 8765。

## 2. VoxCPM2 模型下载卡死

**现象**
首次跑 VoxCPM2，`VoxCPM.from_pretrained("openbmb/VoxCPM2")` 卡住不动；进度条跑到 **53KB 后停滞 15 分钟以上**，最终超时或内存膨胀。国内网络尤其严重。

**根因**
HuggingFace 对**匿名下载**限速很狠，国内到 `huggingface.co` 的连接常在握手后被掐断。53KB 是 TLS 握手后的第一个 chunk 临界点，之后连接挂起但不报错，导致卡死而非快失败。

**解决方案**（按优先级）

**① 复用已下好的本地模型（首选）**
如果机器上或同事处已有 VoxCPM2 权重（如儿童绘本项目下过的），直接指向它：
```bash
set VOXCPM2_MODEL_PATH=D:/HuggingFaceCache/VoxCPM2
```
该目录是扁平结构（`model.safetensors` + `audiovae.pth` + `config.json` + `tokenizer.json` 等），`VoxCPM.from_pretrained` 直接吃目录路径。**免下载、免联网、秒级加载**。

**② 设 HuggingFace 镜像**
没有本地模型必须联网下时，先切镜像：
```bash
set HF_ENDPOINT=https://hf-mirror.com   # Windows CMD
export HF_ENDPOINT=https://hf-mirror.com # Git Bash
```
子进程会继承父进程环境变量（`tts_server.py` 用 `subprocess.run` 调 VoxCPM2 环境），故在**启动 tts_server 的终端**设一次即可。

**③ 设 HF_TOKEN 解除匿名限速**
镜像仍慢时，去 `https://huggingface.co/settings/tokens` 申请免费 token：
```bash
set HF_TOKEN=hf_xxxxxxxxxxxxxxxxxxxx
```
带 token 的下载走认证通道，限速宽松得多。

**代码位置**
- `config.py` 第 18 行：`VOXCPM2_MODEL_PATH = os.environ.get("VOXCPM2_MODEL_PATH", "D:/HuggingFaceCache/VoxCPM2")`。
- `tts_server.py` 第 154 行：`model_path = VOXCPM2_MODEL_PATH or "openbmb/VoxCPM2"`（本地优先，留空回退在线名）。

**如何避免**
新机器第一次跑 VoxCPM2 前，**先确认是否有本地模型可复用**；没有就先设 `HF_ENDPOINT` + `HF_TOKEN` 再启动。永远不要裸连 `huggingface.co` 下大模型。

## 3. subprocess timeout=120 太短

**现象**
首次 `/tts` 请求（engine=auto 或 voxcpm2）长时间无响应，最终后端日志报 `subprocess.TimeoutExpired`，前端 15s 后超时降级。后续请求也一直失败，看起来「VoxCPM2 完全不能用」。

**根因**
VoxCPM2 **冷启动**成本极高：
1. 子进程 `import torch`（CUDA 初始化）—— 几十秒
2. `VoxCPM.from_pretrained` 加载 4.7GB 权重到显存 —— 30~60s
3. 扩散 `generate()`（10 个 inference timesteps）—— 数秒到十几秒

三者叠加常 **>120s**。原 `services/tts_service.py` 写死 `timeout=120`，冷启动请求被砍，且因为模型没加载完，下一次请求又是冷启动，**死循环失败**。

**解决方案**
`config.py` 默认 `VOXCPM2_TIMEOUT=300`，`tts_server.py` 用它：
```bash
# 默认就够，冷启动慢的机器可加大
set VOXCPM2_TIMEOUT=600
```

**代码位置**
- `config.py` 第 36 行：`VOXCPM2_TIMEOUT = int(os.environ.get("VOXCPM2_TIMEOUT", "300"))`。
- `tts_server.py` 第 182 行：`timeout=VOXCPM2_TIMEOUT`（不再是写死的 120）。
- **对比**：`services/tts_service.py` 第 101 行仍是老的 `timeout=120`（那是批量离线生成工具，不是实时服务路径，**实时服务走 tts_server.py 不走这里**）。

**如何避免**
任何「冷启动重 + 子进程」的组合，超时按 `冷启动上限 + 生成上限 + 安全余量` 设，宁可大不可小。冷启动只痛一次（模型加载后稳态在显存里复用），别用首次延迟倒推超时。

## 4. voice_description 参数不存在

**现象**
照着 VoxCPM2 官方文档/HuggingFace 在线 Demo 写：
```python
wav = model.generate(text="...", voice_description="温柔女声", cfg_value=2.0)
```
本地 `voxcpm` 包报 `TypeError: generate() got an unexpected keyword argument 'voice_description'`。

**根因**
`voice_description` 是 **HuggingFace 在线版 API（Spaces / Inference Endpoint）** 的参数。本地 `pip install voxcpm` 装的包，`generate()` 签名**没有** `voice_description`。官方项目文档混用了在线版和本地版接口，导致误导。

**解决方案：text-based 声线设计**
本地包用「声线描述拼到文本前」实现声线控制。描述用括号包住作为指令前缀，紧接正文：
```python
from voxcpm import VoxCPM
import soundfile as sf

model = VoxCPM.from_pretrained("D:/HuggingFaceCache/VoxCPM2", load_denoiser=False)
voice_desc = "(年轻女性，声音温柔甜美，讲故事的语调)"
text = "从前有一只小兔子住在森林里。"
wav = model.generate(
    text=voice_desc + text,   # ← 声线指令拼到正文前
    cfg_value=2.0,
    inference_timesteps=10,
)
sf.write("out.wav", wav, model.tts_model.sample_rate)
```

**代码位置**
- `config.py` 第 42-63 行 `VOICE_PRESETS`：每个音色的 `voxcpm` 字段就是这种括号描述（如 `"(年轻女性，声音温柔甜美，讲故事的语调)"`）。
- `tts_server.py` 第 164-166 行：
  ```python
  full_text = {voice_desc!r} + {text!r}   # 拼接，不是传参
  wav = model.generate(text=full_text, cfg_value=2.0, inference_timesteps=10)
  ```
- `services/tts_service.py` 第 78-83 行：**老版本错误地传了 `voice_description=` 参数**，这是历史遗留的 bug 示例，实时服务不走它。

**如何避免**
看本地包的实际签名，别照搬在线 Demo。本项目的四套 `voxcpm` 描述已验证可用，改声线只改描述文本即可。注意：声线描述每次生成可能有细微差异，重要场景生成 1-3 次选最佳。

## 5. subprocess stderr 被 GBK 字节吞掉

**现象**
VoxCPM2 子进程失败时，`tts_server.py` 报 `UnicodeDecodeError: 'gbk' codec can't decode byte ...`，**错误信息本身丢失**——既看不到 VoxCPM2 的真实报错，又被这个解码异常二次打断。Debug 极其困难。

**根因**
- VoxCPM2 子进程的 stderr 含 **CP936/GBK** 字节（Windows 中文系统默认编码，tqdm 进度条、PyTorch 警告常带）。
- `subprocess.run(..., encoding="utf-8")` 指定了编码但**没指定 errors**，遇到非 UTF-8 字节直接抛 `UnicodeDecodeError`，把原始 stderr 吞掉。

**解决方案**
加 `errors="replace"`，遇到无法解码的字节用替换符替代，**保证 stderr 不丢失**：
```python
result = subprocess.run(
    [str(VOXCPM2_PYTHON), tmp_path],
    capture_output=True,
    text=True,
    timeout=VOXCPM2_TIMEOUT,
    encoding="utf-8",
    errors="replace",          # ← 关键：吞不掉错误信息
)
```

**代码位置**
- `tts_server.py` 第 182-189 行：已加 `errors="replace"`。
- **对比**：`services/tts_service.py` 第 97-103 行**没加** `errors`（老版本，会崩），实时服务不走它。

**如何避免**
Windows 中文环境下，任何 `subprocess` + `text=True` 都默认带 `errors="replace"`。诊断子进程问题时，`stderr` 比返回码更重要，绝不能让它被编码异常吞掉。

## 6. health 假阳性

**现象**
`/health` 返回 `{"voxcpm2": true}`，但实际 `/tts`（engine=voxcpm2 或 auto）每次都超时失败。前端以为 VoxCPM2 可用，结果干等 300s 才降级，体验极差。

**根因**
早期健康检查**只查 `VOXCPM2_PYTHON` 这个 python.exe 文件是否存在**：
```python
# 错误做法
vox_ready = VOXCPM2_PYTHON.exists()  # python 在就算 ready
```
但模型可能：① 还没下完（`.incomplete` 文件）；② 路径配错；③ 4.3GB 权重只下了一半。python 在 ≠ 模型可用，导致假阳性。

**解决方案**
`_voxcpm2_model_ready()` 真实检查**模型权重文件**：
```python
def _voxcpm2_model_ready() -> bool:
    if not VOXCPM2_PYTHON.exists():
        return False
    # 1. 本地模型路径：查 model.safetensors 且 >100MB 才算完整
    if VOXCPM2_MODEL_PATH:
        st = Path(VOXCPM2_MODEL_PATH) / "model.safetensors"
        if st.exists() and st.stat().st_size > 100 * 1024 * 1024:
            return True
    # 2. 回退：HF hub 缓存，排除 .incomplete 文件
    snaps = Path.home() / ".cache/huggingface/hub/models--openbmb--VoxCPM2/snapshots"
    ...  # 遍历找 .safetensors/.bin/.pt 且 >1MB 且非 .incomplete
    return False
```
并且 **`engine=auto` 的引擎选择也用这个函数**（不只是 health 展示用）：
```python
# tts_server.py 第 244 行
if _voxcpm2_model_ready():
    try: _generate_voxcpm2(...)
    except: await _generate_edge(...)  # 失败才降级
else:
    await _generate_edge(...)          # 模型没就绪直接走 edge，不干等
```

**代码位置**
- `tts_server.py` 第 88-124 行 `_voxcpm2_model_ready()`。
- `tts_server.py` 第 199-203 行 `/health` 用它。
- `tts_server.py` 第 244 行 auto 引擎选择用它。

**如何避免**
健康检查要查「真正决定能否工作的资源」（模型权重），不是「相关进程是否在」（python.exe）。缓存文件按大小 + 后缀双校验，排除 `.incomplete`。同一判活函数同时用于展示和决策，避免「展示说行、决策说不行」的撕裂。

## 7. asyncio 嵌套

**现象**
在 FastAPI 的 `async def tts()` 路由里调 edge-tts，照着 edge-tts 同步示例写：
```python
asyncio.run(_run())   # ❌ 在已有事件循环里再起一个
```
报 `RuntimeError: asyncio.run() cannot be called from a running event loop`。

**根因**
FastAPI 路由本身就在事件循环里跑。`asyncio.run()` 会**新建并接管一个事件循环**，与已运行的循环冲突。这是从「同步脚本」搬到「异步 Web 框架」时的典型坑。

**解决方案**
FastAPI 路由是 `async`，**直接 `await`** edge-tts 的协程，不要套 `asyncio.run`：
```python
async def _generate_edge(text, voice_key, out_path):
    comm = edge_tts.Communicate(text, voice=..., rate=..., pitch=...)
    await comm.save(str(out_path))   # ← 直接 await

@app.post("/tts")
async def tts(req):
    await _generate_edge(text, voice, mp3_path)   # ← 路由里也直接 await
```

**代码位置**
- `tts_server.py` 第 132-142 行 `_generate_edge()`：`await comm.save(...)`。
- `tts_server.py` 第 217 行 `async def tts()` + 第 240/250/252 行直接 `await`。
- **对比**：`services/tts_service.py` 第 125-130 行用了 `asyncio.run(_run())`——那是**同步批量脚本**场景（无外部事件循环），用 `asyncio.run` 正确；但**不能照搬到 FastAPI 路由**。

**如何避免**
异步框架（FastAPI / Starlette / Sanic）的路由里，永远 `await` 协程，不要 `asyncio.run`。`asyncio.run` 只用于「无事件循环的同步入口脚本」。

## 8. CORS

**现象**
前端（`file://` 或 `http://localhost:xxxx`）`fetch('http://127.0.0.1:9885/health')` 被浏览器拦，控制台报：
```
Access to fetch at 'http://127.0.0.1:9885/health' from origin 'null' has been blocked by CORS policy
```

**根因**
TTS 服务（9885）和前端（不同端口或 file 协议）是**不同源**，浏览器默认拦截跨域请求。后端没显式允许跨域时，`/health` / `/tts` 全部 401/被拦。

**解决方案**
`tts_server.py` 加 `CORSMiddleware`，全开放（本地服务无安全顾虑）：
```python
from fastapi.middleware.cors import CORSMiddleware
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],      # 本地服务，全开放
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**代码位置**
`tts_server.py` 第 47-52 行。

**如何避免**
任何「前端直连本地后端服务」的方案，后端第一步就配 CORS。生产环境若要收敛，把 `allow_origins` 改成具体域名列表。

## 9. wav/mp3 content-type

**现象**
前端 `<Audio>` 或 `fetch().then(blob)` 播放返回的音频，某些音色/某些浏览器报「格式不支持」或无声。下载文件看，缓存名是 `xxx.mp3`，但内容头是 `RIFF`（WAV）。

**根因**
- `soundfile.write(path, ...)` 按**文件后缀**决定输出格式：传 `.wav` 写 WAV，传 `.mp3` 写 MP3。
- 但 VoxCPM2 经 soundfile 写的是 **RIFF WAV**，edge-tts 写的是 **MP3**。
- 缓存路径统一用 `.mp3` 后缀（`_cache_path` 返回 `f"{hash}.mp3"`），导致「后缀 .mp3 / 实际 WAV 内容 / content-type 写死 audio/mpeg」**三者不一致**，部分播放器按 content-type 解析失败。

**解决方案**
按**文件头魔数**判断真实格式，返回正确 content-type：
```python
def _media_type_by_content(path):
    try:
        with open(path, "rb") as f:
            head = f.read(4)
        if head[:4] == b"RIFF":
            return "audio/wav"
    except Exception:
        pass
    return "audio/mpeg"
```
返回时用它：
```python
return FileResponse(mp3_path, media_type=_media_type_by_content(mp3_path))
```

**代码位置**
- `tts_server.py` 第 74-85 行 `_media_type_by_content()`。
- `tts_server.py` 第 232、257 行（缓存命中 + 新生成）都用它。

**如何避免**
音频/图片等二进制资源，**永远按内容判断类型**（魔数 / 文件头），不要信文件后缀，更不要写死 content-type。后缀只是给人看的，content-type 必须反映真实字节。

## 10. Web Speech 需用户手势

**现象**
页面加载时自动调 `speechSynthesis.speak(utter)`，**Chrome 完全没声音**，也没报错。同一代码在用户点击按钮后调用却正常。

**根因**
Chrome（及多数现代浏览器）的安全策略：**首次 `speechSynthesis` 调用必须由用户手势触发**（click / keydown / touch）。页面加载时的自动播报被静默拦截，`utter.onerror` 可能不触发，极难排查。移动端 Safari 更严。

**解决方案**
1. **首次启用绑定到用户交互**：把「开启语音」做成按钮，或挂在已有按钮（如「开始游戏」「试听」）上。本项目设置面板的「🔊 试听」按钮就是为此设计的入口。
2. **设置面板提供手动开启**：用户进面板勾「启用语音」+ 点「试听」，这一次点击激活了 speechSynthesis，后续自动播报即可生效。
3. **代码兜底**：`_speakViaBrowser` 已有 5s 超时兜底（`setTimeout(finish, 5000)`），即使被拦截也不会卡住队列。

```javascript
// 试听按钮——用户手势激活 speechSynthesis
elPreview.addEventListener('click', function () {
    if (!settings.enabled) {
        settings.enabled = true; elEnabled.checked = true; save();
    }
    speak('你好呀，我是你的小伙伴！');   // ← 这一帧的 click 激活后续播报
});
```

**代码位置**
- `client/voice.js` 设置面板的试听按钮逻辑（`_injectSettingsUI`）。
- `client/voice.js` `_speakViaBrowser` 的 5s 超时兜底。

**如何避免**
永远不要假设 `speechSynthesis` 在页面加载时能用。引导用户做一次点击交互（哪怕是无关按钮）后再启用自动播报。文档和 UI 上明确提示「首次需点击开启」。
