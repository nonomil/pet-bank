# GPT UI 设计工作流调研笔记

## 原始思路

`和 GPT 聊一套提示词 -> 丢给 Figma / 生图工具出界面 -> 把界面图再丢回 GPT 生成设计规范 md -> 最后把图 + md 一起丢给 Codex 执行`

这个思路的本质是对的：不要让 coding agent 一边想需求、一边猜视觉、一边写代码，而是先把视觉目标和规则拆出来。

## 这套方法为什么有效

### 1. 它把“审美判断”前置了

对 0-1 产品来说，很多关键决策并不在 PRD 里，而在：

- 这个产品看起来像不像一个值得信任的产品
- 首页第一屏有没有质感
- 信息密度、留白、层次到底是什么感觉

这些东西如果不先变成图，Codex 只能自己猜。猜得再努力，也很难稳定命中你脑子里的 bar。

### 2. 它把“需求”和“视觉”分成了两套输入

给 Codex 的最佳输入不是一句总 prompt，而是两套约束：

- `spec / md`：功能逻辑、页面结构、状态、交互规则
- `UI 图 / Figma`：视觉目标、版式、风格、组件倾向

这样 AI 写代码时是在执行约束，而不是同时做产品经理、设计师、前端。

### 3. 它允许先出关键帧，再慢慢系统化

一开始不需要生成完整设计系统，先拿下几个最关键页面就够：

- 首页
- onboarding
- 列表页
- 详情页
- 核心操作弹层

关键帧先把气质定住，后面再补组件规则和设计规范，效率反而更高。

## 截至 2026-07-05 的调研结论

### 1. GPT 生图更适合“关键页面定调”

OpenAI 官方资料显示，`GPT Image 2` 已是最新一代高质量图像生成/编辑模型，支持文本和图像输入，适合高保真出图与定向编辑。

同时，OpenAI Academy 对图像生成的建议也很适合 UI 任务：

- prompt 不必很长，`1-3` 句清晰约束通常更有效
- 图片里如果有标题、标签、按钮文字，要明确写文案、位置和样式
- 碰到信息图、密集排版、重文本场景，最好后续进设计工具继续精修

结论就是：`生图` 适合先拿下视觉 bar，不适合单独承担完整产品设计系统。

### 2. Figma Make 让“图 -> 原型”更直接

Figma Make 现在可以：

- 直接上传图片作为 prompt 输入
- 附带现有 Figma 设计、组件一起生成
- 输出 functional prototype / web app / interactive UI
- 把预览再 copy 成 design layers 回到 Figma Design

这意味着，如果我们已经有关键 UI 图，完全可以跳过一部分“图片转 HTML”的步骤，先用 Figma Make 看交互感和页面组织，再沉淀回设计稿。

### 3. Figma MCP 让 Codex 更少“脑补”

Figma 官方 MCP server 已支持 `Codex`，而且可以让 agent 读取：

- components
- variables
- layout data
- styles

甚至还能把结果写回 Figma。相比只看截图，这种结构化上下文对代码实现更友好，因为它告诉 Codex 的不只是“长什么样”，还有“是怎么搭出来的”。

### 4. `html.to.design` 现在是很实用的桥接层

`html.to.design` 已支持两类典型场景：

1. 直接把 `ChatGPT`、`Figma Make`、`OpenAI Codex` 等 AI 预览从浏览器导入 Figma
2. 直接把 HTML 代码或 zip 导入成 Figma 图层

所以如果某一步已经变成“可运行网页”或“AI 预览页面”，不必重画，可以直接桥接进 Figma 做设计协作。

## 推荐的 3 条工作流

### 路线 A：视觉先行，适合高审美产品

`GPT 聊需求 -> GPT Image 2 出关键帧 -> GPT 生成设计规范 md -> Figma 清洗结构 -> Codex 实现`

适合：

- 品牌气质要求高
- 首页 / 营销页 / 内容产品
- 团队更依赖“先看到效果”

优点：

- 最容易先把感觉做对
- 最能避免 Codex 自行脑补视觉

缺点：

- 后续还要补组件系统
- 图片到设计系统之间仍有人工清洗成本

### 路线 B：Figma 内完成更多工作

`GPT 聊需求 -> 出关键 UI 图 -> Figma Make 生成交互原型 -> copy as layers -> GPT 补设计规范 md -> Codex 实现`

适合：

- 想尽快验证交互
- 团队主阵地在 Figma
- 希望减少 HTML 中转

优点：

- 原型感更强
- 设计和交互探索更连续

缺点：

- 仍然需要整理规范
- 生成结果未必天然符合现有设计系统

### 路线 C：代码先行，再回流设计

`GPT / Codex 直接生成前端预览 -> html.to.design 导入 Figma -> GPT 反向生成设计规范 md -> 继续精修 -> Codex 收敛实现`

适合：

- 已有前端工程或组件库
- 想快速看到真实页面
- 设计和开发要双向同步

优点：

- 落地速度快
- 更接近真实可运行页面

缺点：

- 如果一开始没有视觉锚点，代码稿容易普通
- 导回 Figma 后还需要设计再组织

## 真正建议给 Codex 的输入包

不要只给一句“照着这个风格做”。更稳的是给一个完整输入包：

### 1. 产品 spec

至少包含：

- 页面列表
- 核心 user flow
- 状态与异常流
- 数据字段
- 交互规则

### 2. 视觉关键帧

最少准备：

- 首页
- 核心转化页
- 一个高密度内容页
- 一个弹层/表单页

### 3. 设计规范 md

建议让 GPT 根据图反推并结构化成：

- 色板
- 字体层级
- 间距系统
- 圆角/阴影
- 卡片、按钮、输入框规则
- 列表、导航、弹窗的通用模式

### 4. 结构化设计上下文

优先级从高到低：

1. Figma 组件与变量
2. Figma 关键帧
3. HTML/CSS 原型
4. 单张图片

## 一个更实用的 prompt 模板

### 给 GPT 生关键帧

```md
请为一个 [产品类型] 设计移动端/桌面端 UI 关键帧。

目标页面： [首页 / 列表页 / 详情页 / onboarding]
目标气质： [克制 / 高级 / 温暖 / 编辑感 / 科技感]
用户任务： [用户在这一屏要完成什么]
必须出现的信息： [标题、卡片、CTA、筛选、数据等]
版式要求： [大留白 / 信息密集 / 强视觉主图 / 卡片式]
文字要求： 所有标题、按钮、标签都要清晰可读，使用真实产品语气。
约束： 不要出现多余 logo，不要出现无意义装饰，不要生成无法实现的异形控件。
```

### 让 GPT 从关键帧反推设计规范

```md
基于这张 UI 图，输出一份给前端和 Codex 可执行的设计规范 markdown。

要求包含：
1. 页面结构拆解
2. 色彩系统（主色/辅色/背景/文字/边框）
3. 字体层级（页面标题、区块标题、正文、说明、按钮）
4. 间距和圆角规则
5. 卡片、按钮、输入框、标签、导航的样式规范
6. 适配到组件化开发时的命名建议
7. 哪些地方是视觉重点，哪些地方允许工程化近似
```

### 给 Codex 落地

```md
请基于以下两类输入实现页面：

1. 产品规范 markdown：负责功能、结构、状态和交互
2. UI 关键帧 / Figma：负责视觉目标

实现要求：
- 优先保证版式层级、留白、组件比例和核心视觉气质
- 不要自行发明新的视觉风格
- 若图片中存在不利于工程复用的细节，请在保留整体观感的前提下组件化处理
- 先实现首页和核心 flow，再抽取通用组件
```

## 风险提醒

### 1. 不要把单张图当成完整设计系统

单张图通常缺：

- hover / active / disabled
- 空态 / 错态 / skeleton
- 响应式变化
- 长内容溢出处理

所以图只是开始，不是结束。

### 2. GPT 反推设计规范时会“合理化”不存在的规则

如果只给一张图，模型可能会把一些偶然视觉细节说成系统规则。更稳的做法是至少给 `2-4` 张关键帧一起总结规范。

### 3. 导入 Figma 后仍需要人工清洗

无论是 Make 预览转 layers，还是 `html.to.design` 导入，得到的往往是“可编辑对象”，不一定已经是严谨的组件和变量系统。

## 我的判断

这套方法最值得保留的，不是某个具体工具，而是这个原则：

`先定视觉 bar，再做结构化，再交给 Codex 实现。`

如果用一句话概括：

`不要让 Codex 负责“想象你想要的美术效果”，要让它负责“实现你已经明确过的视觉与交互约束”。`

## 本项目首页改版后的实操结论

这次在 `G:\StudyCode\宠物积分系统` 里真正落了一轮首页改版后，有一个很具体的经验值得单独记下来：

`不是“卡片少一点”就会更好，而是首页必须存在一个唯一主入口。`

在改版前，首页的问题并不只是视觉普通，而是：

- 轮播、概览、大卡片、左侧任务同时抢注意力
- 每块都像入口，结果反而没有入口
- 对孩子来说，不知道第一步该点哪里

改到后面真正变顺的版本，核心结构反而很简单：

1. 左侧只放稳定陪伴信息：今日任务、成长账户、孩子信息
2. 中间只放一个最重要的任务主卡：`今天先做这一件`
3. 右侧只放一个情绪型入口：`今日同行伙伴 / 前往宠物养成`
4. 下方只保留少量次级入口，不再做卡片瀑布流

也就是说，`首页的首屏不是信息总览，而是行动分发`。

### 这次主项目里真正验证有效的东西

在正式项目首页里，最后被保留下来的不是“demo 的所有细节”，而是下面这些结构原则：

- 顶部轮播被移除，因为它会抢走主任务卡的注意力
- 中间任务卡被放大，并明确写出 `完成后解锁宠物冒险`
- 宠物区不再只是信息卡，而是一个更像“游戏入口”的主角展示牌
- 左栏视觉被压弱，只做稳定陪衬，不再和主舞台争主次

这一点很关键，因为它说明：

`GPT 生图 / Figma / 设计规范` 真正该给 Codex 的，不只是“好看截图”，而是“哪些块必须强，哪些块必须弱”的层级判断。`

### 反推到 SOP，应该怎么写

如果以后要复用这套流程，给 Codex 的首页输入包里，最好额外明确三类判断：

#### 1. 唯一主行动

- 首屏最希望用户点什么
- 这块应该比其他入口明显多少
- 点击它后会得到什么正反馈

#### 2. 陪衬信息

- 哪些信息只是稳定存在，不应该抢主视觉
- 比如任务列表、账户余额、孩子信息

#### 3. 情绪型入口

- 哪个区域负责“吸引继续点开”
- 在这个项目里，答案是宠物同行伙伴

如果这三件事不先写进 spec / 设计规范，Codex 很容易又把首页实现成“每个卡片都差不多重要”的普通 dashboard。

## 你这条具体链路，再拆深一层

你说的流程是：

`和 GPT 聊一套大概提示词 -> 丢给 Figma / Cowart 画布出 UI 图 -> 把图丢回 GPT 生成设计规范 md -> 图 + md 一起交给 Codex`

这一条链路的价值，不在于“多绕了一圈”，而在于它把本来混在一起的三件事拆开了：

1. `GPT 对话` 负责把模糊想法压缩成一个足够稳定的视觉方向。
2. `Figma / Cowart 画布` 负责把方向变成可以看、可以比较、可以批注的可视对象。
3. `GPT 反推设计规范 md` 负责把图里的视觉规律翻译成 Codex 可执行的文字约束。

最后 `Codex` 做的是“按约束实现”，不是“替你猜设计”。

### 这条链路为什么比“直接让 Codex 做 UI”更稳

因为 `Codex` 在代码实现上强，但它天然不擅长从一句抽象描述里稳定复原下面这些东西：

- 视觉层级
- 留白气质
- 卡片比例
- 图文主次
- 品牌调性
- 局部精致感

如果没有中间那张图，Codex 收到的只是“极简、质感、高级、像苹果一点”这类高度主观描述。模型会尝试补全，但补出来的往往是“平均审美”，不是你真正想要的产品气质。

你这条链路的关键，其实是让“审美判断先落地，再进入工程”。

### 为什么“图 -> GPT 反推规范 md”这一步很关键

很多人做到“出图”就停了，但真正让 Codex 稳下来的是下一步。

因为截图本身只能表达：

- 大概长什么样
- 重点在哪
- 情绪和风格如何

但它不能精确告诉 Codex：

- 间距体系到底是多少
- 哪些颜色是 token，哪些只是装饰
- 哪些组件应该复用
- 标题、正文、辅助文案的层级怎么分
- 不同页面要保持哪些一致性

所以把图再丢回 GPT，要求它输出一份 `设计规范 md`，本质是在做“视觉反编译”：

`把感性的画面，翻译成工程可消费的结构化规则。`

这一层如果做得好，Codex 才真正有机会既保住风格，又写出可维护代码。

### 但这里最容易出错的地方，也正好是“反推规范”

因为 GPT 会倾向于把图里看到的东西“合理化”为一套完整系统，哪怕原图里有些细节只是偶然结果。最常见的失真有三类：

#### 1. 把偶然细节说成系统规则

比如某个卡片圆角看上去像 `20px`，模型就会写成“全局卡片统一 20px 圆角”；但真实情况可能只是这一屏局部强调。

#### 2. 把单屏视觉误判为全局组件规范

例如首页按钮是浅底深字，GPT 就会推断这是全站 primary button 规则；但实际到了支付页、表单页、空态页可能完全不够用。

#### 3. 只总结静态视觉，不总结状态机

模型很容易产出一份“颜色、字体、间距”文档，却遗漏：

- hover
- active
- disabled
- error
- loading
- empty state

而这些恰恰是 Codex 真正开始写代码时最容易失真的地方。

所以更稳的做法不是“拿一张图让 GPT 总结”，而是：

1. 一次给 `2-4` 张关键帧。
2. 明确让 GPT 区分“已观察到的规则”和“建议补充的工程规则”。
3. 单独要求它补齐状态、响应式和组件复用建议。

### Figma 和 Cowart 在这条链路里不是一回事

这点很值得单独说清楚。

#### Figma 更像“结构化设计上下文”

Figma 的强项是：

- 形成 frame、component、variable、style
- 把图进一步整理成可复用设计资产
- 让设计和开发围绕同一个结构化对象协作

如果你的目标是最终交给 Codex 一个尽量稳定的视觉来源，Figma 更接近“设计源文件”。

#### Cowart 更像“Codex 侧的视觉沟通画布”

根据 Cowart 项目 README，它是一个面向 Codex 的 `native infinite-canvas widget plugin`，基于 `tldraw`，主要能力是：

- 在 Codex 里直接打开无限画布
- 在画布上放图、做标注
- 让 Codex 根据标注去生图或改图
- 把画布数据保存在当前项目目录

这意味着 Cowart 更适合：

- 早期概念构思
- 对着图做局部批注
- 让 Codex 理解“这里改大一点、这里换布局、这里减少装饰”
- 作为开发中视觉纠偏和讨论层

但它不天然等于 Figma 那种“组件化、变量化、可交付设计系统”。

一句话区分：

- `Figma` 更适合作为设计源。
- `Cowart` 更适合作为 Codex 的视觉批注与迭代面板。

### 所以你这条链路最稳的版本，其实是四段式

我更推荐把它写成：

`GPT 对话定方向 -> Figma / 生图产关键帧 -> GPT 反推设计规范 md -> Codex 实现 -> Cowart 继续标注纠偏`

也就是把 Cowart 放在“实现后的对图纠偏”阶段，而不是把它等同于 Figma。

原因很简单：

- `Figma` 更适合产“源设计”
- `md` 更适合产“规则约束”
- `Codex` 更适合产“代码实现”
- `Cowart` 更适合产“可视化反馈闭环”

这四者分工清楚时，链路最顺。

### 如果你硬要只保留最小闭环，最不能省的是哪一步

最不能省的是：

`图 -> 规范 md`

因为没有这一步，Codex 看到的只是参考图，它会尽量模仿，但仍然得自己决定：

- 哪些东西抽成组件
- 哪些数值大致取多少
- 哪些风格细节需要保留
- 哪些地方要为工程妥协

而一旦有了规范 md，它至少知道：

- 该优先忠实什么
- 哪些是可近似项
- 哪些组件要优先抽象
- 哪些页面差异是 intentional，不是误差

### 这条链路的最佳使用场景

它最适合下面三类项目：

#### 1. 0-1 新产品

因为这时最大的风险不是“代码写不出来”，而是“产品看起来不成立”。

#### 2. 审美门槛高的界面

例如：

- 首页
- 品牌化强的核心流程
- 营销页
- 面向 C 端、靠情绪价值的产品

#### 3. 产品/设计/开发边界模糊的小团队

这类团队里，往往没人能单独完整产出“规范化设计系统 + 高质量工程实现”，所以用 `图 + md + Codex` 这种半结构化输入，反而效率最高。

### 不适合的场景

#### 1. 已有成熟 design system 的大团队

这时应该优先让 Codex 读现成 Figma 组件、变量和代码库，而不是先生成一张“新图”再反推规则。

#### 2. 数据密集、后台管理型产品

这类产品的难点更多在：

- 信息结构
- 表格密度
- 复杂筛选
- 异常态和边界态

单张效果图的帮助有限，反而更需要线框、组件策略和真实数据样例。

#### 3. 要求强一致性的多页面系统

如果一个产品有十几个以上页面、很多组件态，仅靠“几张关键帧 + 反推规范”容易出现系统漂移，最好尽快转入 Figma 组件化或直接基于现有代码系统约束。

### 真正高质量的提示，不该只描述视觉

你一开始“和 GPT 聊一套大概提示词”这件事，如果只聊画风，其实还不够。更稳的是让这套提示同时覆盖三层：

#### 1. 产品意图

- 这屏要让用户完成什么
- 这是探索页、决策页，还是提交页
- 用户此刻是轻松浏览，还是带着任务焦虑

#### 2. 视觉策略

- 高密度还是大留白
- 内容驱动还是视觉主图驱动
- 情绪关键词是什么

#### 3. 工程边界

- 不要异形控件
- 不要不可读小字
- 不要过多依赖玻璃拟态/超重阴影
- 尽量可组件化

这样最后生成出来的图，才更适合反推为 md，也更适合真正交给 Codex 落地。

## Codex 做 UI 的输入包 SOP

这一节不是再讲理念，而是讲怎么推进。目标只有一个：

`在把任务交给 Codex 之前，先把它真正需要的视觉和规则输入准备齐。`

如果输入包不完整，Codex 就会开始补脑；如果输入包足够完整，Codex 才会稳定进入“执行态”。

### 一、适用范围

这套 SOP 最适合：

- 0-1 新产品
- 首页、核心 flow、品牌感强的页面
- 小团队或个人项目
- 还没有成熟设计系统，但又不想让 Codex 瞎猜 UI

这套 SOP 不太适合：

- 已有强 design system 的大项目
- 复杂后台、表格、配置台类产品
- 需要一次性覆盖很多页面和很多组件态的系统

如果项目已经有现成组件库、Figma variables、成熟 token 和明确视觉规范，优先直接把这些结构化资产交给 Codex，而不是先走“生图 -> 反推规范”。

### 二、输入包标准

在给 Codex 做 UI 前，最少准备四类输入。

#### 1. 产品 spec

最少要有：

- 页面目标
- 用户任务
- 页面结构
- 核心操作路径
- 状态与异常流
- 数据字段或示例内容

不合格的 spec 长这样：

- “做一个宠物积分首页”
- “要高级一点”
- “参考苹果”

合格的 spec 至少要说清：

- 用户来到这页要干什么
- 这页最重要的信息层级是什么
- 点击后会发生什么
- 哪些状态必须覆盖

#### 2. 视觉关键帧

最少准备 `2-4` 张，不建议只给 `1` 张。

推荐最少包含：

- 首页 / 第一屏
- 一个核心操作页
- 一个高密度内容页
- 一个弹层、表单或详情页

关键帧的作用不是展示“所有页面”，而是锁定：

- 视觉气质
- 版式逻辑
- 组件倾向
- 信息密度
- 交互重心

#### 3. 设计规范 md

这是最不能省的一份输入。

至少要覆盖：

- 页面结构拆解
- 色彩系统
- 字体层级
- 间距与圆角
- 卡片、按钮、输入框、标签、导航规则
- 组件复用建议
- 状态规则
- 响应式建议
- 哪些地方必须忠实，哪些地方允许工程近似

如果这份 md 只有“主色、字体、圆角、大概风格”，那还不够，Codex 依然会在状态、组件抽象、交互细节上自己猜。

#### 4. 结构化视觉上下文

优先级如下：

1. Figma components / variables / styles
2. Figma 关键帧
3. 可运行 HTML/CSS 原型
4. Cowart 标注画布
5. 单张截图

这里要特别注意：

- `Figma` 更适合作为设计源
- `Cowart` 更适合作为批注、纠偏、迭代反馈层

不要把 Cowart 当成 Figma 的替代物，它们在链路里的职责不同。

### 三、推进步骤

下面是我更推荐的最小可执行流程。

#### Step 1：先和 GPT 聊方向，不急着要完整页面

第一轮对话不要直接说“帮我设计一个页面”，而是先把下面三层聊清楚：

- 产品意图
- 视觉策略
- 工程边界

也就是先确定：

- 用户在这屏要完成什么
- 这屏是探索、决策、提交还是反馈
- 要高密度还是大留白
- 要不要品牌主视觉
- 有没有不能碰的工程红线

这一轮的目标是收敛视觉方向，不是出最终图。

#### Step 2：生成关键帧，而不是一次做完整系统

第二步只拿关键页面，不追求全量覆盖。

推荐顺序：

1. 首页
2. 核心转化页
3. 高密度内容页
4. 一个弹层 / 表单 / 详情页

关键帧确定后，先检查三件事：

- 气质对不对
- 信息层级对不对
- 组件语言是不是统一

如果这三件事没对，先别进下一步。

#### Step 3：把关键帧丢回 GPT，反推设计规范 md

这一轮不要只让 GPT “总结风格”，要明确让它输出：

- 观察到的视觉规则
- 推断出的组件规则
- 建议补齐的工程规则

其中“建议补齐”特别重要，因为图里经常没有完整展示：

- hover
- active
- disabled
- loading
- empty
- error
- responsive

这些必须强制模型补出来，不然 Codex 写到一半就会重新进入补脑状态。

#### Step 4：如果有条件，把图整理进 Figma

如果项目对后续稳定性有要求，最好把关键帧进一步整理进 Figma，而不是只保留截图。

最理想的是把下面这些东西整理出来：

- frame
- component
- variable
- style
- layout 关系

因为截图只能告诉 Codex “长什么样”，Figma 才更能告诉它“这个东西是怎么搭出来的”。

#### Step 5：把图 + md + spec 一起交给 Codex

交付给 Codex 时，不要只扔一张图和一句话。最好明确说明三类输入分别负责什么：

- `spec` 负责功能与状态
- `设计规范 md` 负责规则与边界
- `图 / Figma` 负责视觉目标

这样 Codex 的角色就很清楚：不是设计者，而是实现者。

#### Step 6：实现后用 Cowart 做视觉纠偏

如果第一版代码已经出来，但你发现：

- 比例不对
- 留白不对
- 某些卡片太笨
- 某些局部需要重排

这时再把实现结果和参考图放进 Cowart，做可视化标注，让 Codex按批注迭代。

Cowart 在这里最好用，因为它擅长：

- 局部改动沟通
- 视觉差异对照
- 多轮纠偏

而不是承担整个设计源文件的职责。

### 四、给 GPT / Codex 的模板

下面这三类模板最值得沉淀下来反复用。

#### 模板 A：和 GPT 聊方向

```md
我们现在不是直接出最终页面，而是先确定 UI 方向。

产品类型： [填写]
目标用户： [填写]
目标页面： [填写]
这一屏的核心任务： [填写]
用户此刻的心理状态： [填写]

我希望这个页面的气质是：
- [如：克制、可信、轻盈、编辑感、温暖、科技感]

我希望它的版式策略是：
- [如：大留白 / 信息密度高 / 主视觉驱动 / 卡片式 / 列表优先]

工程边界：
- 不要不可实现的异形控件
- 不要不可读的小字
- 尽量组件化
- 尽量适配真实产品场景

请先不要直接出完整设计系统，而是先给我 2-3 个 UI 方向，并说明各自适合什么产品气质。
```

#### 模板 B：让 GPT 从图反推规范 md

```md
基于我提供的 2-4 张 UI 关键帧，输出一份给 Codex / 前端可执行的设计规范 markdown。

要求你区分三类内容：
1. 图中明确观察到的规则
2. 从多张图中推断出的组件规律
3. 为了工程落地建议补齐的规则

请至少包含：
- 页面结构拆解
- 色彩系统
- 字体层级
- 间距系统
- 圆角与阴影
- 卡片、按钮、输入框、标签、导航规则
- hover / active / disabled / loading / empty / error
- 响应式建议
- 组件抽象建议
- 哪些视觉细节必须忠实保留
- 哪些地方可以做工程近似
```

#### 模板 C：交给 Codex 执行

```md
请基于以下输入实现页面：

1. 产品 spec
2. 设计规范 markdown
3. UI 关键帧 / Figma 设计

执行要求：
- 优先忠实页面层级、比例、留白和核心视觉气质
- 不要自行发明新的视觉风格
- 对于图中存在但不利于复用的细节，请在保留整体观感的前提下做组件化抽象
- 必须覆盖状态、异常态和响应式
- 如视觉稿与 spec 冲突，以 spec 的功能逻辑为准，并尽量保持视觉一致

请先实现最关键的页面和核心 flow，再抽取可复用组件。
```

### 五、质量门槛

下面这些情况，不建议直接交给 Codex。

#### 1. 图还只是“氛围图”

如果图只有情绪，没有真实信息层级、没有按钮文案、没有真实产品语气，那它更像 moodboard，不像可执行参考图。

#### 2. 只有一张图

只给一张图时，Codex 很容易把局部特殊处理误当全局规范。

#### 3. 设计规范 md 只有视觉词，没有规则

比如只有：

- 极简
- 高级
- 科技感
- 柔和阴影

这类词只能提供方向，不能替代规则。

#### 4. 没有状态和异常流

如果没有：

- disabled
- loading
- error
- empty
- long text overflow

Codex 在真正写代码时几乎一定会自己编。

### 六、常见返工点

这是最容易反复踩坑的地方。

#### 1. 把单屏细节误当全局系统

解决方式：

- 至少给 `2-4` 张图
- 强制 GPT 区分“观察到”与“建议补齐”

#### 2. 只做静态视觉，不做状态设计

解决方式：

- 在设计规范 md 里强制加入状态章节
- 给 Codex 时明确要求实现异常态和交互态

#### 3. 只给图，不给真实文案和数据样例

解决方式：

- 用真实产品语气替代假字
- 给列表、卡片、表单准备最小真实数据样例

#### 4. Cowart 和 Figma 角色混淆

解决方式：

- 用 Figma 做设计源与结构化资产
- 用 Cowart 做批注、纠偏、对照和多轮反馈

#### 5. 太早把任务整包丢给 Codex

解决方式：

- 先定方向
- 再定关键帧
- 再出规范 md
- 最后才进入实现

### 七、最小落地清单

如果你只想记住最核心的推进顺序，可以直接看这个清单：

1. 先和 GPT 确定页面目标、气质、版式策略、工程边界。
2. 生成 `2-4` 张关键帧，不追求一开始做完整系统。
3. 选出最像目标产品的关键帧组合。
4. 把关键帧丢回 GPT，反推设计规范 md。
5. 补齐状态、响应式、组件复用规则。
6. 有条件就整理进 Figma，形成更结构化的视觉上下文。
7. 把 `spec + 图 + 规范 md` 一起交给 Codex。
8. 第一版实现出来后，用 Cowart 做视觉纠偏和局部迭代。

如果用一句话概括这份 SOP：

`先把你想要的 UI 说清、画清、规则化，再让 Codex 来实现，而不是让它一边猜一边写。`
